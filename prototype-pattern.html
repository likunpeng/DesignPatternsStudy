原型模式学习笔记

原型模式解决的是“创建一个对象成本高，但可以基于已有对象复制出新对象”的问题。它关注的不是从零创建，而是从一个现成样板克隆。

创建型模式 Prototype Pattern TypeScript + Python 对象克隆

今日目标

  • 理解原型模式解决的创建成本问题。
  • 知道浅拷贝和深拷贝的区别。
  • 能用 TypeScript 实现一个可克隆对象。
  • 能用 Python 的 copy 模块表达原型模式。
  • 知道什么时候不应该滥用克隆。

1. 这个模式解决什么问题

有些对象创建很麻烦，不只是 new 一下就结束。它可能需要读取配置、加载资源、做复杂计算，或者有一堆默认字段。

从零创建很贵，但从一个已有样板复制，再改少量字段很便宜。

原型模式就是先准备一个“原型对象”，后续新对象通过复制它得到。

2. 坏代码示例

假设系统里经常创建活动页面配置：

const page1 = {
  theme: "dark",
  layout: "landing",
  modules: ["banner", "coupon", "product-list"],
  tracking: { channel: "app", campaign: "spring" },
};

const page2 = {
  theme: "dark",
  layout: "landing",
  modules: ["banner", "coupon", "product-list"],
  tracking: { channel: "app", campaign: "summer" },
};

这段代码的问题是：

  • 大量字段重复。
  • 默认配置散落在多个地方。
  • 改默认模块时，容易漏改某些复制出来的配置。
  • 对象结构越复杂，重复创建越容易出错。

3. 一个生活类比

你做一份周报，不会每周都从空白文档开始排版。更常见的做法是复制上周的周报模板，然后修改日期、数据和总结。

这里的上周周报就是原型，新周报就是克隆后修改得到的新对象。

4. 适用场景

  • 对象创建成本高，复制比重新创建便宜。
  • 对象有一套稳定默认配置，只需要改少数字段。
  • 运行时才知道要复制哪种对象。
  • 想避免调用方知道复杂构造细节。
  • 创建测试数据时，需要在模板基础上快速变体。

5. 不适用场景

  • 对象很简单，直接构造更清楚。
  • 对象包含数据库连接、文件句柄、锁、线程等不适合复制的资源。
  • 对象内部可变嵌套结构很多，但没有明确深拷贝规则。
  • 克隆后需要大量修改，说明原型并不合适。

6. 核心结构

角色职责
Prototype声明克隆方法，例如 clone()。
ConcretePrototype具体可复制对象，实现克隆逻辑。
Client不直接从零创建对象，而是复制原型。
Prototype Registry可选，用名称保存多个原型，按需取出克隆。

7. TypeScript 示例

type Tracking = {
  channel: string;
  campaign: string;
};

class PageConfig {
  constructor(
    public theme: string,
    public layout: string,
    public modules: string[],
    public tracking: Tracking,
  ) {}

  clone(): PageConfig {
    return new PageConfig(
      this.theme,
      this.layout,
      [...this.modules],
      { ...this.tracking },
    );
  }
}

const springTemplate = new PageConfig(
  "dark",
  "landing",
  ["banner", "coupon", "product-list"],
  { channel: "app", campaign: "spring" },
);

const summerPage = springTemplate.clone();
summerPage.tracking.campaign = "summer";

console.log(springTemplate.tracking.campaign); // spring
console.log(summerPage.tracking.campaign); // summer

这里的 clone() 不只是复制引用，而是复制了 modules 数组和 tracking 对象，避免两个配置互相影响。

8. Python 示例

from copy import deepcopy
from dataclasses import dataclass


@dataclass
class Tracking:
    channel: str
    campaign: str


@dataclass
class PageConfig:
    theme: str
    layout: str
    modules: list[str]
    tracking: Tracking

    def clone(self) -> "PageConfig":
        return deepcopy(self)


spring_template = PageConfig(
    theme="dark",
    layout="landing",
    modules=["banner", "coupon", "product-list"],
    tracking=Tracking(channel="app", campaign="spring"),
)

summer_page = spring_template.clone()
summer_page.tracking.campaign = "summer"

print(spring_template.tracking.campaign)  # spring
print(summer_page.tracking.campaign)  # summer

Python 里可以直接用 copy.copy 做浅拷贝，用 copy.deepcopy 做深拷贝。示例里用深拷贝，是因为对象中有嵌套的可变结构。

9. 浅拷贝和深拷贝

类型含义风险
浅拷贝只复制对象第一层，嵌套对象仍然共享引用。修改嵌套对象时，原对象也可能被影响。
深拷贝递归复制嵌套对象，新旧对象尽量互不影响。成本更高，也可能遇到不能复制的资源。
原型模式最容易踩坑的地方就是拷贝深度。只要对象里有数组、字典、子对象，就要明确哪些应该共享，哪些应该复制。

10. 优缺点

优点缺点
避免重复复杂构造过程。克隆规则需要认真设计。
适合基于模板快速创建变体。深拷贝可能有性能成本。
调用方不必知道对象构造细节。包含外部资源的对象不一定能安全复制。
运行时可以动态选择原型。对象状态复杂时，调试共享引用问题会比较麻烦。

11. 和相似模式的区别

模式重点区别
原型模式复制已有对象适合基于模板创建相似对象。
工厂方法模式决定创建哪种对象通常还是从构造逻辑创建新对象。
建造者模式分步骤组装复杂对象关注构建过程清晰，而不是复制已有对象。
单例模式控制全局唯一实例单例避免多个实例，原型鼓励复制多个实例。

12. 练习题

你有一个优惠券模板对象，包含名称、折扣规则、适用渠道和有效期配置。现在每个活动都要基于这个模板复制一份，只改活动名称和有效期。

请思考：

  • 哪些字段应该深拷贝？
  • 哪些字段可以共享？
  • 如果折扣规则对象被多个优惠券共享，会有什么风险？
参考思路：优惠券模板可以提供 clone()。如果折扣规则是不可变对象，可以共享；如果规则会被活动单独修改，就必须复制。

今日总结

原型模式的核心是：不要每次都从零创建复杂对象，可以复制一个已有样板，再修改少量差异。真正的重点不是 clone() 这个方法，而是清楚地区分哪些状态应该复制，哪些状态可以共享。
