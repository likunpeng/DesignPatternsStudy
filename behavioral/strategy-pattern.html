策略模式教程

策略模式解决的是"同一个业务动作有多种可替换算法，而且这些算法经常变化"的问题。它把变化的部分拆出去，让主流程保持稳定。

今日目标

  • 理解策略模式解决的问题。
  • 知道为什么大量同类型 if-else 会让代码难维护。
  • 能用 TypeScript 实现策略模式。
  • 能用 Python 实现策略模式。
  • 知道策略模式适合和不适合的场景。

1. 这个模式解决什么问题

策略模式主要解决这种问题：

同一个业务动作，有多种不同算法或规则，而且这些规则可能经常变化。

例如：

  • 不同会员等级有不同折扣规则。
  • 不同支付方式有不同支付逻辑。
  • 不同物流方式有不同运费计算规则。
  • 不同登录方式有不同认证逻辑。

如果全部写在一个函数里，就很容易出现大量 if-else 或 switch。

2. 坏代码示例

假设我们要根据会员等级计算折扣。

function getDiscount(level: string): number {
  if (level === "normal") {
    return 1;
  }

  if (level === "vip") {
    return 0.9;
  }

  if (level === "svip") {
    return 0.8;
  }

  return 1;
}

这段代码的问题是：

  • 每新增一种会员等级，都要修改 getDiscount。
  • 折扣规则都挤在一个函数里，函数会越来越长。
  • 修改旧函数时，可能影响已有逻辑。
  • 每种折扣规则不方便单独测试。

这类代码违反了开闭原则：对扩展开放，对修改关闭。新增功能时，尽量通过增加新代码完成，而不是频繁修改稳定的旧代码。

3. 一个生活类比

出门去公司，可以选择不同出行策略：坐地铁、打车、骑车、步行。你的目标都是"到公司"，但具体路线和成本不同。这里的"出行方式"就是策略。

4. 适用场景

  • 有多种算法或规则可以互相替换。
  • 代码里有大量同类型的 if-else 或 switch。
  • 以后经常新增规则。
  • 每种规则需要独立测试。
  • 主流程稳定，但局部规则经常变化。

常见业务例子：支付方式、折扣规则、物流计费、登录方式。

5. 不适用场景

  • 规则很少，而且几乎不会变化。
  • 简单 if-else 已经足够清晰。
  • 业务逻辑还不稳定，过早抽象会增加理解成本。
  • 使用策略模式后，类或函数数量明显变多，但收益很小。

6. 核心结构

角色职责
Strategy定义所有策略的公共接口。
ConcreteStrategy具体策略实现。
Context持有策略引用，调用策略方法，不关心具体是哪种策略。

7. TypeScript 示例

interface DiscountStrategy {
  calculate(amount: number): number;
}

class NormalDiscount implements DiscountStrategy {
  calculate(amount: number): number { return amount; }
}

class VipDiscount implements DiscountStrategy {
  calculate(amount: number): number { return amount * 0.9; }
}

class SvipDiscount implements DiscountStrategy {
  calculate(amount: number): number { return amount * 0.8; }
}

class DiscountContext {
  constructor(private strategy: DiscountStrategy) {}

  getPrice(amount: number): number {
    return this.strategy.calculate(amount);
  }
}

const context = new DiscountContext(new VipDiscount());
console.log(context.getPrice(100)); // 90

新增黑金会员折扣时，只新增一个策略类，不需要改 DiscountContext。

8. Python 示例

from abc import ABC, abstractmethod


class DiscountStrategy(ABC):
    @abstractmethod
    def calculate(self, amount: float) -> float:
        pass


class NormalDiscount(DiscountStrategy):
    def calculate(self, amount: float) -> float:
        return amount


class VipDiscount(DiscountStrategy):
    def calculate(self, amount: float) -> float:
        return amount * 0.9


class SvipDiscount(DiscountStrategy):
    def calculate(self, amount: float) -> float:
        return amount * 0.8


class DiscountContext:
    def __init__(self, strategy: DiscountStrategy) -> None:
        self.strategy = strategy

    def get_price(self, amount: float) -> float:
        return self.strategy.calculate(amount)


context = DiscountContext(VipDiscount())
print(context.get_price(100))  # 90.0

Python 也可以更轻量地直接传函数：

def normal_discount(amount: float) -> float:
    return amount

def vip_discount(amount: float) -> float:
    return amount * 0.9

def get_price(strategy, amount: float) -> float:
    return strategy(amount)

print(get_price(vip_discount, 100))  # 90.0

这种函数式写法更短，也更符合 Python 的灵活风格。简单场景下不需要类。

9. 优缺点

优点缺点
新增规则不需要修改主流程。策略类数量会增加。
每种规则可以独立测试。简单场景下会过度设计。
符合开闭原则。调用方需要知道选择哪种策略。

10. 和相似模式的区别

模式重点区别
策略模式算法替换同一接口，不同实现可互换。
状态模式状态驱动行为状态切换会改变对象行为，策略由外部选择。
模板方法固定流程，可变步骤模板方法用继承，策略模式用组合。

11. 练习题

把下面的支付逻辑改造成策略模式：

function pay(type: string, amount: number) {
  if (type === "alipay") {
    console.log(`支付宝支付 ${amount} 元`);
  } else if (type === "wechat") {
    console.log(`微信支付 ${amount} 元`);
  } else if (type === "bank") {
    console.log(`银行卡支付 ${amount} 元`);
  }
}
参考思路：定义 PaymentStrategy 接口，每种支付方式实现一个策略类，PaymentContext 持有策略引用。新增支付方式时只加新策略类。

今日总结

策略模式的核心是：当一组规则经常变化时，把变化的部分拆出去，让主流程保持稳定。以后看到大量同类型 if-else 时，可以先思考：这里是不是适合使用策略模式？
