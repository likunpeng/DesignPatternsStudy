命令模式教程

命令模式解决的是"希望把请求发送者和请求接收者解耦，并且支持请求的排队、撤销、日志记录等操作"的问题。它把请求封装成对象。

行为型模式 Command Pattern TypeScript + Python 请求封装

今日目标

  • 理解命令模式解决的问题。
  • 知道什么时候应该把操作封装成命令对象。
  • 能用 TypeScript 实现命令模式。
  • 能用 Python 实现命令模式。
  • 知道命令模式的适用和不适用场景。

1. 这个模式解决什么问题

把"执行操作"封装成对象后，可以：

  • 把命令存入队列，按顺序执行
  • 记录命令历史，支持撤销/重做
  • 把命令发送给远程服务执行
  • 日志记录命令执行

这样调用者不需要知道具体怎么执行，只管发命令。

2. 坏代码示例

直接在调用者中写执行逻辑：

class Button {
  constructor(private editor: TextEditor) {}

  click() {
    this.editor.copy(); // 紧耦合
  }
}

问题：按钮和编辑器紧耦合，无法复用按钮控制其他操作，也无法支持撤销。

3. 一个生活类比

餐厅点餐：顾客只管点菜（发送命令），不需要知道厨师怎么炒菜（执行者）。服务员把订单（命令对象）记录下来，厨房按顺序做（命令队列）。

4. 适用场景

  • 需要支持撤销/重做操作。
  • 需要把操作排队、按顺序执行。
  • 需要日志记录操作。
  • 需要把请求发送者和接收者解耦。
  • 事务操作，支持回滚。

5. 不适用场景

  • 操作很简单，不需要队列、撤销等功能。
  • 不需要解耦调用者和执行者。

6. 核心结构

角色职责
Command定义执行操作的接口。
ConcreteCommand绑定接收者，实现具体的执行逻辑。
Invoker发起命令，不知道具体执行者。
Receiver知道如何执行操作。

7. 示例代码

TypeScript

interface Command {
  execute(): void;
  undo(): void;
}

class Light {
  on() { console.log("开灯"); }
  off() { console.log("关灯"); }
}

class LightOnCommand implements Command {
  constructor(private light: Light) {}
  execute() { this.light.on(); }
  undo() { this.light.off(); }
}

class RemoteControl {
  private history: Command[] = [];

  execute(command: Command) {
    command.execute();
    this.history.push(command);
  }

  undo() {
    const command = this.history.pop();
    if (command) command.undo();
  }
}

const light = new Light();
const lightOn = new LightOnCommand(light);
const remote = new RemoteControl();
remote.execute(lightOn);
remote.undo();

Python

from abc import ABC, abstractmethod

class Command(ABC):
    @abstractmethod
    def execute(self): pass

    @abstractmethod
    def undo(self): pass

class Light:
    def on(self): print("开灯")
    def off(self): print("关灯")

class LightOnCommand(Command):
    def __init__(self, light):
        self.light = light

    def execute(self): self.light.on()
    def undo(self): self.light.off()

class RemoteControl:
    def __init__(self):
        self.history = []

    def execute(self, command):
        command.execute()
        self.history.append(command)

    def undo(self):
        if self.history:
            self.history.pop().undo()

light = Light()
light_on = LightOnCommand(light)
remote = RemoteControl()
remote.execute(light_on)
remote.undo()

8. 优缺点

优点缺点
发送者和接收者完全解耦类数量增加，每个操作都要一个命令类
支持撤销、重做、排队、日志如果操作很简单，模式可能过度设计
符合开闭原则，可以新增命令命令对象可能暴露内部实现细节

9. 和相似模式的区别

模式区别
策略模式策略模式侧重于选择算法；命令模式侧重于把操作封装成对象。
责任链模式责任链是让请求在多个处理器间传递；命令模式是让命令排队执行。

10. 练习题

场景：实现一个文本编辑器，支持复制、粘贴、删除操作，并且支持撤销。写出代码结构。

参考答案：定义 Command 接口和 CopyCommand、PasteCommand、DeleteCommand 三个实现类。Editor 类作为 Receiver，真正执行操作。RemoteControl 或 EditorManager 作为 Invoker，维护命令历史并支持 undo。

今日总结

命令模式把每个操作封装成独立的对象，实现请求发送者和接收者的解耦。它是实现撤销/重做、命令队列、日志记录等功能的基础。
