责任链模式教程

责任链模式解决的是"一个请求需要经过多个处理器处理，但发送方不知道谁来处理"的问题。请求沿着链传递，直到某个处理器处理它。

1. 这个模式解决什么问题

责任链模式解决的是：一个请求可能由多个对象中的某一个处理，但请求发送者不应该知道最终是谁处理。

想象一个请假审批流程：请假 1 天以内由组长审批，3 天以内由经理审批，7 天以内由总监审批。申请人只管提交申请，不需要判断应该找谁。每个审批人只判断自己能不能处理，不能处理就交给下一个审批人。

这种场景在审批流程、权限校验、异常处理、日志级别处理等地方很常见。

2. 坏代码示例

如果把所有审批规则都写在一个函数里：

type LeaveRequest = {
  name: string;
  days: number;
};

function approveLeave(request: LeaveRequest): string {
  // 所有审批规则都挤在一个函数里。
  if (request.days <= 1) {
    return `组长审批通过：${request.name} 请假 ${request.days} 天`;
  }

  if (request.days <= 3) {
    return `经理审批通过：${request.name} 请假 ${request.days} 天`;
  }

  if (request.days <= 7) {
    return `总监审批通过：${request.name} 请假 ${request.days} 天`;
  }

  // 新增更高级审批人时，还要继续修改这个函数。
  return `${request.name} 请假 ${request.days} 天，没人能审批`;
}

问题：审批等级越多，这个函数越长。新增 CEO 审批、调整审批顺序、修改某个审批人的规则，都要改这个稳定函数，容易影响已有逻辑。

3. 一个生活类比

公司客服转接系统：客户打电话进来，客服能解决就解决，解决不了就转给专家，专家解决不了就转给经理。这个过程中，客户只打了一个电话，不知道最终是谁处理的。

4. 适用场景

  • 有多个对象可以处理同一个请求，但不确定由哪个处理。
  • 想让请求发送者和接收者解耦。
  • 需要动态调整处理器顺序或增删处理器。
  • 日志级别处理、权限校验、审批流程等。

5. 不适用场景

  • 请求只需要一个处理器处理，不需要链式传递。
  • 处理器数量固定且很少，直接 if-else 更简单。
  • 需要同步处理所有处理器，不适合链式。

6. 核心结构

角色职责
Handler（处理器）定义处理请求的接口，通常持有下一个处理器的引用。
ConcreteHandler（具体处理器）实现具体处理逻辑，能处理就处理，不能处理就传给下一个。
Client（客户端 / 组装者）创建处理器链，并把请求交给链头，不需要知道最终谁处理。

用中文理解，责任链的核心就是"一级一级往后问"：当前处理器能处理，请求就结束；当前处理器不能处理，就交给下一个处理器。

7. 示例代码

TypeScript

type LeaveRequest = {
  name: string;
  days: number;
};

// 抽象审批人：保存下一个审批人，并提供默认的传递逻辑。
abstract class Approver {
  private nextApprover: Approver | null = null;

  // 建立审批链，并返回下一个审批人，方便继续串联。
  setNext(approver: Approver): Approver {
    this.nextApprover = approver;
    return approver;
  }

  // 默认处理方式：自己不能审批时，把申请交给下一个审批人。
  handle(request: LeaveRequest): string {
    if (this.nextApprover) {
      return this.nextApprover.handle(request);
    }

    // 链条末尾也没人能处理，给出兜底结果。
    return `${request.name} 请假 ${request.days} 天，没人能审批`;
  }
}

// 具体审批人：组长只处理 1 天以内的请假。
class TeamLeader extends Approver {
  handle(request: LeaveRequest): string {
    if (request.days <= 1) {
      return `组长审批通过：${request.name} 请假 ${request.days} 天`;
    }

    // 组长处理不了，就交给下一个审批人。
    return super.handle(request);
  }
}

// 具体审批人：经理处理 3 天以内的请假。
class Manager extends Approver {
  handle(request: LeaveRequest): string {
    if (request.days <= 3) {
      return `经理审批通过：${request.name} 请假 ${request.days} 天`;
    }

    // 经理处理不了，就继续往后传。
    return super.handle(request);
  }
}

// 具体审批人：总监处理 7 天以内的请假。
class Director extends Approver {
  handle(request: LeaveRequest): string {
    if (request.days <= 7) {
      return `总监审批通过：${request.name} 请假 ${request.days} 天`;
    }

    return super.handle(request);
  }
}

// 客户端只负责组装链条，不需要知道最终是谁审批。
const leader = new TeamLeader();
const manager = new Manager();
const director = new Director();

leader.setNext(manager).setNext(director);

// 请求从链头开始，谁能处理就由谁处理。
console.log(leader.handle({ name: "小李", days: 2 }));

Python

from abc import ABC, abstractmethod
from dataclasses import dataclass


@dataclass
class LeaveRequest:
    name: str
    days: int


# 抽象审批人：保存下一个审批人，并提供默认的传递逻辑。
class Approver(ABC):
    def __init__(self):
        # 下一个审批人，默认没有。
        self._next_approver = None

    def set_next(self, approver):
        # 建立审批链，并返回下一个审批人，方便继续串联。
        self._next_approver = approver
        return approver

    @abstractmethod
    def handle(self, request: LeaveRequest) -> str:
        # 默认处理方式：自己不能审批时，把申请交给下一个审批人。
        if self._next_approver:
            return self._next_approver.handle(request)

        # 链条末尾也没人能处理，给出兜底结果。
        return f"{request.name} 请假 {request.days} 天，没人能审批"


# 具体审批人：组长只处理 1 天以内的请假。
class TeamLeader(Approver):
    def handle(self, request: LeaveRequest) -> str:
        if request.days <= 1:
            return f"组长审批通过：{request.name} 请假 {request.days} 天"

        # 组长处理不了，就交给下一个审批人。
        return super().handle(request)


# 具体审批人：经理处理 3 天以内的请假。
class Manager(Approver):
    def handle(self, request: LeaveRequest) -> str:
        if request.days <= 3:
            return f"经理审批通过：{request.name} 请假 {request.days} 天"

        # 经理处理不了，就继续往后传。
        return super().handle(request)


# 具体审批人：总监处理 7 天以内的请假。
class Director(Approver):
    def handle(self, request: LeaveRequest) -> str:
        if request.days <= 7:
            return f"总监审批通过：{request.name} 请假 {request.days} 天"

        return super().handle(request)


# 客户端只负责组装链条，不需要知道最终是谁审批。
leader = TeamLeader()
manager = Manager()
director = Director()

leader.set_next(manager).set_next(director)

# 请求从链头开始，谁能处理就由谁处理。
print(leader.handle(LeaveRequest(name="小李", days=2)))

8. 优缺点

优点缺点
发送者和接收者解耦链过长时性能可能受影响
符合开闭原则，易于增删处理器调试困难，难以确定哪个处理器处理了请求
可以灵活调整处理器顺序如果所有处理器都不处理请求，请求会丢失

9. 和相似模式的区别

模式区别
策略模式策略模式是调用方主动选择一个算法；责任链模式是请求沿链传递，由链上的处理器自己判断谁来处理。
装饰器模式装饰器在原对象上添加功能；责任链决定是否传递请求。
中间件 / 管道中间件通常每一层都会执行；经典责任链通常是某个处理器处理后就结束，除非它主动继续传递。

10. 练习题

场景：实现一个日志系统，支持 DEBUG、INFO、WARN、ERROR 四种级别，每个级别有对应的处理器，只处理该级别及以上的日志。写出代码结构。

参考答案：定义抽象 Handler 类，每个 ConcreteHandler 判断是否能处理该日志级别，能处理则处理，否则传递给下一个。如果都不处理，可以添加一个默认的 NullHandler 防止请求丢失。

今日总结

责任链模式通过将请求沿着处理者链传递，实现发送者和接收者的解耦。适合请求处理者不确定、需要动态调整处理顺序或增删处理逻辑的场景。
