责任链模式教程

责任链模式解决的是"一个请求需要经过多个处理器处理，但发送方不知道谁来处理"的问题。请求沿着链传递，直到某个处理器处理它。

行为型模式 Chain of Responsibility Pattern TypeScript + Python 请求传递

今日目标

  • 理解责任链模式解决的问题。
  • 知道什么时候应该用链式处理而不是集中处理。
  • 能用 TypeScript 实现责任链模式。
  • 能用 Python 实现责任链模式。
  • 知道责任链模式的适用和不适用场景。

1. 这个模式解决什么问题

想象一个审批流程：请假申请 → 组长审批 → 经理审批 → 总监审批。申请人只管提交，不知道谁会处理。处理者根据条件决定是自己处理还是传给下一个。

这种场景在日志处理、异常处理、权限校验等地方很常见。

2. 坏代码示例

如果用一个类处理所有逻辑：

function handleRequest(request) {
  if (handlerA.canHandle(request)) {
    return handlerA.handle(request);
  }
  if (handlerB.canHandle(request)) {
    return handlerB.handle(request);
  }
  if (handlerC.canHandle(request)) {
    return handlerC.handle(request);
  }
  // 每次新增处理器都要改这里
}

问题：每次增加或调整处理器都要修改这个函数，违反开闭原则。而且调用方需要知道所有处理器。

3. 一个生活类比

公司客服转接系统：客户打电话进来，客服能解决就解决，解决不了就转给专家，专家解决不了就转给经理。这个过程中，客户只打了一个电话，不知道最终是谁处理的。

4. 适用场景

  • 有多个对象可以处理同一个请求，但不确定由哪个处理。
  • 想让请求发送者和接收者解耦。
  • 需要动态调整处理器顺序或增删处理器。
  • 日志级别处理、权限校验、审批流程等。

5. 不适用场景

  • 请求只需要一个处理器处理，不需要链式传递。
  • 处理器数量固定且很少，直接 if-else 更简单。
  • 需要同步处理所有处理器，不适合链式。

6. 核心结构

角色职责
Handler定义处理请求的接口，持有下一个处理者的引用。
ConcreteHandler实现具体处理逻辑，决定是否传递给下一个。
Client创建链并发送请求，不需要知道谁处理。

7. 示例代码

TypeScript

interface Handler {
  setNext(handler: Handler): Handler;
  handle(request: string): string | null;
}

abstract class AbstractHandler implements Handler {
  private nextHandler: Handler | null = null;

  setNext(handler: Handler): Handler {
    this.nextHandler = handler;
    return handler;
  }

  handle(request: string): string | null {
    if (this.nextHandler) {
      return this.nextHandler.handle(request);
    }
    return null;
  }
}

class AuthHandler extends AbstractHandler {
  handle(request: string): string | null {
    if (request === "login") {
      return "已通过身份验证";
    }
    return super.handle(request);
  }
}

class LogHandler extends AbstractHandler {
  handle(request: string): string | null {
    console.log("记录日志:", request);
    return super.handle(request);
  }
}

// 使用
const auth = new AuthHandler();
const log = new LogHandler();
auth.setNext(log);
console.log(auth.handle("login"));

Python

from abc import ABC, abstractmethod

class Handler(ABC):
    def __init__(self):
        self._next_handler = None

    def set_next(self, handler):
        self._next_handler = handler
        return handler

    @abstractmethod
    def handle(self, request):
        if self._next_handler:
            return self._next_handler.handle(request)
        return None

class AuthHandler(Handler):
    def handle(self, request):
        if request == "login":
            return "已通过身份验证"
        return super().handle(request)

class LogHandler(Handler):
    def handle(self, request):
        print(f"记录日志: {request}")
        return super().handle(request)

# 使用
auth = AuthHandler()
log = LogHandler()
auth.set_next(log)
print(auth.handle("login"))

8. 优缺点

优点缺点
发送者和接收者解耦链过长时性能可能受影响
符合开闭原则，易于增删处理器调试困难，难以确定哪个处理器处理了请求
可以灵活调整处理器顺序如果所有处理器都不处理请求，请求会丢失

9. 和相似模式的区别

模式区别
策略模式策略模式选择一个算法执行；责任链模式让请求在链中传递，多个处理器都可能处理。
装饰器模式装饰器在原对象上添加功能；责任链决定是否传递请求。

10. 练习题

场景：实现一个日志系统，支持 DEBUG、INFO、WARN、ERROR 四种级别，每个级别有对应的处理器，只处理该级别及以上的日志。写出代码结构。

参考答案：定义抽象 Handler 类，每个 ConcreteHandler 判断是否能处理该日志级别，能处理则处理，否则传递给下一个。如果都不处理，可以添加一个默认的 NullHandler 防止请求丢失。

今日总结

责任链模式通过将请求沿着处理者链传递，实现发送者和接收者的解耦。适合请求处理者不确定、需要动态调整处理顺序或增删处理逻辑的场景。
