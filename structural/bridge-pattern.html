桥接模式教程

桥接模式解决的是“两个变化维度互相绑死，导致类数量爆炸”的问题。它把抽象和实现拆开，让它们可以独立变化。

结构型模式Bridge PatternTypeScript + Python组合优于继承

今日目标

  • 理解桥接模式为什么能减少继承层级。
  • 识别两个独立变化维度。
  • 能用组合把抽象层和实现层连接起来。
  • 能区分桥接和适配器。

1. 这个模式解决什么问题

如果一个系统有两个独立变化维度，直接用继承会产生组合爆炸。

通知类型：普通通知、紧急通知
  
2. 一个生活类比
遥控器和电视品牌是两个变化维度。遥控器可以有基础版、高级版；电视可以是不同品牌。如果每个遥控器都绑定一个电视品牌，类会越来越多。更好的做法是遥控器持有一个电视接口。

  
3. 适用场景
  • 两个维度都可能扩展，并且可以独立变化。
  • 继承层级已经因为组合关系变得臃肿。
  • 希望运行时切换实现。
  • 抽象逻辑和底层实现需要分开演进。

  
4. 不适用场景
  • 只有一个变化维度。
  • 组合关系并不稳定，强行抽象会增加理解成本。
  • 简单业务中几个类就能解决问题。

  
5. 核心结构
角色职责
Abstraction抽象层，面向业务提供高层能力。
RefinedAbstraction抽象层的具体变体。
Implementor实现层接口。
ConcreteImplementor实现层的具体实现。

  
6. TypeScript 示例
interface MessageSender {
  send(content: string): void;
}

class EmailSender implements MessageSender {
  send(content: string): void { console.log(`邮件：${content}`); }
}

class SmsSender implements MessageSender {
  send(content: string): void { console.log(`短信：${content}`); }
}

class Notification {
  constructor(protected sender: MessageSender) {}
  notify(content: string): void { this.sender.send(content); }
}

class UrgentNotification extends Notification {
  notify(content: string): void { this.sender.send(`[紧急] ${content}`); }
}

new UrgentNotification(new SmsSender()).notify("服务器 CPU 过高");
new Notification(new EmailSender()).notify("日报已生成");

  
7. Python 示例
from typing import Protocol


class MessageSender(Protocol):
    def send(self, content: str) -> None:
        ...


class EmailSender:
    def send(self, content: str) -> None:
        print(f"邮件：{content}")


class SmsSender:
    def send(self, content: str) -> None:
        print(f"短信：{content}")


class Notification:
    def __init__(self, sender: MessageSender) -> None:
        self.sender = sender

    def notify(self, content: str) -> None:
        self.sender.send(content)


class UrgentNotification(Notification):
    def notify(self, content: str) -> None:
        self.sender.send(f"[紧急] {content}")


UrgentNotification(SmsSender()).notify("服务器 CPU 过高")
Notification(EmailSender()).notify("日报已生成")

  
8. 优缺点
优点缺点
避免多维继承组合爆炸。需要先识别稳定的变化维度。
抽象和实现可独立扩展。比直接继承多一层接口和组合关系。
符合组合优于继承。对初学者不如适配器、外观直观。

  
9. 和相似模式的区别
模式重点区别
桥接拆分两个变化维度通常在设计前期主动使用。
适配器兼容已有接口通常在已有接口不匹配时补救。
策略替换算法策略多关注行为算法，桥接关注结构维度解耦。

  
10. 练习题
报表有 PDF、Excel 两种格式，输出位置有本地文件、对象存储两种。请用桥接模式避免 PdfLocalReport、PdfOssReport 这类组合类。
参考思路：格式作为抽象层，存储作为实现层；报表对象持有 ReportStorage。

  
今日总结
桥接模式的核心是“拆开两个变化维度，再用组合连接”。它不是为了多写接口，而是为了避免继承组合爆炸。
