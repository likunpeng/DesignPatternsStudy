外观模式学习笔记

外观模式解决的是“子系统太复杂，调用方不应该了解所有细节”的问题。它提供一个更高层、更简单的入口。

结构型模式Facade PatternTypeScript + Python简化入口

今日目标

  • 理解外观模式如何降低调用方复杂度。
  • 知道外观不是封锁底层能力，而是提供常用路径。
  • 能用外观封装一组服务调用。
  • 能区分外观和适配器。

1. 这个模式解决什么问题

一个业务动作可能要协调很多子系统。调用方如果直接调用每个子系统，就会知道太多细节，顺序、异常处理和组合规则都会散落在各处。

下单 = 校验库存 + 创建订单 + 扣减库存 + 发起支付 + 发送通知

外观模式把这些步骤封装成一个简单方法，例如 placeOrder()。

2. 一个生活类比

酒店前台就是外观。你不需要分别找保洁、餐厅、维修、财务，只要告诉前台“我要入住”或“我要退房”。前台背后会协调多个部门。

3. 适用场景

  • 一组类经常按固定流程一起使用。
  • 调用方不应该关心子系统内部顺序。
  • 需要为复杂库提供一个更简单的业务 API。
  • 想把系统分层，上层通过外观访问下层。

4. 不适用场景

  • 只是一个简单转发方法，没有降低复杂度。
  • 外观变成万能类，什么业务都塞进去。
  • 调用方确实需要灵活控制每个子系统细节。

5. 核心结构

角色职责
Facade提供简单入口，协调多个子系统。
Subsystem真正完成具体工作的类。
Client优先使用 Facade，不直接编排复杂流程。

6. TypeScript 示例

class InventoryService { reserve(sku: string) { console.log(`锁定库存 ${sku}`); } }
class OrderService { create(sku: string) { console.log(`创建订单 ${sku}`); return "order-1"; } }
class PaymentService { pay(orderId: string) { console.log(`支付订单 ${orderId}`); } }
class NotifyService { send(orderId: string) { console.log(`通知订单 ${orderId}`); } }

class CheckoutFacade {
  constructor(
    private inventory: InventoryService,
    private orders: OrderService,
    private payments: PaymentService,
    private notify: NotifyService,
  ) {}

  checkout(sku: string): void {
    this.inventory.reserve(sku);
    const orderId = this.orders.create(sku);
    this.payments.pay(orderId);
    this.notify.send(orderId);
  }
}

const checkout = new CheckoutFacade(new InventoryService(), new OrderService(), new PaymentService(), new NotifyService());
checkout.checkout("book-1");

7. Python 示例

class InventoryService:
    def reserve(self, sku: str) -> None:
        print(f"锁定库存 {sku}")

class OrderService:
    def create(self, sku: str) -> str:
        print(f"创建订单 {sku}")
        return "order-1"

class PaymentService:
    def pay(self, order_id: str) -> None:
        print(f"支付订单 {order_id}")

class NotifyService:
    def send(self, order_id: str) -> None:
        print(f"通知订单 {order_id}")

class CheckoutFacade:
    def __init__(self) -> None:
        self.inventory = InventoryService()
        self.orders = OrderService()
        self.payments = PaymentService()
        self.notify = NotifyService()

    def checkout(self, sku: str) -> None:
        self.inventory.reserve(sku)
        order_id = self.orders.create(sku)
        self.payments.pay(order_id)
        self.notify.send(order_id)

CheckoutFacade().checkout("book-1")

8. 优缺点

优点缺点
降低调用方复杂度，流程集中。外观容易膨胀成上帝对象。
减少上层对子系统的耦合。过度封装会限制高级调用方灵活性。
更容易统一异常处理和事务边界。子系统变化时，外观也可能频繁调整。

9. 和相似模式的区别

模式重点区别
外观简化复杂流程提供更易用的高层入口。
适配器接口兼容解决接口不匹配，而不是流程复杂。
中介者对象协作解耦中介者管理对象间双向协作，外观主要给外部使用。

10. 练习题

把“上传头像”封装成外观：压缩图片、上传文件、更新用户资料、清理缓存。请设计一个 AvatarFacade。

参考思路：外观提供 updateAvatar(userId, file)，内部协调图片服务、存储服务、用户服务和缓存服务。

今日总结

外观模式的核心是“给复杂子系统一个简单入口”。它不是隐藏所有能力，而是把常用流程变成稳定、清晰的业务 API。
