装饰器模式学习笔记

装饰器模式解决的是“在不修改原对象的前提下，动态叠加新能力”的问题。它强调增强行为，而不是替换对象。

结构型模式Decorator PatternTypeScript + Python动态增强

今日目标

  • 理解为什么继承不适合组合大量功能开关。
  • 掌握装饰器的共同接口和包裹关系。
  • 能区分装饰器模式和代理模式。
  • 知道 Python 语言装饰器和设计模式装饰器的联系与区别。

1. 这个模式解决什么问题

当一个对象需要可选能力时，如果用继承组合，会很快出现类爆炸。

基础通知 + 短信通知 + 邮件通知 + 日志通知 + 重试通知

装饰器模式让每个增强能力成为一个独立包装器，需要什么就包什么。

2. 一个生活类比

一杯基础咖啡可以加奶、加糖、加燕麦奶。咖啡本身没变，每个配料都在原有饮品外面叠加一层新效果。你可以自由组合，而不需要为每种组合创建一个新咖啡类。

3. 适用场景

  • 给对象增加日志、缓存、重试、鉴权、压缩、加密等横向能力。
  • 增强能力可以自由组合，且顺序可能有影响。
  • 不希望修改原类，或原类来自第三方库。
  • 希望比继承更灵活地扩展行为。

4. 不适用场景

  • 增强能力固定且很少，直接修改类更简单。
  • 包装层太多，调试调用链会变困难。
  • 调用方必须识别具体类型，装饰器会让类型判断更复杂。

5. 核心结构

角色职责
Component原对象和装饰器共同遵守的接口。
ConcreteComponent基础对象，提供核心能力。
Decorator持有一个 Component，并实现同样接口。
ConcreteDecorator在调用前后增加新行为。

6. TypeScript 示例

interface Notifier {
  send(message: string): void;
}

class EmailNotifier implements Notifier {
  send(message: string): void {
    console.log(`发送邮件：${message}`);
  }
}

class SmsDecorator implements Notifier {
  constructor(private wrapped: Notifier) {}

  send(message: string): void {
    this.wrapped.send(message);
    console.log(`发送短信：${message}`);
  }
}

class LogDecorator implements Notifier {
  constructor(private wrapped: Notifier) {}

  send(message: string): void {
    console.log("记录发送日志");
    this.wrapped.send(message);
  }
}

const notifier = new LogDecorator(new SmsDecorator(new EmailNotifier()));
notifier.send("订单已支付");

7. Python 示例

from typing import Protocol


class Notifier(Protocol):
    def send(self, message: str) -> None:
        ...


class EmailNotifier:
    def send(self, message: str) -> None:
        print(f"发送邮件：{message}")


class SmsDecorator:
    def __init__(self, wrapped: Notifier) -> None:
        self.wrapped = wrapped

    def send(self, message: str) -> None:
        self.wrapped.send(message)
        print(f"发送短信：{message}")


class LogDecorator:
    def __init__(self, wrapped: Notifier) -> None:
        self.wrapped = wrapped

    def send(self, message: str) -> None:
        print("记录发送日志")
        self.wrapped.send(message)


notifier = LogDecorator(SmsDecorator(EmailNotifier()))
notifier.send("订单已支付")

Python 的 @decorator 常用于包装函数；设计模式里的装饰器更通用，可以包装对象。两者思想相同：不改原逻辑，外面加一层。

8. 优缺点

优点缺点
比继承更灵活，可运行时组合功能。对象链变长后，排查问题更费力。
每个增强职责独立，符合单一职责。装饰器顺序可能影响结果，需要约定清楚。
不需要修改原对象。过度使用会产生很多小类。

9. 和相似模式的区别

模式重点区别
装饰器动态增强通常为了叠加新功能，可多层包装。
代理访问控制通常控制是否、何时、如何访问真实对象。
适配器接口转换让不兼容接口变成目标接口。

10. 练习题

给一个文件读取器增加 gzip 解压和读取日志功能。请思考：用继承会产生哪些组合类？用装饰器如何组合？

参考思路：定义 Reader.read()，基础类读取原始内容，GzipReaderDecorator 负责解压，LogReaderDecorator 负责记录耗时。

今日总结

装饰器模式的核心是“同接口、包一层、加能力”。当功能组合多、又不想改原类时，它比继承更灵活。
