享元模式教程

享元模式解决的是“大量相似对象占用太多内存”的问题。它把可共享的内部状态抽出来复用，把变化的外部状态放到调用时传入。

结构型模式Flyweight PatternTypeScript + Python对象共享

今日目标

  • 理解内部状态和外部状态的区别。
  • 知道享元模式为什么能节省内存。
  • 能实现一个简单享元工厂。
  • 知道什么时候不要为了“省对象”过度设计。

1. 这个模式解决什么问题

如果系统需要创建大量细粒度对象，而这些对象有很多相同数据，就会浪费内存。享元模式把相同部分共享起来。

一百万个字符对象：字体、颜色、字号可能重复；坐标和字符内容可能变化。

2. 一个生活类比

图书馆不会给每个读者都买一本同样的书，而是保存一批可共享的书。读者借阅时记录“谁借了哪本、什么时候还”，书本内容是共享的，借阅记录是外部状态。

3. 适用场景

  • 对象数量非常大，内存压力真实存在。
  • 对象可以拆分为可共享的内部状态和变化的外部状态。
  • 共享对象最好是不可变的。
  • 常见于文本编辑器字符样式、地图图标、游戏粒子、缓存池。

4. 不适用场景

  • 对象数量少，内存不是问题。
  • 对象状态高度个性化，无法有效共享。
  • 共享对象会被频繁修改，容易引发串数据问题。
  • 为了微小优化牺牲代码可读性。

5. 核心结构

角色职责
Flyweight可共享对象，保存内部状态。
FlyweightFactory按 key 复用或创建享元对象。
External State不能共享的变化状态，由调用方传入。
Client使用工厂获取享元，并提供外部状态。

6. TypeScript 示例

class TextStyle {
  constructor(
    public readonly font: string,
    public readonly size: number,
    public readonly color: string,
  ) {}

  render(char: string, x: number, y: number): void {
    console.log(`${char} at (${x}, ${y}) with ${this.font}/${this.size}/${this.color}`);
  }
}

class TextStyleFactory {
  private styles = new Map<string, TextStyle>();

  getStyle(font: string, size: number, color: string): TextStyle {
    const key = `${font}-${size}-${color}`;
    if (!this.styles.has(key)) {
      this.styles.set(key, new TextStyle(font, size, color));
    }
    return this.styles.get(key)!;
  }
}

const factory = new TextStyleFactory();
factory.getStyle("Arial", 12, "black").render("A", 0, 0);
factory.getStyle("Arial", 12, "black").render("B", 10, 0);

7. Python 示例

from dataclasses import dataclass


@dataclass(frozen=True)
class TextStyle:
    font: str
    size: int
    color: str

    def render(self, char: str, x: int, y: int) -> None:
        print(f"{char} at ({x}, {y}) with {self.font}/{self.size}/{self.color}")


class TextStyleFactory:
    def __init__(self) -> None:
        self.styles: dict[tuple[str, int, str], TextStyle] = {}

    def get_style(self, font: str, size: int, color: str) -> TextStyle:
        key = (font, size, color)
        if key not in self.styles:
            self.styles[key] = TextStyle(font, size, color)
        return self.styles[key]


factory = TextStyleFactory()
factory.get_style("Arial", 12, "black").render("A", 0, 0)
factory.get_style("Arial", 12, "black").render("B", 10, 0)

Python 示例用 dataclass(frozen=True) 表示共享对象不可变，降低被误改的风险。

8. 优缺点

优点缺点
显著减少重复对象带来的内存占用。需要区分内部状态和外部状态，设计复杂度提高。
共享对象集中管理，便于复用。调用方需要额外传入外部状态。
适合大量细粒度对象场景。共享对象如果可变，会产生难查的副作用。

9. 和相似模式的区别

模式重点区别
享元共享对象节省内存关注大量相似对象的复用。
单例全局唯一实例单例强调唯一，享元强调按 key 共享多个实例。
对象池复用昂贵对象生命周期对象池借出归还可变对象，享元通常共享不可变内部状态。

10. 练习题

地图上有十万个标记点，但图标类型只有餐厅、酒店、加油站三种。请用享元模式共享图标对象，并把经纬度作为外部状态。

参考思路：IconFactory 按类型缓存 MapIcon，渲染时传入 lat 和 lng。

今日总结

享元模式的核心是“共享不变的，外传变化的”。只有当对象数量巨大且内存问题真实存在时，它才值得引入。
