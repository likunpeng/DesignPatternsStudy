简单工厂模式教程

简单工厂模式解决的是"调用方不应该知道具体创建哪个类"的问题。它把创建逻辑集中到一个工厂函数里，调用方只传参数。

创建型模式 Simple Factory Pattern TypeScript + Python 集中创建

今日目标

  • 理解简单工厂和工厂方法的区别。
  • 知道简单工厂不是严格意义上的 GoF 模式。
  • 能用 TypeScript 和 Python 实现简单工厂。
  • 知道什么时候用简单工厂就够了，什么时候需要升级。

1. 这个模式解决什么问题

调用方需要根据条件创建不同对象，但不想在每个调用点都写 if-else 和 new。简单工厂把这段逻辑集中到一个地方。

调用方只说"我要一个 XX"，工厂负责决定创建哪个具体类。

它真正想解决的不是“少写一个 new”，而是：当创建规则发生变化时，不要让每个调用点都跟着改。

2. 一个生活类比

你去奶茶店点单：你说"一杯珍珠奶茶"，店员负责决定用哪种茶底、加什么料。你不需要知道制作细节，只需要说出名字。

3. 适用场景

  • 创建逻辑简单，产品种类不多。
  • 调用方只关心接口，不关心具体实现。
  • 创建逻辑可能变化，但变化不频繁。
  • 作为工厂方法模式的简化版本。

4. 不适用场景

  • 产品种类经常新增，每次都要改工厂函数，违反开闭原则。
  • 创建逻辑很复杂，一个工厂函数会变得很长。
  • 需要让子类决定创建哪种对象，应该用工厂方法模式。

5. 核心结构

角色职责
Product产品接口，所有具体产品实现它。
ConcreteProduct具体产品类。
Factory包含一个静态方法，根据参数创建具体产品。

6. TypeScript 示例

坏代码：调用方自己判断并创建

function bootLogger(type: "file" | "console") {
  let logger;

  if (type === "file") {
    // 调用方自己决定创建哪个具体类
    logger = new FileLogger();
  } else {
    logger = new ConsoleLogger();
  }

  logger.log("服务启动");
}

问题不是这段代码写不出来，而是类似判断一旦出现在多个模块里，创建规则就会散落各处。以后想统一改动，比如 file 日志要增加路径配置，就得回头找所有调用点。

TypeScript 示例

interface Logger {
  // 产品接口：调用方只依赖日志能力
  log(message: string): void;
}

class FileLogger implements Logger {
  constructor(private readonly path: string = "/var/log/app.log") {}

  log(message: string): void {
    console.log(`[File:${this.path}] ${message}`);
  }
}

class ConsoleLogger implements Logger {
  log(message: string): void {
    console.log(`[Console] ${message}`);
  }
}

class LoggerFactory {
  // 简单工厂：根据参数集中创建具体产品
  static create(type: "file" | "console"): Logger {
    switch (type) {
      case "file":
        // 创建细节集中在工厂里，调用方不用知道默认路径
        return new FileLogger();
      case "console":
        return new ConsoleLogger();
      default:
        throw new Error(`未知日志类型: ${type}`);
    }
  }
}

const logger = LoggerFactory.create("file");
logger.log("服务启动");

调用方只传 "file" 或 "console"，不直接 new 具体类。这样日志对象怎么初始化、默认参数是什么，都集中在工厂里维护。

判断标准：如果调用方只是在“选择一种现成实现”，而产品种类又不多，简单工厂就很好用。它适合把分散的创建判断收口，不适合承受高频扩展。

7. Python 示例

from abc import ABC, abstractmethod


class Logger(ABC):
    # 产品抽象：调用方只依赖日志能力
    @abstractmethod
    def log(self, message: str) -> None:
        pass


class FileLogger(Logger):
    def __init__(self, path: str = "/var/log/app.log") -> None:
        self.path = path

    def log(self, message: str) -> None:
        print(f"[File:{self.path}] {message}")


class ConsoleLogger(Logger):
    def log(self, message: str) -> None:
        print(f"[Console] {message}")


class LoggerFactory:
    @staticmethod
    def create(type_: str) -> Logger:
        # 简单工厂：把选择具体实现的规则集中起来
        if type_ == "file":
            return FileLogger()
        if type_ == "console":
            return ConsoleLogger()
        raise ValueError(f"未知日志类型: {type_}")


logger = LoggerFactory.create("file")
logger.log("服务启动")

Python 这里也一样，重点不是把 if-else 藏起来，而是把“根据参数选择实现、补默认配置”这类创建规则集中起来。

Python 里也可以用字典映射替代 if-else，更简洁：

class LoggerFactory:
    # 用注册表替代 if-else，仍然属于简单工厂
    _registry: dict[str, type[Logger]] = {
        "file": FileLogger,
        "console": ConsoleLogger,
    }

    @staticmethod
    def create(type_: str) -> Logger:
        cls = LoggerFactory._registry.get(type_)
        if cls is None:
            raise ValueError(f"未知日志类型: {type_}")
        return cls()

当类型数量不多时，if-else 和注册表都可以。注册表更像是在给简单工厂做轻量升级，但它本质上仍然是“一个工厂统一负责创建”。

8. 优缺点

优点缺点
调用方不需要知道具体类名。新增产品要改工厂，违反开闭原则。
创建逻辑集中，便于维护。产品多时工厂函数会变长。
实现简单，上手快。不是 GoF 标准模式，只是工厂方法的前置简化。

9. 和相似模式的区别

模式重点区别
简单工厂一个函数根据参数创建对象创建逻辑集中在一个地方，但新增产品通常要改工厂。
工厂方法子类决定创建哪种对象把“创建哪种产品”的变化交给不同工厂类处理，更适合持续扩展。
抽象工厂创建一组相关对象关注产品族，而不是单个产品。

可以先这么理解：简单工厂适合“种类少、规则集中”；工厂方法适合“种类会持续增加，不能老改同一个工厂”；抽象工厂适合“成套创建一组相关对象”。

10. 练习题

你有一个通知发送器，目前支持邮件和短信。请用简单工厂实现 NotificationFactory.create("email")。然后思考：如果新增企业微信通知，需要改哪些地方？这是不是违反了开闭原则？

参考思路：需要改工厂函数的 switch 或字典。确实违反开闭原则。如果通知类型经常新增，应该升级为工厂方法模式。

今日总结

简单工厂模式的核心是"把分散的创建判断收口到一个地方"。它不是为了炫技地把 new 藏起来，而是为了让调用方少知道具体类、少重复写创建规则。产品种类少且稳定时，简单工厂就够用了；产品经常新增时，应该升级为工厂方法模式。
