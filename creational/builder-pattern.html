建造者模式教程

建造者模式。它解决的不是“创建哪种对象”，而是“一个复杂对象怎么一步步组装，并且让创建过程更清楚”。

创建型模式 Builder Pattern TypeScript + Python 复杂对象构建

今日目标

  • 理解建造者模式解决的问题。
  • 知道为什么参数很多的构造函数会难维护。
  • 能用 TypeScript 实现建造者模式。
  • 能用 Python 实现建造者模式。
  • 能区分建造者模式、工厂方法模式和抽象工厂模式。
  • 知道什么时候不应该为了链式调用强行使用建造者模式。

1. 这个模式解决什么问题

建造者模式主要解决这个问题：

一个对象的创建过程很复杂，包含很多可选参数、组合步骤或校验规则。

常见情况包括：

  • 构造函数参数很多，而且有些参数是可选的。
  • 创建对象需要多个步骤，顺序还可能有要求。
  • 对象创建前需要做参数校验、默认值填充或组合计算。
  • 同一个对象有多种配置方式，但最终产物类型相同。

坏代码示例

假设我们要创建一份报表配置：

class Report {
  constructor(
    public title: string,
    public author: string,
    public showSummary: boolean,
    public showChart: boolean,
    public theme: string,
    public pageSize: string,
    public watermark?: string,
  ) {}
}

const report = new Report(
  "订单日报",
  "数据平台",
  true,
  true,
  "dark",
  "A4",
  "内部资料",
);

这段代码的问题是：参数太多，true, true, "dark", "A4" 调用可读性差，新增参数时很多调用点要改，默认值和校验逻辑容易散落在调用方。

建造者模式不是为了“少写代码”，而是为了让复杂对象的创建过程更明确、更可维护。

2. 一个生活类比

你去买一台定制电脑。

你不是一次性说出所有零件型号，而是一步步选择：

  • 选择 CPU。
  • 选择内存。
  • 选择硬盘。
  • 选择显卡。
  • 选择机箱。

最后商家根据这些配置组装成一台完整电脑。

这里的“配置过程”就是建造过程，最终的电脑就是被创建出来的复杂对象。

3. 适用场景

  • 对象参数很多，尤其是可选参数很多。
  • 创建对象需要多个步骤。
  • 需要隐藏复杂的组装细节。
  • 需要在构建完成前统一校验参数。
  • 同一个对象需要支持多种配置组合。

常见业务例子：

  • 报表配置：标题、作者、图表、汇总、页脚、水印。
  • HTTP 请求：URL、方法、Header、Query、Body、超时时间。
  • 消息模板：标题、正文、按钮、跳转链接、发送渠道。
  • 复杂查询条件：分页、排序、过滤、聚合字段。
  • 页面组件配置：主题、布局、模块、权限、埋点。

4. 不适用场景

  • 对象只有两三个参数，而且含义很清楚。
  • 构造过程没有步骤，也没有复杂校验。
  • 对象只是一个简单数据结构。
  • 使用对象字面量或配置对象已经足够清楚。
  • 为了链式调用好看而引入额外类。

例如：

type User = {
  id: string;
  name: string;
};

const user: User = { id: "u1", name: "张三" };

这种简单对象不需要建造者模式。

5. 核心结构

建造者模式通常包含这些角色：

  • 产品：最终被创建出来的复杂对象。
  • 建造者：一步步设置产品的各个部分。
  • 构建方法：通常叫 build，返回最终对象。
  • 可选的指挥者：固定一套构建流程，简单场景下可以不用。
设置标题 -> 设置作者 -> 设置图表 -> 设置主题 -> build() -> 得到报表

6. TypeScript 示例

type ReportOptions = {
  title: string;
  author: string;
  showSummary: boolean;
  showChart: boolean;
  theme: string;
  pageSize: string;
  watermark?: string;
};

class Report {
  // 最终产物只接收完整配置，不关心拼装过程
  constructor(private options: ReportOptions) {}

  print(): void {
    console.log(`标题: ${this.options.title}`);
    console.log(`作者: ${this.options.author}`);
    console.log(`主题: ${this.options.theme}`);
    console.log(`纸张: ${this.options.pageSize}`);
  }
}

class ReportBuilder {
  // 先给一套默认值，调用方只改关心的部分
  private options: ReportOptions = {
    title: "未命名报表",
    author: "系统",
    showSummary: false,
    showChart: false,
    theme: "light",
    pageSize: "A4",
  };

  // 每个方法只负责一个配置项，方便链式拼装
  setTitle(title: string): this {
    this.options.title = title;
    return this;
  }

  setAuthor(author: string): this {
    this.options.author = author;
    return this;
  }

  enableSummary(): this {
    this.options.showSummary = true;
    return this;
  }

  enableChart(): this {
    this.options.showChart = true;
    return this;
  }

  setTheme(theme: string): this {
    this.options.theme = theme;
    return this;
  }

  setWatermark(watermark: string): this {
    this.options.watermark = watermark;
    return this;
  }

  // 在出口统一校验，避免无效配置进入业务层
  build(): Report {
    if (!this.options.title.trim()) {
      throw new Error("报表标题不能为空");
    }

    return new Report({ ...this.options });
  }
}

const report = new ReportBuilder()
  .setTitle("订单日报")
  .setAuthor("数据平台")
  .enableSummary()
  .enableChart()
  .setTheme("dark")
  .setWatermark("内部资料")
  .build();

report.print();

关键职责：

  • Report：最终产物，代表一份报表。
  • ReportOptions：报表配置结构。
  • ReportBuilder：负责一步步收集配置。
  • build：统一校验并返回最终对象。

7. Python 示例

from dataclasses import dataclass


@dataclass(frozen=True)
class Report:
    # 最终产物只接收完整配置，不关心拼装过程
    title: str
    author: str
    show_summary: bool
    show_chart: bool
    theme: str
    page_size: str
    watermark: str | None = None

    def print(self) -> None:
        print(f"标题: {self.title}")
        print(f"作者: {self.author}")
        print(f"主题: {self.theme}")
        print(f"纸张: {self.page_size}")


class ReportBuilder:
    def __init__(self):
        # 先放默认值，调用方只改关心的部分
        self._title = "未命名报表"
        self._author = "系统"
        self._show_summary = False
        self._show_chart = False
        self._theme = "light"
        self._page_size = "A4"
        self._watermark = None

    # 每个方法只负责一个配置项，方便链式拼装
    def set_title(self, title: str):
        self._title = title
        return self

    def set_author(self, author: str):
        self._author = author
        return self

    def enable_summary(self):
        self._show_summary = True
        return self

    def enable_chart(self):
        self._show_chart = True
        return self

    def set_theme(self, theme: str):
        self._theme = theme
        return self

    def set_watermark(self, watermark: str):
        self._watermark = watermark
        return self

    # 在出口统一校验，避免无效配置进入业务层
    def build(self) -> Report:
        if not self._title.strip():
            raise ValueError("报表标题不能为空")

        return Report(
            title=self._title,
            author=self._author,
            show_summary=self._show_summary,
            show_chart=self._show_chart,
            theme=self._theme,
            page_size=self._page_size,
            watermark=self._watermark,
        )


report = (
    ReportBuilder()
    .set_title("订单日报")
    .set_author("数据平台")
    .enable_summary()
    .enable_chart()
    .set_theme("dark")
    .set_watermark("内部资料")
    .build()
)

report.print()

Python 中很多时候可以用 dataclass 加默认参数解决问题，不一定需要完整建造者：

@dataclass(frozen=True)
class Report:
    title: str = "未命名报表"
    author: str = "系统"
    show_summary: bool = False
    show_chart: bool = False
    theme: str = "light"
    page_size: str = "A4"
    watermark: str | None = None


report = Report(
    title="订单日报",
    author="数据平台",
    show_summary=True,
    show_chart=True,
    theme="dark",
    watermark="内部资料",
)

如果只是参数多但没有复杂构建过程，Python 的命名参数通常比建造者模式更直接。

8. 优缺点

优点 缺点
让复杂对象的创建过程更清楚。 会增加额外的 Builder 类。
避免构造函数参数过长。 简单对象使用 Builder 会过度设计。
可以集中处理默认值和校验逻辑。 链式调用可能隐藏必填字段是否设置完整。
同一个构建过程可以复用。 如果 Builder 可变，复用同一个实例时要小心状态污染。
TypeScript vs Python 实现差异：TypeScript 常用 Builder 解决参数多、配置复杂、需要链式调用的场景；Python 有命名参数、默认参数和 dataclass，很多场景不需要 Builder。Python 的 Builder 更适合存在复杂校验、构建步骤或复用流程的场景。两种语言都应该优先考虑配置对象或命名参数，只有复杂度足够高时再引入 Builder。

9. 和相似模式的区别

对比点 建造者模式 工厂方法模式
关注点 一个复杂对象如何一步步组装 应该创建哪一种具体对象
典型问题 参数多、步骤多、校验复杂 对象类型多、创建逻辑分散
返回对象 通常返回同一种产品的不同配置 通常返回同一接口下的不同实现
常见方法 setX()、enableX()、build() create()、make() 
如果问题是“对象参数太多，怎么组装更清楚”，更像建造者模式。
如果问题是“根据类型创建哪个类”，更像工厂方法模式。

和抽象工厂模式的区别

抽象工厂通常创建一组相关产品，建造者通常创建一个复杂产品。

对比点 建造者模式 抽象工厂模式
关注点 一个复杂对象的构建步骤 一组相关对象的创建
例子 一步步构建一份报表 创建深色主题下的按钮、输入框、弹窗
产物数量 通常是一个复杂对象 通常是一组对象
核心问题 怎么组装 创建哪一族

和配置对象模式的区别

很多时候，配置对象比建造者模式更简单。

type ReportOptions = {
  title: string;
  author?: string;
  showSummary?: boolean;
  showChart?: boolean;
  theme?: string;
};

function createReport(options: ReportOptions): Report {
  return new Report({
    author: "系统",
    showSummary: false,
    showChart: false,
    theme: "light",
    ...options,
  });
}

如果只是为了避免长参数列表，配置对象通常已经够了。

只有当配置过程需要步骤、校验、组合、复用流程时，建造者模式才更有价值。

10. 练习题

把下面的 HTTP 请求配置改造成建造者模式。

class HttpRequest {
  constructor(
    public url: string,
    public method: string,
    public headers: Record<string, string>,
    public query: Record<string, string>,
    public timeout: number,
    public body?: string,
  ) {}
}

const request = new HttpRequest(
  "/api/orders",
  "POST",
  { Authorization: "Bearer token" },
  { page: "1" },
  3000,
  JSON.stringify({ id: "A1001" }),
);

要求：

  • 定义 HttpRequestBuilder。
  • 支持设置 URL、Method、Header、Query、Timeout、Body。
  • build 时校验 URL 不能为空。
  • 用 TypeScript 写一版。
  • 用 Python 写一版。

思考题：

  • 如果只是普通请求对象，用配置对象是否更简单？
  • 如果请求构建过程有签名、鉴权、默认 Header，Builder 是否更合适？

参考答案

TypeScript 可以这样改：

class HttpRequestBuilder {
  private url = "";
  private method = "GET";
  private headers: Record<string, string> = {};
  private query: Record<string, string> = {};
  private timeout = 3000;
  private body?: string;

  setUrl(url: string): this {
    this.url = url;
    return this;
  }

  setMethod(method: string): this {
    this.method = method;
    return this;
  }

  addHeader(key: string, value: string): this {
    this.headers[key] = value;
    return this;
  }

  addQuery(key: string, value: string): this {
    this.query[key] = value;
    return this;
  }

  setTimeout(timeout: number): this {
    this.timeout = timeout;
    return this;
  }

  setBody(body: string): this {
    this.body = body;
    return this;
  }

  build(): HttpRequest {
    if (!this.url.trim()) {
      throw new Error("URL 不能为空");
    }

    return new HttpRequest(
      this.url,
      this.method,
      { ...this.headers },
      { ...this.query },
      this.timeout,
      this.body,
    );
  }
}

const request = new HttpRequestBuilder()
  .setUrl("/api/orders")
  .setMethod("POST")
  .addHeader("Authorization", "Bearer token")
  .addQuery("page", "1")
  .setBody(JSON.stringify({ id: "A1001" }))
  .build();

这段代码比长构造函数更清楚，因为每一步都表达了配置含义。但如果项目里没有复杂默认值、签名、校验、复用构建流程，配置对象可能更轻。

16. 今日总结

建造者模式的重点不是链式调用，而是这句话：

当一个复杂对象需要分步骤配置、校验和组装时，把构建过程封装到 Builder 里。

以后看到类似代码时：

new Xxx(a, b, true, false, "dark", 3000, undefined, options)

可以先思考：这里是不是构造参数已经影响可读性和维护性了？

如果只是参数多，优先考虑配置对象。如果还有复杂构建步骤、默认值、校验和复用流程，再考虑建造者模式。
