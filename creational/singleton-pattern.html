单例模式教程

单例模式解决的是"一个类在整个程序中只需要一个实例"的问题。它控制实例数量，而不是创建哪种对象。

创建型模式 Singleton Pattern TypeScript + Python 唯一实例

今日目标

  • 理解单例模式为什么存在，而不是到处 new。
  • 知道懒加载和饿加载的区别。
  • 能用 TypeScript 和 Python 实现线程安全的单例。
  • 知道单例模式最常见的误用场景。

1. 这个模式解决什么问题

有些对象在程序中只应该存在一份，例如：

  • 全局配置对象。
  • 数据库连接池。
  • 日志管理器。
  • 缓存管理器。
  • 设备驱动程序。

如果多处 new 出多个实例，可能导致配置不一致、资源浪费或状态冲突。单例模式保证一个类只有一个实例，并提供全局访问点。

2. 一个生活类比

一个国家只有一个总统，一个学校只有一个校长。不管谁去问，得到的都是同一个人。单例模式就是给类加了这个"唯一性"约束。

3. 适用场景

  • 全局配置，整个程序共享同一份设置。
  • 资源管理器，例如连接池、线程池、文件系统。
  • 日志服务，所有模块写同一个日志输出。
  • 缓存，避免多处维护不同缓存副本。
  • 硬件设备访问，避免重复初始化。

4. 不适用场景

  • 对象本身没有"全局唯一"的需求。
  • 为了方便传参而把类改成单例，这是滥用。
  • 需要多实例测试时，单例会让测试互相影响。
  • 单例承担了太多职责，变成上帝对象。
  • 分布式系统中，单例只保证进程内唯一，不保证跨进程唯一。

5. 核心结构

要素说明
私有构造函数禁止外部 new。
静态实例类内部保存唯一实例。
静态获取方法提供全局访问点，首次调用时创建实例。

6. TypeScript 示例

class AppConfig {
  private static instance: AppConfig;
  public readonly env: string;
  public readonly version: string;

  private constructor() {
    this.env = process.env.NODE_ENV ?? "development";
    this.version = "1.0.0";
  }

  static getInstance(): AppConfig {
    if (!AppConfig.instance) {
      AppConfig.instance = new AppConfig();
    }
    return AppConfig.instance;
  }
}

const config1 = AppConfig.getInstance();
const config2 = AppConfig.getInstance();
console.log(config1 === config2); // true

TypeScript 中更简洁的做法是利用模块机制：模块顶层变量天然是单例。

class AppConfig {
  public readonly env = process.env.NODE_ENV ?? "development";
  public readonly version = "1.0.0";
}

export const appConfig = new AppConfig();
在 Node.js / 前端模块系统中，模块只会被加载一次，export const 天然就是单例。大多数情况下不需要手写 getInstance()。

7. Python 示例

class AppConfig:
    _instance: "AppConfig | None" = None

    def __new__(cls) -> "AppConfig":
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance.env = "development"
            cls._instance.version = "1.0.0"
        return cls._instance


config1 = AppConfig()
config2 = AppConfig()
print(config1 is config2)  # True

Python 也可以用模块变量实现单例，更简单也更常见：

class _AppConfig:
    def __init__(self) -> None:
        self.env = "development"
        self.version = "1.0.0"

app_config = _AppConfig()

Python 模块同样只加载一次，app_config 就是进程内唯一的实例。

8. 优缺点

优点缺点
保证全局只有一个实例。隐藏依赖，调用方不知道自己用了全局状态。
提供统一的全局访问点。难以单元测试，测试之间会互相影响。
延迟初始化，按需创建。容易膨胀成上帝对象。
避免重复创建资源。多线程环境下需要额外加锁。

9. 和相似模式的区别

模式重点区别
单例模式控制实例数量为1关注"只有一个"。
原型模式复制已有对象原型鼓励复制多个实例，单例禁止。
静态类工具方法集合静态类不能实现接口、不能延迟加载、不能传参。
依赖注入由外部管理生命周期依赖注入可以更好地控制单例作用域，不需要类自己管。

10. 练习题

你有一个日志管理器，需要全局共享。请思考：

  • 用模块变量实现和用 getInstance() 实现，哪个更简单？
  • 如果日志管理器需要支持运行时切换输出目标（文件、控制台），单例会不会成为阻碍？
  • 测试时如何隔离单例状态？
参考思路：模块变量实现更简单。如果需要运行时切换行为，可以考虑策略模式配合单例。测试隔离可以在每次测试前重置单例实例，或者用依赖注入替代单例。

今日总结

单例模式的核心是"一个类只有一个实例"。它适合全局配置、资源管理、日志服务等场景。但单例也是最容易滥用的模式：如果只是图方便传参，或者把各种逻辑都塞进单例，就会制造隐藏依赖和测试困难。能用依赖注入解决的，就不要用单例。
