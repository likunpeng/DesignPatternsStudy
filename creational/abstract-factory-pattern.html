抽象工厂模式教程

抽象工厂模式解决的是"一组相关对象必须保持风格一致"的问题。它提供一个创建产品族的接口，而不是单个产品。

创建型模式 Abstract Factory Pattern TypeScript + Python 产品族

今日目标

  • 理解抽象工厂和工厂方法的本质区别。
  • 知道什么是"产品族"。
  • 能用 TypeScript 和 Python 实现抽象工厂。
  • 知道什么时候该用抽象工厂，什么时候用工厂方法就够了。

1. 这个模式解决什么问题

有些场景下，一组对象必须属于同一种风格或同一个系列。例如：

  • UI 组件：Windows 风格的按钮、输入框、窗口必须配套。
  • 数据库访问：MySQL 的连接、命令、读取器必须配套。
  • 主题系统：暗色主题的背景、文字、按钮颜色必须配套。
如果按钮是 Windows 风格，输入框却是 Mac 风格，界面就会不协调。

抽象工厂保证同一工厂创建出来的产品都是配套的。

它不是在解决“怎么创建一个按钮”，而是在解决“怎么保证按钮、输入框、弹窗这些部件是一整套的”。

2. 一个生活类比

你去宜家买家具。你可以选"北欧风"套装，也可以选"日式风"套装。每个套装里的沙发、茶几、书架风格一致。你不需要自己搭配，只要选一个套装就行。这里的"套装"就是产品族，"宜家"就是抽象工厂。

3. 适用场景

  • 一组对象需要保持风格或平台一致。
  • 系统需要支持多种风格或平台，且运行时切换。
  • 不想让调用方知道具体使用了哪种风格。
  • 产品族中的产品经常一起使用。

判断关键词通常是：成套、配套、一致性、同一平台、同一主题、同一数据库方言。只要你在想“这几个对象最好一起切换”，抽象工厂就值得考虑。

4. 不适用场景

  • 只需要创建一种产品，不需要产品族。
  • 产品之间没有配套关系，各自独立。
  • 产品族很少变化，简单工厂或工厂方法更直接。
  • 新增产品类型（而不是产品族）时，抽象工厂需要改所有工厂接口。

5. 核心结构

角色职责
AbstractFactory声明创建一组产品的方法。
ConcreteFactory实现具体风格的产品创建。
AbstractProduct每种产品的接口。
ConcreteProduct具体风格的产品实现。
Client只依赖抽象工厂和抽象产品。

6. TypeScript 示例

坏代码：调用方自己拼风格，容易混搭

const button = new DarkButton();
const input = new LightInput();

// 调用方自己拼装，容易把不同风格混在一起
button.render();
input.render();

单看每个对象都没问题，问题是调用方自己拼搭时，很容易出现“按钮是暗色，输入框是亮色”的混用。抽象工厂就是把这种“配套约束”也一起收进创建逻辑里。

TypeScript 示例

interface Button {
  // 抽象产品 A：按钮
  render(): void;
}

class DarkButton implements Button {
  render(): void { console.log("暗色按钮"); }
}

class LightButton implements Button {
  render(): void { console.log("亮色按钮"); }
}

interface Input {
  // 抽象产品 B：输入框
  render(): void;
}

class DarkInput implements Input {
  render(): void { console.log("暗色输入框"); }
}

class LightInput implements Input {
  render(): void { console.log("亮色输入框"); }
}

interface UIFactory {
  // 抽象工厂：一次创建一整套相关产品
  createButton(): Button;
  createInput(): Input;
}

class DarkUIFactory implements UIFactory {
  createButton(): Button { return new DarkButton(); }
  createInput(): Input { return new DarkInput(); }
}

class LightUIFactory implements UIFactory {
  createButton(): Button { return new LightButton(); }
  createInput(): Input { return new LightInput(); }
}

function renderUI(factory: UIFactory) {
  // 调用方只依赖工厂抽象，保证拿到的是同一套风格
  const button = factory.createButton();
  const input = factory.createInput();

  button.render();
  input.render();
}

renderUI(new DarkUIFactory());
renderUI(new LightUIFactory());

调用方只依赖 UIFactory，不关心具体是暗色还是亮色。切换主题只需要换一个工厂实例，而且同一个工厂天然保证按钮和输入框是同一套风格。

判断标准：如果你只是创建一个对象，抽象工厂通常太重；如果你每次都要成套创建多个相关对象，而且这些对象必须一起切换，抽象工厂就很合适。

7. Python 示例

from abc import ABC, abstractmethod


class Button(ABC):
    # 抽象产品 A：按钮
    @abstractmethod
    def render(self) -> None:
        pass


class DarkButton(Button):
    def render(self) -> None:
        print("暗色按钮")


class LightButton(Button):
    def render(self) -> None:
        print("亮色按钮")


class Input(ABC):
    # 抽象产品 B：输入框
    @abstractmethod
    def render(self) -> None:
        pass


class DarkInput(Input):
    def render(self) -> None:
        print("暗色输入框")


class LightInput(Input):
    def render(self) -> None:
        print("亮色输入框")


class UIFactory(ABC):
    # 抽象工厂：定义一整套 UI 组件的创建方法
    @abstractmethod
    def create_button(self) -> Button:
        pass

    @abstractmethod
    def create_input(self) -> Input:
        pass


class DarkUIFactory(UIFactory):
    def create_button(self) -> Button:
        return DarkButton()

    def create_input(self) -> Input:
        return DarkInput()


class LightUIFactory(UIFactory):
    def create_button(self) -> Button:
        return LightButton()

    def create_input(self) -> Input:
        return LightInput()


def render_ui(factory: UIFactory) -> None:
    # 调用方只依赖工厂抽象，不自己拼风格
    button = factory.create_button()
    input_box = factory.create_input()

    button.render()
    input_box.render()


render_ui(DarkUIFactory())
render_ui(LightUIFactory())

Python 里这类模式在跨平台 UI、数据库适配、主题系统里都很常见。关键收益仍然是“成套切换”，而不是单个对象创建得更优雅。

8. 优缺点

优点缺点
保证产品族风格一致。新增产品类型时，所有工厂都要改。
调用方不需要知道具体风格。类数量增加较快。
切换产品族只需要换工厂。产品族少时显得过度设计。
符合开闭原则（新增产品族）。不符合开闭原则（新增产品类型）。

9. 和相似模式的区别

模式重点区别
抽象工厂创建一组相关对象关注产品族，保证风格一致。
工厂方法子类决定创建哪种对象只创建一种产品，关注"谁决定"。
简单工厂一个函数根据参数创建对象最简单，但不保证产品族一致性。
建造者分步骤组装复杂对象关注构建过程，而不是产品族。

可以先这么理解：工厂方法解决“创建一个什么对象”，抽象工厂解决“创建一整套什么风格的对象”。一旦你的脑子里出现“按钮、输入框、菜单要一起换”，你就已经靠近抽象工厂了。

10. 练习题

你需要支持 MySQL 和 PostgreSQL 两种数据库，每种数据库都需要 Connection 和 Command 两种对象。请用抽象工厂模式实现，并思考：如果新增一种数据库（例如 SQLite），需要改哪些地方？如果新增一种产品类型（例如 Transaction），需要改哪些地方？

参考思路：新增数据库只需要新增一个 ConcreteFactory，符合开闭原则。新增产品类型需要改 AbstractFactory 接口和所有 ConcreteFactory，违反开闭原则。这是抽象工厂的经典取舍。

今日总结

抽象工厂模式的核心是"保证一组相关对象风格一致"。它适合多主题、多平台、多数据库等需要产品族配套的场景。记住一句话：工厂方法创建一种产品，抽象工厂创建一组产品。
