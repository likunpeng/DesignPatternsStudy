设计模式选择方法

设计模式不是越多越好。AI 很容易过度抽象，程序员要判断什么时候用模式、什么时候保持简单。

AI 工程能力Pattern SelectionTypeScript + Python克制设计

今日目标

  • 用变化点选择模式，而不是用名词选择模式。
  • 判断 if-else 是否真的需要策略模式。
  • 识别 AI 过度设计。

1. 这个方法解决什么问题

AI 听到“优化结构”后，可能立刻生成接口、工厂、策略、注册表。问题是：如果变化不存在，抽象就是负担。

2. 一个生活类比

出门买菜不需要开货车。工具要匹配场景，设计模式也是。

3. 适用场景

  • 准备重构重复逻辑。
  • AI 建议引入某个模式。
  • 需求未来可能有多种变化方向。

4. 不适用场景

  • 为了学习模式强行套代码。
  • 业务规则还不稳定，抽象会频繁推翻。
  • 简单代码已经清楚。

5. 核心结构

问题可能选择
算法/规则可替换策略模式
创建逻辑复杂或多产品族工厂/抽象工厂
外部接口不兼容适配器
访问需要控制代理
流程固定、步骤可变模板方法

6. 示例代码

TypeScript 示例

function shouldUseStrategy(ruleCount: number, changesOften: boolean) {
  return ruleCount >= 3 && changesOften;
}

Python 示例

def should_use_strategy(rule_count: int, changes_often: bool) -> bool:
    return rule_count >= 3 and changes_often

7. 优缺点

收益代价
避免 AI 过度设计。需要理解业务变化。
模式选择更稳定。没有固定公式。

8. 和架构选择的区别

模式选择架构选择
解决局部对象协作。解决模块、系统、团队边界。

9. 一个练习题

支付方式只有支付宝和微信，半年不会新增。是否需要策略模式？

10. 参考答案或改进建议

不一定。两个简单分支可以先保留。只有当支付规则复杂、经常新增、需要独立测试时，再引入策略或支付端口。

今日总结

设计模式服务于变化点。AI 能快速生成模式结构，但人必须判断这个抽象是否值得存在。
