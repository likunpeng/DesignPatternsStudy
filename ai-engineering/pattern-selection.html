设计模式选择方法

设计模式不是越多越好。AI 很容易过度抽象，程序员要判断什么时候用模式、什么时候保持简单。

AI 工程能力 Pattern Selection TypeScript + Python 克制设计

今日目标

  • 用变化点选择模式，而不是用名词选择模式。
  • 判断 if-else 是否真的需要策略模式。
  • 识别 AI 过度设计。

1. 这个方法解决什么问题

AI 听到“优化结构”后，可能立刻生成接口、工厂、策略、注册表。问题是：如果变化不存在，抽象就是负担。

2. 一个生活类比

出门买菜不需要开货车。工具要匹配场景，设计模式也是。

3. 适用场景

  • 准备重构重复逻辑。
  • AI 建议引入某个模式。
  • 需求未来可能有多种变化方向。

4. 不适用场景

  • 为了学习模式强行套代码。
  • 业务规则还不稳定，抽象会频繁推翻。
  • 简单代码已经清楚。

5. 核心结构

问题可能选择
算法/规则可替换策略模式
创建逻辑复杂或多产品族工厂/抽象工厂
外部接口不兼容适配器
访问需要控制代理
流程固定、步骤可变模板方法

6. 示例代码

不要把模式选择写成“规则数量大于 3 就用策略模式”这种公式。更好的判断方式是：先看现在的代码是否清楚，再看变化点是否已经独立到值得抽象。

阅读方法：下面用支付场景说明三件事：简单分支不一定是坏味道；AI 容易为了“优雅”提前抽象；只有当每种支付方式的变化原因已经分开时，策略模式才值得引入。

TypeScript 示例

AI 过度设计的版本

type PayMethod = "alipay" | "wechat";

interface PayStrategy {
  canHandle(method: PayMethod): boolean;
  pay(amount: number): string;
}

class AlipayStrategy implements PayStrategy {
  canHandle(method: PayMethod) {
    return method === "alipay";
  }

  pay(amount: number) {
    return `支付宝支付 ${amount} 元`;
  }
}

class WechatStrategy implements PayStrategy {
  canHandle(method: PayMethod) {
    return method === "wechat";
  }

  pay(amount: number) {
    return `微信支付 ${amount} 元`;
  }
}

class PayStrategyRegistry {
  constructor(private readonly strategies: PayStrategy[]) {}

  get(method: PayMethod): PayStrategy {
    const strategy = this.strategies.find((item) => item.canHandle(method));
    if (!strategy) {
      throw new Error(`不支持的支付方式: ${method}`);
    }
    return strategy;
  }
}

const registry = new PayStrategyRegistry([
  new AlipayStrategy(),
  new WechatStrategy(),
]);

function pay(method: PayMethod, amount: number) {
  return registry.get(method).pay(amount);
}

这段代码不是错，而是太早。当前只有两种支付方式，也没有手续费、风控校验、外部网关差异，这种抽象会让简单需求变得更重。

更合适的当前版本

type PayMethod = "alipay" | "wechat";

function pay(method: PayMethod, amount: number) {
  if (method === "alipay") {
    return `支付宝支付 ${amount} 元`;
  }
  return `微信支付 ${amount} 元`;
}

什么时候再引入策略模式

type PayMethod = "alipay" | "wechat" | "creditCard";

type PayRequest = {
  amount: number;
  userRegion: "CN" | "US";
};

interface PayStrategy {
  pay(request: PayRequest): string;
}

class AlipayStrategy implements PayStrategy {
  pay(request: PayRequest): string {
    return `支付宝支付 ${request.amount} 元`;
  }
}

class WechatStrategy implements PayStrategy {
  pay(request: PayRequest): string {
    return `微信支付 ${request.amount} 元`;
  }
}

class CreditCardStrategy implements PayStrategy {
  pay(request: PayRequest): string {
    if (request.userRegion !== "US") {
      throw new Error("信用卡支付当前只支持 US 区域");
    }
    const fee = request.amount * 0.02;
    return `信用卡支付 ${request.amount + fee} 元`;
  }
}

const strategies: Record<PayMethod, PayStrategy> = {
  alipay: new AlipayStrategy(),
  wechat: new WechatStrategy(),
  creditCard: new CreditCardStrategy(),
};

function payWithStrategy(method: PayMethod, request: PayRequest) {
  return strategies[method].pay(request);
}

真正该抽象的信号是：不同支付方式开始拥有不同校验规则、费用规则和外部依赖。此时变化原因分开了，策略模式才开始有收益。

Python 示例

当前先保持简单

from typing import Literal

PayMethod = Literal["alipay", "wechat"]

def pay(method: PayMethod, amount: int) -> str:
    if method == "alipay":
        return f"支付宝支付 {amount} 元"
    return f"微信支付 {amount} 元"

变化点出现后，用函数当策略更自然

from typing import Callable, Literal, TypedDict

PayMethod = Literal["alipay", "wechat", "credit_card"]

class PayRequest(TypedDict):
    amount: int
    user_region: Literal["CN", "US"]

PayStrategy = Callable[[PayRequest], str]

def pay_by_alipay(request: PayRequest) -> str:
    return f"支付宝支付 {request['amount']} 元"

def pay_by_wechat(request: PayRequest) -> str:
    return f"微信支付 {request['amount']} 元"

def pay_by_credit_card(request: PayRequest) -> str:
    if request["user_region"] != "US":
        raise ValueError("信用卡支付当前只支持 US 区域")
    fee = request["amount"] * 0.02
    return f"信用卡支付 {request['amount'] + fee} 元"

strategies: dict[PayMethod, PayStrategy] = {
    "alipay": pay_by_alipay,
    "wechat": pay_by_wechat,
    "credit_card": pay_by_credit_card,
}

def pay_with_strategy(method: PayMethod, request: PayRequest) -> str:
    return strategies[method](request)

Python 里不一定要先写一组类。函数本身就是很好用的策略。等策略内部需要共享状态、缓存连接、注入 SDK 客户端时，再升级为类就行。

7. 优缺点

收益代价
避免 AI 过度设计。需要理解业务变化。
模式选择更稳定。没有固定公式。

8. 和架构选择的区别

模式选择架构选择
解决局部对象协作。解决模块、系统、团队边界。

9. 一个练习题

支付方式只有支付宝和微信，半年不会新增。是否需要策略模式？如果 AI 已经帮你生成了策略注册表，你会保留还是删掉？

10. 参考答案或改进建议

大多数情况下先删掉。两个简单分支可读性更高，维护成本更低。只有当支付规则复杂、经常新增、需要独立测试或依赖不同外部网关时，再引入策略或支付端口。

今日总结

设计模式服务于变化点。AI 能快速生成模式结构，但人必须判断这个抽象是否值得存在。先让代码对当前问题足够清楚，再为真实变化付出抽象成本。
