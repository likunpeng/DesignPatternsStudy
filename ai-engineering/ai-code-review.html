AI 代码评审方法

AI 生成代码后，评审重点不是“它写了多少”，而是边界、行为、风险和测试是否可靠。

AI 工程能力 AI Code Review TypeScript + Python 质量验收

今日目标

  • 掌握 AI 代码评审的检查维度。
  • 识别看似正确但有隐患的代码。
  • 学会让 AI 先自查再提交。

1. 这个方法解决什么问题

AI 代码可能语法正确、功能看似完成，但隐藏边界破坏、异常遗漏、安全问题和测试缺口。评审是把生成速度转化为可靠交付的关键。

2. 一个生活类比

流水线生产很快，但出厂前仍要质检。速度越快，质检越重要。

3. 适用场景

  • AI 修改多个文件。
  • AI 新增业务流程。
  • AI 重构核心模块。
  • AI 生成测试或架构代码。

4. 不适用场景

  • 纯格式化或文案改动。
  • 临时实验代码。

5. 核心结构

维度检查问题
行为是否满足需求和边界条件？
架构依赖方向是否正确？
测试核心规则是否有测试？
安全输入、权限、敏感信息是否处理？
维护是否引入重复和过度抽象？

6. 示例代码

下面不是“如何写一个 review 函数”，而是“如何审一段 AI 写出来、表面能跑但埋了雷的业务代码”。场景是电商退款。

TypeScript 示例

AI 生成的版本：能跑，但风险很高

type Order = {
  id: string;
  userId: string;
  status: "paid" | "refunded";
  amount: number;
};

async function refund(order: Order, operatorId: string) {
  if (order.status === "refunded") {
    return "already refunded";
  }

  await paymentGateway.refund(order.id, order.amount);
  order.status = "refunded";
  await orderRepository.save(order);
  await auditLog.write(`operator=${operatorId}, order=${order.id}`);
  return "ok";
}

评审时要指出什么

type ReviewFinding = {
  category: "behavior" | "security" | "architecture" | "test";
  message: string;
};

function reviewRefundCode(): ReviewFinding[] {
  return [
    {
      category: "behavior",
      message: "幂等不足：并发请求可能同时通过 status 检查，导致重复退款。",
    },
    {
      category: "security",
      message: "没有校验 operatorId 是否有退款权限。",
    },
    {
      category: "architecture",
      message: "直接在用例里操作 paymentGateway 和 repository，缺少事务边界和失败补偿说明。",
    },
    {
      category: "test",
      message: "缺少已退款、网关失败、日志失败、并发退款等场景测试。",
    },
  ];
}

更稳的改进方向

async function refundOrder(command: {
  orderId: string;
  operatorId: string;
}) {
  const order = await orderRepository.getById(command.orderId);

  if (!permissionService.canRefund(command.operatorId, order)) {
    throw new Error("forbidden");
  }

  if (order.status === "refunded") {
    return "already refunded";
  }

  await refundService.refund(order);
  return "ok";
}

这里的重点不是把所有实现写完，而是评审者要能指出：权限应该独立校验，退款流程应该有统一服务处理，幂等和失败补偿不能靠“看起来没问题”蒙混过关。

Python 示例

Python 里可以把评审结果沉淀成清单函数

from dataclasses import dataclass

@dataclass
class ReviewFinding:
    category: str
    message: str

def review_refund_code() -> list[ReviewFinding]:
    return [
        ReviewFinding("behavior", "并发退款时可能重复调用支付网关。"),
        ReviewFinding("security", "没有校验操作人是否有退款权限。"),
        ReviewFinding("architecture", "用例直接依赖网关和仓储，失败补偿边界不清楚。"),
        ReviewFinding("test", "缺少已退款、网关失败、日志失败等测试。"),
    ]

把清单用于自查 Prompt

def build_review_prompt(change_summary: str) -> str:
    checklist = [
        "是否存在越权操作？",
        "是否考虑幂等和并发？",
        "是否破坏模块边界？",
        "是否补了核心测试？",
    ]
    return (
        f"请审查这次改动：{change_summary}\n"
        "输出格式：风险等级 + 问题说明 + 建议修改。\n"
        f"检查清单：{'; '.join(checklist)}"
    )

Python 这里更自然的做法不是复刻一套 TypeScript 类型系统，而是把评审经验沉淀成简单可复用的检查清单或脚本。

9. 一个练习题

AI 新增了退款功能，代码里已经调用了支付网关并更新订单状态。请列出至少 5 个你会优先检查的风险点，并说明为什么。

10. 参考答案或改进建议

优先检查幂等、权限、状态流转、失败补偿、测试覆盖。然后再看日志、监控、跨层依赖和异常处理。顺序不要反过来，因为这些前 5 项直接决定是否会产生真实业务事故。

今日总结

AI 代码评审的核心是风险识别。不要只看代码能不能跑，要看它是否在并发、异常、权限和模块边界上站得住。
