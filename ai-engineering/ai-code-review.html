AI 代码评审方法

AI 生成代码后，评审重点不是“它写了多少”，而是边界、行为、风险和测试是否可靠。

今日目标

  • 掌握 AI 代码评审的检查维度。
  • 识别看似正确但有隐患的代码。
  • 学会让 AI 先自查再提交。

1. 这个方法解决什么问题

AI 代码可能语法正确、功能看似完成，但隐藏边界破坏、异常遗漏、安全问题和测试缺口。评审是把生成速度转化为可靠交付的关键。

2. 一个生活类比

流水线生产很快，但出厂前仍要质检。速度越快，质检越重要。

3. 适用场景

  • AI 修改多个文件。
  • AI 新增业务流程。
  • AI 重构核心模块。
  • AI 生成测试或架构代码。

4. 不适用场景

  • 纯格式化或文案改动。
  • 临时实验代码。

5. 核心结构

维度检查问题
行为是否满足需求和边界条件？
架构依赖方向是否正确？
测试核心规则是否有测试？
安全输入、权限、敏感信息是否处理？
维护是否引入重复和过度抽象？

6. 示例代码

TypeScript 示例

type ReviewResult = { passed: boolean; risks: string[] };
function review(hasTests: boolean, crossesLayer: boolean): ReviewResult {
  const risks = [];
  if (!hasTests) risks.push("缺少核心测试");
  if (crossesLayer) risks.push("可能破坏分层依赖");
  return { passed: risks.length === 0, risks };
}

Python 示例

def review(has_tests: bool, crosses_layer: bool) -> dict:
    risks = []
    if not has_tests:
        risks.append("缺少核心测试")
    if crosses_layer:
        risks.append("可能破坏分层依赖")
    return {"passed": len(risks) == 0, "risks": risks}

9. 一个练习题

AI 新增了退款功能，你会检查哪些点？

10. 参考答案或改进建议

检查退款状态流转、幂等、权限、支付网关适配、失败重试、测试覆盖、是否直接改外部模块数据、是否记录关键日志。

今日总结

AI 代码评审的核心是风险识别。不要只看代码能不能跑，要看它是否破坏系统长期可维护性。
