从 Prompt 到架构方案

AI 协作不是一句话让它写代码，而是先让它理解需求、提出方案、比较取舍，再进入实现和验证。

AI 工程能力 Prompt to Architecture TypeScript + Python 方案生成

今日目标

  • 学会让 AI 先分析而不是直接实现。
  • 要求 AI 给多个方案并比较取舍。
  • 把方案落到目录、接口、测试和 ADR。

1. 这个方法解决什么问题

直接让 AI “实现某功能”，它通常会按局部最短路径写代码。架构方案需要先明确目标、约束、风险和演进方向。

2. 一个生活类比

盖房前不能直接叫工人开工，要先看地形、预算、用途、工期，再出方案。

3. 适用场景

  • 新增模块。
  • 重构旧系统。
  • 接入外部系统。
  • 需要比较多种架构方案。

4. 不适用场景

  • 小修小补。
  • 需求还完全不清楚。

5. 核心结构

步骤要求 AI 输出
理解需求目标、范围、未知点。
给方案2-3 个候选方案。
比较取舍复杂度、风险、演进成本。
落地设计目录、接口、数据流、测试。
记录决策ADR 草案。

6. 示例代码

下面用“通知系统”举例。坏 Prompt 的问题不是不够长，而是缺少边界、约束和输出格式；好 Prompt 的重点是把 AI 推进到方案比较阶段，而不是直接写代码。

TypeScript 示例

坏 Prompt：只会把 AI 推向局部实现

const badPrompt = `
帮我实现一个通知系统，支持邮件和短信，尽快给代码。
`;

更好的架构 Prompt 模板

type ArchitecturePromptInput = {
  businessGoal: string;
  constraints: string[];
  channels: string[];
};

function buildArchitecturePrompt(input: ArchitecturePromptInput): string {
  return `
你是系统架构师，请先不要写实现代码。

业务目标：
${input.businessGoal}

约束：
${input.constraints.map((item) => `- ${item}`).join("\n")}

通知渠道：
${input.channels.map((item) => `- ${item}`).join("\n")}

请按以下结构输出：
1. 需求理解与未知点
2. 候选方案 A / B / C
3. 每个方案的优缺点、复杂度、演进成本
4. 推荐方案及理由
5. 模块划分、接口边界、失败重试、测试策略
6. ADR 草案
`;
}

更有用的输出：能直接进入评审

const expectedOutput = `
推荐方案：模块化单体中的 notification 模块，内部用渠道适配器。

为什么不先做独立微服务：
- 当前通知量不大，独立部署会增加链路、监控和运维成本。
- 团队只有 4 人，先把模块边界守住比拆服务更重要。

模块边界：
- notification/application：发送通知用例、重试策略。
- notification/domain：通知任务、渠道选择规则。
- notification/infrastructure：邮件、短信、站内信适配器。

需要先确认的问题：
- 失败后重试几次？是否需要死信队列？
- 营销通知和交易通知是否共享通道？
- 用户退订规则由哪个模块负责？

测试策略：
- 渠道选择规则单元测试。
- 发送失败重试测试。
- 适配器用 contract test 隔离第三方 SDK。

ADR 草案：
- 背景：需要统一邮件、短信、站内信发送能力。
- 决策：先采用模块化单体 + 渠道适配器。
- 后果：部署简单，但必须通过模块规则约束直接调用第三方 SDK。
`;

这才是“从 Prompt 到架构”的完整链路：AI 的输出应该能进入评审和落地，而不是只生成一个看起来规整的对象结构。真正有价值的是边界、取舍、未知点、测试和 ADR。

Python 示例

Python 里适合把 Prompt 模板做成轻量工具

def build_architecture_prompt(goal: str, constraints: list[str], channels: list[str]) -> str:
    return f"""
你是系统架构师，请先不要写代码。

业务目标：
{goal}

约束：
{chr(10).join(f"- {item}" for item in constraints)}

通知渠道：
{chr(10).join(f"- {item}" for item in channels)}

请输出：
1. 需求理解与未知点
2. 候选方案
3. 推荐方案与理由
4. 模块边界、失败重试、测试策略
5. ADR 草案
"""

示例输出片段

expected_sections = [
    "推荐方案：模块化单体 + 渠道适配器",
    "不选微服务的原因：当前团队和运维复杂度不匹配",
    "模块边界：application / domain / infrastructure",
    "需要确认的问题：重试、退订、交易通知优先级",
    "测试策略：规则单测、失败重试、第三方 SDK contract test",
    "ADR 草案：背景、决策、后果",
]

Python 这里重点不在类型炫技，而在于快速组织“需求输入”和“期望输出”。它适合做 Prompt 组装器、方案检查器或 ADR 生成脚本。

7. 优缺点

收益代价
减少 AI 直接写错方向。前期多一步分析。
方案可比较、可记录。需要人做最终取舍。

8. 和普通 Prompt 的区别

普通 Prompt架构 Prompt
帮我实现订单模块。先分析订单模块边界，给 3 个方案并比较，再生成推荐方案。

9. 一个练习题

为“通知系统”写一个架构 Prompt，要求 AI 不直接写代码，而是先给出 2 到 3 个方案，并比较单体模块方案和独立服务方案的取舍。

10. 参考答案或改进建议

至少要包含业务目标、约束条件、集成渠道、输出格式和比较维度。没有这些，AI 很容易跳过方案阶段，直接给你一个默认实现。

今日总结

好的 Prompt 不是命令 AI 写代码，而是引导 AI 走完整个工程设计流程。先把问题问对，再让它写方案，最后才进入实现。
