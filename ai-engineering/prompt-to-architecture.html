从 Prompt 到架构方案

AI 协作不是一句话让它写代码，而是先让它理解需求、提出方案、比较取舍，再进入实现和验证。

AI 工程能力 Prompt to Architecture TypeScript + Python 方案生成

今日目标

  • 学会让 AI 先分析而不是直接实现。
  • 要求 AI 给多个方案并比较取舍。
  • 把方案落到目录、接口、测试和 ADR。

1. 这个方法解决什么问题

直接让 AI “实现某功能”，它通常会按局部最短路径写代码。架构方案需要先明确目标、约束、风险和演进方向。

2. 一个生活类比

盖房前不能直接叫工人开工，要先看地形、预算、用途、工期，再出方案。

3. 适用场景

  • 新增模块。
  • 重构旧系统。
  • 接入外部系统。
  • 需要比较多种架构方案。

4. 不适用场景

  • 小修小补。
  • 需求还完全不清楚。

5. 核心结构

步骤要求 AI 输出
理解需求目标、范围、未知点。
给方案2-3 个候选方案。
比较取舍复杂度、风险、演进成本。
落地设计目录、接口、数据流、测试。
记录决策ADR 草案。

6. 示例代码

下面用“通知系统”举例。坏 Prompt 的问题不是不够长，而是缺少边界、约束和输出格式；好 Prompt 的重点是把 AI 推进到方案比较阶段，而不是直接写代码。

TypeScript 示例

坏 Prompt：只会把 AI 推向局部实现

const badPrompt = `
帮我实现一个通知系统，支持邮件和短信，尽快给代码。
`;

更好的架构 Prompt 模板

type ArchitecturePromptInput = {
  businessGoal: string;
  constraints: string[];
  channels: string[];
};

function buildArchitecturePrompt(input: ArchitecturePromptInput): string {
  return `
你是系统架构师，请先不要写实现代码。

业务目标：
${input.businessGoal}

约束：
${input.constraints.map((item) => `- ${item}`).join("\n")}

通知渠道：
${input.channels.map((item) => `- ${item}`).join("\n")}

请按以下结构输出：
1. 需求理解与未知点
2. 候选方案 A / B / C
3. 每个方案的优缺点、复杂度、演进成本
4. 推荐方案及理由
5. 模块划分、接口边界、失败重试、测试策略
6. ADR 草案
`;
}

把结果落成方案对象

type ArchitectureOption = {
  name: string;
  pros: string[];
  cons: string[];
  recommended: boolean;
};

const options: ArchitectureOption[] = [
  {
    name: "模块化单体 + 渠道适配器",
    pros: ["开发快", "部署简单", "适合团队规模不大"],
    cons: ["需要持续约束模块边界"],
    recommended: true,
  },
  {
    name: "独立通知微服务",
    pros: ["边界清晰", "可独立扩展"],
    cons: ["运维和链路复杂度更高"],
    recommended: false,
  },
];

这才是“从 Prompt 到架构”的完整链路：先明确输入，再要求比较，再沉淀成可讨论的方案对象。不是让 AI 一步跳到代码文件。

Python 示例

Python 里适合把 Prompt 和输出结构做成轻量工具

from dataclasses import dataclass

@dataclass
class ArchitectureOption:
    name: str
    pros: list[str]
    cons: list[str]
    recommended: bool = False

def build_architecture_prompt(goal: str, constraints: list[str], channels: list[str]) -> str:
    return f"""
你是系统架构师，请先不要写代码。

业务目标：
{goal}

约束：
{chr(10).join(f"- {item}" for item in constraints)}

通知渠道：
{chr(10).join(f"- {item}" for item in channels)}

请输出：
1. 需求理解与未知点
2. 候选方案
3. 推荐方案与理由
4. 模块边界、失败重试、测试策略
5. ADR 草案
"""

示例候选方案

options = [
    ArchitectureOption(
        name="模块化单体 + 渠道适配器",
        pros=["开发快", "部署简单"],
        cons=["需要约束模块依赖"],
        recommended=True,
    ),
    ArchitectureOption(
        name="独立通知微服务",
        pros=["服务边界清楚", "可以独立扩容"],
        cons=["链路和运维成本更高"],
    ),
]

Python 这里重点不在类型炫技，而在于快速组织“需求输入”和“方案输出”。它适合做 Prompt 组装器、方案比对器或 ADR 生成脚本。

7. 优缺点

收益代价
减少 AI 直接写错方向。前期多一步分析。
方案可比较、可记录。需要人做最终取舍。

8. 和普通 Prompt 的区别

普通 Prompt架构 Prompt
帮我实现订单模块。先分析订单模块边界，给 3 个方案并比较，再生成推荐方案。

9. 一个练习题

为“通知系统”写一个架构 Prompt，要求 AI 不直接写代码，而是先给出 2 到 3 个方案，并比较单体模块方案和独立服务方案的取舍。

10. 参考答案或改进建议

至少要包含业务目标、约束条件、集成渠道、输出格式和比较维度。没有这些，AI 很容易跳过方案阶段，直接给你一个默认实现。

今日总结

好的 Prompt 不是命令 AI 写代码，而是引导 AI 走完整个工程设计流程。先把问题问对，再让它写方案，最后才进入实现。
