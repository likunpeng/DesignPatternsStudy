代码坏味道与重构

AI 能快速生成代码，但也可能快速制造坏味道。程序员需要识别结构问题，并用小步重构把代码带回可维护状态。

AI 工程能力Code Smells & RefactoringTypeScript + Python结构治理

今日目标

  • 识别 AI 常见坏味道：大函数、胖类、重复逻辑、错误依赖、过度抽象。
  • 知道坏味道和设计模式之间的关系。
  • 掌握小步、安全、可回滚的重构方法。

1. 这个能力解决什么问题

AI 生成代码通常先追求完成任务，不一定关注长期结构。坏味道不会马上报错，但会让下一次改动更贵。

2. 一个生活类比

房间临时堆东西最快，但堆久了找东西越来越难。重构就是在不改变用途的前提下重新收纳。

3. 适用场景

  • AI 一次生成了很长函数。
  • 多个文件出现重复逻辑。
  • 业务代码直接依赖数据库、SDK、页面框架。
  • 新增需求时总要改同一个大类。

4. 不适用场景

  • 还没有测试保护的高风险核心流程，先补测试。
  • 一次性脚本，维护周期很短。
  • 为了“看起来高级”而抽象。

5. 核心结构

坏味道常见处理
大函数按意图拆函数，先保留行为。
重复逻辑抽规则函数或策略。
胖类按变化原因拆职责。
错误依赖引入端口和适配器。
过度抽象删除没有变化支撑的层。

6. 示例代码

这个例子想说明的不是“校验金额也要包装一层”，而是：AI 常把校验、规则、持久化、通知都塞进一个函数。重构的目标是把变化原因分开，不是为了多写函数。

阅读方法：先看“重构前”里标出的坏味道，再看“重构后”每个新增方法的中文注释。判断代码好不好，不只看行数少不少，而是看变化点有没有被分开。

TypeScript 示例

重构前：坏味道写法

type OrderInput = { userId: string; amount: number; level: "normal" | "vip" };
type Order = { userId: string; amount: number; payable: number; status: "created" };

function createOrderBad(input: OrderInput) {
  // 坏味道 1：校验、折扣规则、数据组装、保存、通知都塞在一个函数里。
  if (input.amount <= 0) throw new Error("金额必须大于 0");

  // 坏味道 2：会员折扣规则直接写死在流程中，后续增加黑金会员会继续堆 if。
  const payable = input.level === "vip" ? input.amount * 0.9 : input.amount;

  const order = {
    userId: input.userId,
    amount: input.amount,
    payable,
    status: "created" as const,
  };

  // 坏味道 3：业务流程直接依赖具体副作用，测试时很难只验证下单逻辑。
  database.orders.insert(order);
  sms.send(input.userId, "订单已创建");

  return order;
}

重构后：较好的写法

type OrderInput = { userId: string; amount: number; level: "normal" | "vip" };
type Order = { userId: string; amount: number; payable: number; status: "created" };

// 只负责校验订单输入，输入规则变化时优先改这里。
function validateOrderInput(input: OrderInput): void {
  if (input.amount <= 0) throw new Error("金额必须大于 0");
}

// 只负责计算应付金额，折扣规则变化时优先改这里。
function calculatePayable(input: OrderInput): number {
  return input.level === "vip" ? input.amount * 0.9 : input.amount;
}

// 只负责组装订单对象，字段结构变化时优先改这里。
function buildOrder(input: OrderInput): Order {
  return {
    userId: input.userId,
    amount: input.amount,
    payable: calculatePayable(input),
    status: "created" as const,
  };
}

// 负责串联下单流程，保存和通知通过参数传入，方便替换和测试。
function createOrder(input: OrderInput, save: (order: Order) => void, notify: (userId: string) => void) {
  validateOrderInput(input);
  const order = buildOrder(input);

  save(order);
  notify(input.userId);
  return order;
}

好在哪里：校验、金额规则、订单组装、外部副作用被分开了。以后改折扣规则时，不需要同时碰保存和通知；测试下单流程时，也可以传入假的 save 和 notify。

Python 示例

重构前：坏味道写法

from typing import Literal, TypedDict

class OrderInput(TypedDict):
    user_id: str
    amount: int
    level: Literal["normal", "vip"]

def create_order_bad(input: OrderInput) -> dict:
    # 坏味道 1：校验、折扣规则、数据组装、保存、通知都塞在一个函数里。
    if input["amount"] <= 0:
        raise ValueError("金额必须大于 0")

    # 坏味道 2：会员折扣规则直接写死在流程中，后续新增等级会继续堆分支。
    payable = input["amount"] * 0.9 if input["level"] == "vip" else input["amount"]

    order = {
        "user_id": input["user_id"],
        "amount": input["amount"],
        "payable": payable,
        "status": "created",
    }

    # 坏味道 3：业务流程直接依赖具体副作用，单元测试会被数据库和短信拖住。
    database.orders.insert(order)
    sms.send(input["user_id"], "订单已创建")

    return order

重构后：较好的写法

from typing import Callable, Literal, TypedDict

class OrderInput(TypedDict):
    user_id: str
    amount: int
    level: Literal["normal", "vip"]

def validate_order_input(input: OrderInput) -> None:
    """只负责校验订单输入，输入规则变化时优先改这里。"""
    if input["amount"] <= 0:
        raise ValueError("金额必须大于 0")

def calculate_payable(input: OrderInput) -> float:
    """只负责计算应付金额，折扣规则变化时优先改这里。"""
    return input["amount"] * 0.9 if input["level"] == "vip" else input["amount"]

def build_order(input: OrderInput) -> dict:
    """只负责组装订单对象，字段结构变化时优先改这里。"""
    return {
        "user_id": input["user_id"],
        "amount": input["amount"],
        "payable": calculate_payable(input),
        "status": "created",
    }

def create_order(input: OrderInput, save: Callable[[dict], None], notify: Callable[[str], None]) -> dict:
    """负责串联下单流程，保存和通知通过参数传入，方便替换和测试。"""
    validate_order_input(input)
    order = build_order(input)
    save(order)
    notify(input["user_id"])
    return order

好在哪里：Python 版本没有为了模式而创建一堆类，只用函数和可传入的回调把变化点隔开。等保存、通知、折扣真的变复杂，再考虑协议、策略或适配器。

这个拆法仍然很克制：没有急着上策略、仓储、事件总线。只有当折扣规则继续增加、保存方式需要替换、通知失败要补偿时，才继续引入更明确的模式或架构层。

7. 优缺点

收益代价
代码更容易理解、测试、扩展。需要避免一次性大改。
能纠正 AI 生成代码的结构问题。没有测试时容易引入回归。

8. 和设计模式的关系

重构设计模式
先发现具体坏味道。作为解决某类坏味道的工具。
不一定引入新抽象。可能增加结构和类。

9. 一个练习题

找一段 AI 生成的大函数，标出其中的校验、业务规则、外部调用和响应转换。

10. 参考答案或改进建议

先不要急着套模式。把函数按意图拆开，补测试，再判断重复变化点是否值得引入策略、适配器或用例层。

今日总结

AI 时代重构能力更重要。你不需要比 AI 写得快，但要能判断哪些代码正在变成未来的维护成本。
