代码坏味道与重构

AI 能快速生成代码，但也可能快速制造坏味道。程序员需要识别结构问题，并用小步重构把代码带回可维护状态。

今日目标

  • 识别 AI 常见坏味道：大函数、胖类、重复逻辑、错误依赖、过度抽象。
  • 知道坏味道和设计模式之间的关系。
  • 掌握小步、安全、可回滚的重构方法。

1. 这个能力解决什么问题

AI 生成代码通常先追求完成任务，不一定关注长期结构。坏味道不会马上报错，但会让下一次改动更贵。

2. 一个生活类比

房间临时堆东西最快，但堆久了找东西越来越难。重构就是在不改变用途的前提下重新收纳。

3. 适用场景

  • AI 一次生成了很长函数。
  • 多个文件出现重复逻辑。
  • 业务代码直接依赖数据库、SDK、页面框架。
  • 新增需求时总要改同一个大类。

4. 不适用场景

  • 还没有测试保护的高风险核心流程，先补测试。
  • 一次性脚本，维护周期很短。
  • 为了“看起来高级”而抽象。

5. 核心结构

坏味道常见处理
大函数按意图拆函数，先保留行为。
重复逻辑抽规则函数或策略。
胖类按变化原因拆职责。
错误依赖引入端口和适配器。
过度抽象删除没有变化支撑的层。

6. 示例代码

TypeScript 示例

function validateAmount(amount: number) {
  if (amount <= 0) throw new Error("金额必须大于 0");
}

function createOrder(amount: number) {
  validateAmount(amount);
  return { amount, status: "created" };
}

Python 示例

def validate_amount(amount: int) -> None:
    if amount <= 0:
        raise ValueError("金额必须大于 0")

def create_order(amount: int) -> dict:
    validate_amount(amount)
    return {"amount": amount, "status": "created"}

7. 优缺点

收益代价
代码更容易理解、测试、扩展。需要避免一次性大改。
能纠正 AI 生成代码的结构问题。没有测试时容易引入回归。

8. 和设计模式的关系

重构设计模式
先发现具体坏味道。作为解决某类坏味道的工具。
不一定引入新抽象。可能增加结构和类。

9. 一个练习题

找一段 AI 生成的大函数，标出其中的校验、业务规则、外部调用和响应转换。

10. 参考答案或改进建议

先不要急着套模式。把函数按意图拆开，补测试，再判断重复变化点是否值得引入策略、适配器或用例层。

今日总结

AI 时代重构能力更重要。你不需要比 AI 写得快，但要能判断哪些代码正在变成未来的维护成本。
