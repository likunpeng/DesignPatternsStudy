上下文工程

上下文工程是把项目规则、架构边界、决策记录和代码约束组织起来，让 AI 在正确的信息环境中工作。

AI 工程能力 Context Engineering TypeScript + Python AI 协作上下文

今日目标

  • 理解上下文质量决定 AI 产出质量。
  • 知道 AGENTS、README、ADR、模块说明各放什么。
  • 能写一段模块级 AI 约束。

1. 这个能力解决什么问题

AI 不知道项目历史、模块边界和团队约定时，会按通用经验乱写。上下文工程让规则可见、可复用、可执行。

2. 一个生活类比

新同事入职需要项目文档、代码规范和业务背景。AI 也需要这些上下文，只是它读得更快。

3. 适用场景

  • 项目长期使用 AI 协作。
  • 多人维护同一代码库。
  • 架构规则经常被误改。
  • 希望 AI 自动遵守目录和依赖约束。

4. 不适用场景

  • 一次性实验。
  • 没有稳定规则的早期探索。

5. 核心结构

文档职责
AGENTS.mdAI 工作规则和项目约束。
README项目入口和运行方式。
ADR重要架构决策原因。
模块 README模块职责、依赖、禁止事项。
测试说明验证方式和质量门槛。

6. 示例代码

上下文工程的关键不是“存一份规则对象”，而是让 AI 真能据此行动。下面用订单模块举例，展示没有上下文时 AI 容易怎么写，以及有上下文后应该给它什么约束。

TypeScript 示例

没有上下文时，AI 容易直接跨层访问

async function cancelOrder(orderId: string) {
  const order = await db.orders.findById(orderId);
  await db.inventory.restore(order.items);
  await db.payments.refund(order.paymentId);
  order.status = "cancelled";
  await db.orders.save(order);
}

这段代码的危险点是：订单模块直接摸库存和支付数据库，短期看很快，长期会把模块边界写穿。

给 AI 的模块上下文应该长这样

const orderModuleContext = `
# order 模块约束

## 职责
- 订单创建、取消、支付后状态更新。
- 维护订单状态机，不负责库存扣减和支付退款的具体实现。

## 禁止事项
- 禁止从 order 模块直接 import payment/infrastructure。
- 禁止直接访问 inventory 数据表。
- 禁止在用例里直接调用第三方支付 SDK。

## 协作方式
- 支付退款通过 PaymentPort。
- 库存恢复通过 InventoryPort。
- 跨模块数据只能通过公开 API 或端口传递。

## 改动要求
- 修改订单状态流转时，必须补状态机测试。
- 修改取消订单流程时，必须补幂等测试。
`;

AI 在约束下更应该产出的代码方向

interface PaymentPort {
  refund(paymentId: string): Promise<void>;
}

interface InventoryPort {
  restore(items: string[]): Promise<void>;
}

class CancelOrderUseCase {
  constructor(
    private readonly paymentPort: PaymentPort,
    private readonly inventoryPort: InventoryPort
  ) {}

  async execute(orderId: string) {
    // 订单模块通过端口协作，而不是直接跨模块改数据库。
    return orderId;
  }
}

这里的目标不是把代码写满，而是让读者看到：上下文工程会改变 AI 的默认实现方向，从“直接连数据库”变成“通过端口协作”。

Python 示例

Python 里可以把模块上下文作为普通文本维护

ORDER_MODULE_CONTEXT = """
# order 模块约束

职责：订单创建、取消、支付后状态更新。
禁止：直接访问 payment.infrastructure、inventory.database、第三方支付 SDK。
协作：退款通过 PaymentPort，库存恢复通过 InventoryPort。
测试：状态流转必须有测试，取消订单必须覆盖重复请求。
"""

把上下文放进 AI 任务说明

def build_ai_task(task: str) -> str:
    return f"""
请在以下模块约束内完成任务。

{ORDER_MODULE_CONTEXT}

任务：{task}

先说明你会读取哪些文件和不会修改哪些边界，再给出实现步骤。
"""

Python 这里更自然的价值是把上下文快速拼装成任务说明、校验脚本或生成器，而不是复制一套很重的类型结构。

7. 优缺点

收益代价
AI 更容易遵守项目规则。上下文需要持续更新。
减少重复解释。文档过多会造成噪音。

8. 和普通文档的区别

普通文档上下文工程
主要给人读。同时给人和 AI 执行。
偏说明。偏约束、入口、决策和验证。

9. 一个练习题

为订单模块写一段 AI 约束，说明它负责什么、不能依赖什么、改动后必须测什么。再想一想：如果没有这段约束，AI 最容易犯什么错？

10. 参考答案或改进建议

订单模块负责订单生命周期；不能直接访问支付和库存数据库；跨模块通过端口或公开 API；修改状态流转必须补测试；新增外部依赖必须写适配器。没有这些约束时，AI 最常见的问题就是跨层直连和边界污染。

今日总结

上下文工程是 AI 时代的软件工程基础设施。规则写得清楚，AI 才能稳定地在正确边界内工作。你不是在“写文档”，而是在给 AI 建工作护栏。
