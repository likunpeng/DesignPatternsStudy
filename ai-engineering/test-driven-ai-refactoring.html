测试驱动的 AI 重构

AI 很适合做重构，但没有测试保护的 AI 重构风险很高。正确流程是先锁定行为，再小步改变结构。

AI 工程能力Test-Driven AI RefactoringTypeScript + Python安全重构

今日目标

  • 理解重构前为什么要先补测试。
  • 掌握 characterization test 的作用。
  • 学会让 AI 小步重构而不是一次性大改。

1. 这个方法解决什么问题

AI 可以快速移动代码、抽函数、拆模块，但它可能无意改变行为。测试驱动重构先把当前行为固定住，再让 AI 改结构。

2. 一个生活类比

维修桥梁前要先搭临时支撑和测量基准。没有支撑就施工，很容易塌。

3. 适用场景

  • 重构旧模块。
  • 拆大函数、大类。
  • 迁移架构层次。
  • AI 要批量修改多个文件。

4. 不适用场景

  • 明确要改变行为的新需求。
  • 实验性代码。
  • 没有可验证输出的临时脚本。

5. 核心结构

步骤做什么
记录现状补 characterization tests。
小步重构一次只改一个结构点。
持续验证每步运行测试。
评审边界检查依赖和职责是否改善。

6. 示例代码

TypeScript 示例

function price(amount: number, vip: boolean) {
  return vip ? amount * 0.9 : amount;
}

console.assert(price(100, true) === 90);
console.assert(price(100, false) === 100);

Python 示例

def price(amount: int, vip: bool) -> float:
    return amount * 0.9 if vip else amount

def test_price():
    assert price(100, True) == 90
    assert price(100, False) == 100

7. 优缺点

收益代价
AI 重构更安全。前期需要补测试。
行为不变更容易验证。旧代码行为本身错误时需要先确认期望。

8. 和直接重写的区别

直接重写测试驱动重构
风险高，容易改变行为。先锁行为，再改结构。
适合废弃重做。适合保留业务行为。

9. 一个练习题

一个折扣函数里有大量 if-else。请先写测试，再让 AI 改成策略模式。

10. 参考答案或改进建议

先覆盖普通用户、VIP、SVIP、边界金额等现有行为。测试通过后再抽策略。每抽一步都运行测试，避免 AI 在重构中改变折扣规则。

今日总结

AI 重构不是让 AI 一次性大改，而是用测试建立护栏，让 AI 在安全边界内逐步改善结构。
