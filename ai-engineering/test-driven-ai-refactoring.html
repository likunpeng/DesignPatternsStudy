测试驱动的 AI 重构

AI 很适合做重构，但没有测试保护的 AI 重构风险很高。正确流程是先锁定行为，再小步改变结构。

AI 工程能力 Test-Driven AI Refactoring TypeScript + Python 安全重构

今日目标

  • 理解重构前为什么要先补测试。
  • 掌握 characterization test 的作用。
  • 学会让 AI 小步重构而不是一次性大改。

1. 这个方法解决什么问题

AI 可以快速移动代码、抽函数、拆模块，但它可能无意改变行为。测试驱动重构先把当前行为固定住，再让 AI 改结构。

2. 一个生活类比

维修桥梁前要先搭临时支撑和测量基准。没有支撑就施工，很容易塌。

3. 适用场景

  • 重构旧模块。
  • 拆大函数、大类。
  • 迁移架构层次。
  • AI 要批量修改多个文件。

4. 不适用场景

  • 明确要改变行为的新需求。
  • 实验性代码。
  • 没有可验证输出的临时脚本。

5. 核心结构

步骤做什么
记录现状补 characterization tests。
小步重构一次只改一个结构点。
持续验证每步运行测试。
评审边界检查依赖和职责是否改善。

6. 示例代码

下面用一个典型的折扣函数做例子。重点不是“写个测试函数”，而是展示重构前的大分支、先锁定现有行为、再逐步抽出策略的过程。

TypeScript 示例

重构前：业务能跑，但函数开始变脏

function calculatePrice(amount: number, userType: "normal" | "vip" | "svip") {
  if (userType === "svip") {
    return amount > 200 ? amount * 0.7 : amount * 0.8;
  }
  if (userType === "vip") {
    return amount > 100 ? amount * 0.85 : amount * 0.9;
  }
  return amount;
}

先补 characterization tests

console.assert(calculatePrice(80, "normal") === 80);
console.assert(calculatePrice(80, "vip") === 72);
console.assert(calculatePrice(150, "vip") === 127.5);
console.assert(calculatePrice(150, "svip") === 120);
console.assert(calculatePrice(300, "svip") === 210);

测试护栏建立后，再让 AI 小步重构

type UserType = "normal" | "vip" | "svip";
type PriceStrategy = (amount: number) => number;

const strategies: Record<UserType, PriceStrategy> = {
  normal: (amount) => amount,
  vip: (amount) => (amount > 100 ? amount * 0.85 : amount * 0.9),
  svip: (amount) => (amount > 200 ? amount * 0.7 : amount * 0.8),
};

function calculatePriceRefactored(amount: number, userType: UserType) {
  return strategies[userType](amount);
}

这里最重要的是顺序：先测旧行为，再抽策略。如果先让 AI “优化成设计模式”，它很可能顺手把折扣规则也改了。

Python 示例

先把当前行为写进测试

def calculate_price(amount: int, user_type: str) -> float:
    if user_type == "svip":
        return amount * 0.7 if amount > 200 else amount * 0.8
    if user_type == "vip":
        return amount * 0.85 if amount > 100 else amount * 0.9
    return amount

def test_calculate_price():
    assert calculate_price(80, "normal") == 80
    assert calculate_price(80, "vip") == 72
    assert calculate_price(150, "vip") == 127.5
    assert calculate_price(150, "svip") == 120
    assert calculate_price(300, "svip") == 210

再做结构重构

from typing import Callable

PriceStrategy = Callable[[int], float]

strategies: dict[str, PriceStrategy] = {
    "normal": lambda amount: amount,
    "vip": lambda amount: amount * 0.85 if amount > 100 else amount * 0.9,
    "svip": lambda amount: amount * 0.7 if amount > 200 else amount * 0.8,
}

def calculate_price_refactored(amount: int, user_type: str) -> float:
    return strategies[user_type](amount)

Python 这类重构很适合先用函数和字典完成。只有当折扣规则越来越复杂，才需要升级成类、配置对象或策略实例。

7. 优缺点

收益代价
AI 重构更安全。前期需要补测试。
行为不变更容易验证。旧代码行为本身错误时需要先确认期望。

8. 和直接重写的区别

直接重写测试驱动重构
风险高，容易改变行为。先锁行为，再改结构。
适合废弃重做。适合保留业务行为。

9. 一个练习题

一个折扣函数里有大量 if-else。请先列出你会写的 characterization tests，再决定是否把它改成策略模式。注意：不是所有 if-else 都必须重构。

10. 参考答案或改进建议

先覆盖普通用户、VIP、SVIP、边界金额和异常输入等现有行为。测试通过后再抽策略。每抽一步都运行测试，避免 AI 在重构中顺手改变折扣规则。如果规则短且不再变化，也可以只补测试，不急着上模式。

今日总结

AI 重构不是让 AI 一次性大改，而是用测试建立护栏，让 AI 在安全边界内逐步改善结构。先锁行为，再谈优雅。
