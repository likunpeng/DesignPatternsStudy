代理模式学习笔记

代理模式解决的是“不能或不应该直接访问真实对象”的问题。代理和真实对象通常接口相同，但代理会在访问前后加入控制逻辑。

结构型模式Proxy PatternTypeScript + Python访问控制

今日目标

  • 理解代理不是简单中转，而是访问控制点。
  • 掌握虚拟代理、保护代理、缓存代理的典型场景。
  • 能写出缓存代理示例。
  • 能区分代理和装饰器。

1. 这个模式解决什么问题

真实对象可能很昂贵、很敏感、在远程，或需要统一控制访问。代理对象站在调用方和真实对象之间，决定是否转发请求、何时创建真实对象、是否直接返回缓存。

2. 一个生活类比

你想见一位高管，通常不会直接冲进办公室，而是先通过助理预约。助理不会替高管做所有事情，但会控制访问时机、过滤无效请求、安排日程。助理就是代理。

3. 适用场景

  • 远程服务调用，需要隐藏网络细节。
  • 大对象延迟加载，例如图片、报表、模型文件。
  • 需要权限校验、限流、审计。
  • 同一查询很频繁，可以用缓存代理减少真实调用。

4. 不适用场景

  • 真实对象访问成本低，也没有控制需求。
  • 只是想增加功能组合，更适合装饰器。
  • 代理里塞入大量业务逻辑，导致真实对象职责被掏空。

5. 核心结构

角色职责
Subject真实对象和代理共同遵守的接口。
RealSubject真正完成工作的对象。
Proxy持有或创建真实对象，并控制访问。
Client面向 Subject 编程，不直接依赖 RealSubject。

6. TypeScript 示例

interface UserRepository {
  findName(userId: string): string;
}

class RemoteUserRepository implements UserRepository {
  findName(userId: string): string {
    console.log("调用远程用户服务");
    return userId === "u1" ? "Alice" : "Unknown";
  }
}

class CachedUserRepositoryProxy implements UserRepository {
  private cache = new Map<string, string>();

  constructor(private real: UserRepository) {}

  findName(userId: string): string {
    if (!this.cache.has(userId)) {
      this.cache.set(userId, this.real.findName(userId));
    }
    return this.cache.get(userId)!;
  }
}

const repo = new CachedUserRepositoryProxy(new RemoteUserRepository());
console.log(repo.findName("u1"));
console.log(repo.findName("u1"));

7. Python 示例

from typing import Protocol


class UserRepository(Protocol):
    def find_name(self, user_id: str) -> str:
        ...


class RemoteUserRepository:
    def find_name(self, user_id: str) -> str:
        print("调用远程用户服务")
        return "Alice" if user_id == "u1" else "Unknown"


class CachedUserRepositoryProxy:
    def __init__(self, real: UserRepository) -> None:
        self.real = real
        self.cache: dict[str, str] = {}

    def find_name(self, user_id: str) -> str:
        if user_id not in self.cache:
            self.cache[user_id] = self.real.find_name(user_id)
        return self.cache[user_id]


repo = CachedUserRepositoryProxy(RemoteUserRepository())
print(repo.find_name("u1"))
print(repo.find_name("u1"))

8. 优缺点

优点缺点
调用方不需要知道远程、缓存、权限等细节。多一层间接调用，排查链路要看代理逻辑。
可以延迟创建昂贵对象。代理和真实对象接口要保持一致。
集中管理访问控制。如果代理做太多事，会变成隐藏的业务核心。

9. 和相似模式的区别

模式重点区别
代理控制访问决定是否访问真实对象，常见于权限、缓存、懒加载。
装饰器增强能力重点是叠加行为，通常可多层自由组合。
适配器接口转换改变接口形状，让不兼容对象可用。

10. 练习题

设计一个图片加载代理。真实图片对象加载很慢，代理在第一次调用 display() 时才创建真实图片对象。

参考思路：代理实现 Image.display()，内部保存文件路径和可选的真实图片实例，首次展示时再加载。

今日总结

代理模式的核心是“接口相同，访问受控”。当你需要缓存、权限、懒加载或远程调用隔离时，可以考虑代理。
