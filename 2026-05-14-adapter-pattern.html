2026-05-14 适配器模式学习笔记

适配器模式解决的是“已有对象能用，但接口不符合当前系统要求”的问题。它让新旧接口可以协作，而不是强行改动稳定代码。

结构型模式Adapter PatternTypeScript + Python接口转换

今日目标

  • 理解适配器模式为什么不是简单封装。
  • 能识别接口不兼容导致的耦合问题。
  • 能用 TypeScript 和 Python 实现一个最小适配器。
  • 知道适配器和外观、代理的区别。

1. 这个模式解决什么问题

当系统期望一种接口，但现有类、第三方库或遗留系统提供的是另一种接口时，就会出现“不兼容但有用”的对象。

调用方想要 pay(amount)，旧系统只有 charge(cents)。

如果到处写转换逻辑，调用方会同时依赖新接口和旧接口，后续替换第三方库会很痛苦。适配器把转换集中在一个对象里。

2. 一个生活类比

你的笔记本是 USB-C 接口，投影仪是 HDMI 接口。投影仪能用，电脑也能用，只是接口不匹配。转接头就是适配器：它不改变电脑，也不改变投影仪，只负责把一种接口转换成另一种接口。

3. 适用场景

  • 接入第三方 SDK，但它的方法名、参数单位或返回结构和系统约定不同。
  • 改造遗留系统，希望新代码使用统一接口。
  • 多个供应商能力类似，但接口细节不同。
  • 测试时需要把复杂外部服务适配成简单接口。

4. 不适用场景

  • 接口可以直接修改，而且没有外部依赖或历史包袱。
  • 只是想给方法改个名字，增加适配器反而让代码绕。
  • 业务规则本身不同，不只是接口形状不同。

5. 核心结构

角色职责
Target当前系统希望使用的目标接口。
Adaptee已有对象或第三方对象，功能可用但接口不匹配。
Adapter实现目标接口，并在内部调用 Adaptee 完成转换。
Client只依赖 Target，不关心背后接的是谁。

6. TypeScript 示例

interface PaymentGateway {
  pay(amount: number): void; // 单位：元
}

class LegacyPaySdk {
  charge(cents: number): void {
    console.log(`旧支付系统扣款 ${cents} 分`);
  }
}

class LegacyPayAdapter implements PaymentGateway {
  constructor(private sdk: LegacyPaySdk) {}

  pay(amount: number): void {
    this.sdk.charge(amount * 100);
  }
}

function checkout(gateway: PaymentGateway) {
  gateway.pay(99);
}

checkout(new LegacyPayAdapter(new LegacyPaySdk()));

checkout 只认识 PaymentGateway。旧 SDK 的单位转换被限制在 LegacyPayAdapter 内部。

7. Python 示例

from typing import Protocol


class PaymentGateway(Protocol):
    def pay(self, amount: float) -> None:
        ...


class LegacyPaySdk:
    def charge(self, cents: int) -> None:
        print(f"旧支付系统扣款 {cents} 分")


class LegacyPayAdapter:
    def __init__(self, sdk: LegacyPaySdk) -> None:
        self.sdk = sdk

    def pay(self, amount: float) -> None:
        self.sdk.charge(int(amount * 100))


def checkout(gateway: PaymentGateway) -> None:
    gateway.pay(99)


checkout(LegacyPayAdapter(LegacyPaySdk()))

Python 更自然地使用鸭子类型或 Protocol。只要对象有 pay 方法，就能被 checkout 使用。

8. 优缺点

优点缺点
隔离第三方或遗留接口，调用方更稳定。类数量增加，简单场景可能显得多余。
转换逻辑集中，便于替换外部实现。适配器过多时，需要统一命名和组织。
符合开闭原则，新增供应商通常只加新适配器。只能解决接口不兼容，不能解决语义不一致。

9. 和相似模式的区别

模式重点区别
适配器接口转换已有对象接口不符合当前系统。
外观简化复杂子系统不是为了兼容接口，而是为了降低使用复杂度。
代理控制访问代理通常保持相同接口，增加权限、缓存、延迟加载等控制。

10. 练习题

你的系统统一使用 send(to, content) 发送短信，但某短信平台 SDK 只有 pushSms(phone, text, sign)。请设计一个适配器，让业务代码不依赖该 SDK。

参考思路：定义 SmsSender 接口，再实现 VendorSmsAdapter，把 send 的参数转换成平台需要的参数，并在适配器中补充签名。

今日总结

适配器模式的核心是：不要让调用方到处理解外部接口差异。把差异封装在适配器里，让业务代码依赖自己的稳定接口。
