MVP 架构模式

MVP 让 View 尽量被动，Presenter 负责展示逻辑和交互流程，因此更容易测试复杂表单和传统客户端页面。

架构学习MVPTypeScript + Python可测试表现逻辑

今日目标

  • 理解 MVP 如何解决胖 View 和胖 Controller。
  • 知道 Passive View 和 Supervising Controller 的差异。
  • 能写一个可测试 Presenter。

1. 这个模式解决什么问题

复杂表单里，校验、加载、提交、错误提示都写在 View 中会很难测试。MVP 把这些展示逻辑放进 Presenter，View 只暴露接口。

2. 一个生活类比

主持人 Presenter 控制节目流程，舞台 View 只负责呈现灯光、字幕和演员位置。舞台不决定节目流程。

3. 适用场景

  • 传统桌面、移动端、复杂表单。
  • View 框架绑定能力较弱。
  • 需要对交互逻辑做单元测试。

4. 不适用场景

  • 响应式绑定非常成熟的现代前端，MVVM 可能更自然。
  • 页面很简单，Presenter 会显得多余。

5. 核心结构

角色职责
View展示和转发用户事件。
Presenter处理展示逻辑，调用 Model 或 UseCase。
Model数据和业务能力。

6. 示例代码

TypeScript 示例

interface LoginView { showLoading(): void; showError(msg: string): void; }
class LoginPresenter {
  constructor(private view: LoginView) {}
  submit(email: string) {
    if (!email.includes("@")) return this.view.showError("邮箱不合法");
    this.view.showLoading();
  }
}

Python 示例

from typing import Protocol

class LoginView(Protocol):
    def show_loading(self) -> None: ...
    def show_error(self, msg: str) -> None: ...

class LoginPresenter:
    def __init__(self, view: LoginView):
        self.view = view
    def submit(self, email: str) -> None:
        if "@" not in email:
            self.view.show_error("邮箱不合法")
            return
        self.view.show_loading()

7. 优缺点

优点缺点
Presenter 易测试，View 更薄。接口和样板代码较多。
适合复杂交互。双向调用关系需要管理。

8. 和 MVC/MVVM 的区别

模式区别
MVCController 常直接协调 Model 和 View。
MVVMView 绑定 ViewModel 状态，减少手动调用 View。

9. 一个练习题

把注册表单的校验、加载、错误提示拆到 Presenter 中。

10. 参考答案或改进建议

View 暴露 showLoading、showError、showSuccess；Presenter 校验输入并调用注册用例；注册规则复杂时应继续下沉到 UseCase 或 Domain。

今日总结

MVP 的关键是让 View 被动化。AI 生成客户端页面时，可以要求交互流程进入 Presenter，而不是散落在 UI 事件里。
