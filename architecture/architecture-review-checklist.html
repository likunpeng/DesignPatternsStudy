架构评审清单

架构评审清单把“感觉不对”变成可检查的问题，尤其适合审查 AI 生成或重构后的代码。

架构学习Architecture Review ChecklistTypeScript + Python质量评审

今日目标

  • 建立一套通用架构评审问题。
  • 能识别边界破坏、依赖倒置失败、过度抽象。
  • 学会把清单用于 AI 代码审查。

1. 这个清单解决什么问题

架构问题常常不表现为立刻报错，而是未来难改、难测、难拆。清单帮助你在合并前发现长期风险。

2. 一个生活类比

飞机起飞前要按清单检查，不是因为飞行员不专业，而是复杂系统不能只靠记忆。架构评审也是一样。

3. 适用场景

  • 新增模块、重构模块、接入外部系统。
  • AI 一次性修改多个文件。
  • 准备拆服务或调整目录结构。
  • 核心业务流程上线前。

4. 不适用场景

  • 纯文案、样式、配置的小改动。
  • 无长期维护价值的一次性实验。

5. 核心结构

维度问题
边界这个模块负责什么，不负责什么？
依赖源码依赖方向是否正确？
规则业务规则是否放在领域或用例层？
外部系统是否通过端口和适配器隔离？
测试核心规则和关键流程是否有测试？
演进新增变化是否更容易，而不是更难？

6. 示例代码

TypeScript 示例

type ReviewItem = { question: string; passed: boolean; note?: string };

const checklist: ReviewItem[] = [
  { question: "业务层是否避免直接依赖数据库客户端？", passed: true },
  { question: "新增第三方 SDK 是否被适配器隔离？", passed: false, note: "短信 SDK 直接出现在 use case 中" },
];

const failed = checklist.filter((item) => !item.passed);

Python 示例

checklist = [
    {"question": "业务规则是否集中？", "passed": True},
    {"question": "外部 SDK 是否被隔离？", "passed": False, "note": "支付 SDK 泄漏到用例层"},
]

failed = [item for item in checklist if not item["passed"]]

7. 优缺点

优点缺点
评审更稳定，减少主观争论。清单不能替代业务判断。
适合 AI 生成代码后的验收。过度机械执行会拖慢小改动。

8. 和代码 Review 的区别

代码 Review架构评审
关注实现正确性、可读性、Bug关注边界、依赖、演进和长期成本
通常针对改动文件可能需要看模块和系统上下文

9. 一个练习题

审查“新增支付渠道”方案，列出至少 6 个架构评审问题。

10. 参考答案或改进建议

是否定义 PaymentGateway 端口；是否避免业务层直接依赖 SDK；是否有失败重试和幂等；是否有回调验签；是否有测试；是否记录 ADR；是否影响已有支付方式。

今日总结

架构评审清单的价值，是把长期风险提前暴露。AI 写得越快，越需要稳定的评审标准。
