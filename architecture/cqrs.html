CQRS 读写分离

CQRS 把改变系统状态的命令和读取数据的查询分开。它适合读写复杂度差异明显的业务，但不应该滥用到所有 CRUD。

架构学习CQRSTypeScript + Python读写分离

今日目标

  • 理解 Command 和 Query 的职责差异。
  • 知道轻量 CQRS 和复杂 CQRS 的区别。
  • 判断什么时候不需要 CQRS。

1. 这个架构解决什么问题

很多系统写入需要严格业务规则，读取需要复杂聚合和性能优化。把读写强行放一个模型里，会让模型又复杂又难优化。

2. 一个生活类比

银行柜台办理转账需要严格校验，这是命令；手机 App 查看流水需要快速展示，这是查询。两者关注点不同。

3. 适用场景

  • 写入规则复杂，查询展示复杂。
  • 读多写少，读模型需要缓存或宽表。
  • 命令流程需要审计和权限控制。

4. 不适用场景

  • 简单 CRUD。
  • 读写模型几乎一样。
  • 团队不想处理数据同步和一致性。

5. 核心结构

角色职责
Command表达要改变什么。
Command Handler执行业务规则和状态变化。
Query表达要查询什么。
Read Model为展示优化的数据结构。

6. 示例代码

TypeScript 示例

type CreateOrderCommand = { userId: string; amount: number };
class CreateOrderHandler { handle(command: CreateOrderCommand) { return "order-id"; } }

type OrderListQuery = { userId: string };
class OrderQueryService { list(query: OrderListQuery) { return [{ id: "1", amountText: "99 元" }]; } }

Python 示例

class CreateOrderHandler:
    def handle(self, command: dict) -> str:
        return "order-id"

class OrderQueryService:
    def list(self, query: dict) -> list[dict]:
        return [{"id": "1", "amountText": "99 元"}]

7. 优缺点

优点缺点
写入规则和读取展示各自清晰。模型变多，复杂度增加。
读模型可以针对性能优化。可能引入最终一致性。

8. 和普通分层的区别

普通分层CQRS
同一服务同时处理读写。命令和查询分开建模。

9. 一个练习题

为“订单列表”和“创建订单”分别设计 Query 和 Command。

10. 参考答案或改进建议

CreateOrderCommand 包含用户、商品、数量；OrderListQuery 包含用户、状态、分页。创建订单返回订单 ID，列表查询返回展示模型。

今日总结

CQRS 的价值是分离不同复杂度。读写一样简单时不要用；读写差异明显时，它能让模型更清楚。
