MVVM 架构模式

MVVM 用 ViewModel 承载展示状态和交互逻辑，让 View 通过绑定或订阅状态来更新界面。

架构学习MVVMTypeScript + Python状态驱动 UI

今日目标

  • 理解 ViewModel 和 Model 的区别。
  • 知道哪些状态应该放 ViewModel，哪些应该放 Domain。
  • 避免 ViewModel 变成新的上帝对象。

1. 这个模式解决什么问题

现代 UI 常由状态驱动。MVVM 把页面展示状态、加载状态、错误状态和交互命令放进 ViewModel，View 只负责绑定和展示。

2. 一个生活类比

仪表盘 View 显示速度、油量、警告灯；ViewModel 像把汽车内部数据转换成仪表盘可展示状态的控制模块。

3. 适用场景

  • React、Vue、SwiftUI、Android 等状态驱动 UI。
  • 页面状态复杂，有加载、筛选、分页、错误。
  • 需要把展示状态和 UI 组件分离。

4. 不适用场景

  • 简单静态页面。
  • 把所有业务规则都塞进 ViewModel。
  • ViewModel 直接依赖数据库或 SDK。

5. 核心结构

角色职责
View绑定状态、触发命令。
ViewModel展示状态、交互命令、派生数据。
Model/UseCase业务规则和数据能力。

6. 示例代码

TypeScript 示例

type User = { name: string };
class UserViewModel {
  loading = false;
  error = "";
  user: User | null = null;
  constructor(private loadUser: () => Promise<User>) {}
  async init() {
    this.loading = true;
    try { this.user = await this.loadUser(); }
    catch { this.error = "加载失败"; }
    finally { this.loading = false; }
  }
}

Python 示例

class UserViewModel:
    def __init__(self, load_user):
        self.load_user = load_user
        self.loading = False
        self.error = ""
        self.user = None
    def init(self):
        self.loading = True
        try:
            self.user = self.load_user()
        except Exception:
            self.error = "加载失败"
        finally:
            self.loading = False

7. 优缺点

优点缺点
适合状态驱动 UI，View 更简单。ViewModel 容易膨胀。
展示状态易测试。复杂业务仍需 UseCase/Domain 承接。

8. 和 MVP 的区别

MVPMVVM
Presenter 主动调用 View 方法。View 绑定 ViewModel 状态。
适合被动 View。适合响应式 UI。

9. 一个练习题

为“订单列表页”设计 ViewModel 状态：加载、列表、筛选条件、错误、分页。

10. 参考答案或改进建议

ViewModel 可以持有 query、loading、error、orders、page，并提供 load、refresh、changeFilter 命令。订单能否取消这类核心规则不应只放 ViewModel。

今日总结

MVVM 的核心是展示状态模型。AI 生成页面时，应要求 ViewModel 管状态和交互，业务规则继续交给 UseCase 或 Domain。
