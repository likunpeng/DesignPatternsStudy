六边形架构

六边形架构也叫端口与适配器架构。它把业务核心放在中间，所有外部输入输出都通过端口和适配器连接。

架构学习Hexagonal ArchitectureTypeScript + Python端口与适配器

今日目标

  • 理解端口和适配器如何隔离外部系统。
  • 知道输入端口和输出端口的区别。
  • 能把数据库、HTTP、消息、第三方 SDK 放到外侧。

1. 这个架构解决什么问题

业务代码经常被 Web 框架、数据库、消息队列、第三方 SDK 绑死。六边形架构让业务核心只依赖端口，外部技术通过适配器接入。

2. 一个生活类比

电脑主机有 USB、HDMI、电源等标准接口。外设可以换，但主机核心不需要知道每个外设内部怎么工作。

3. 适用场景

  • 需要替换数据库、支付、短信等外部实现。
  • 同一业务需要支持 HTTP、CLI、消息消费等多个入口。
  • 希望核心业务可以脱离框架测试。

4. 不适用场景

  • 简单 CRUD，没有外部依赖变化。
  • 团队无法维护端口命名和适配器组织。

5. 核心结构

角色职责
Application Core用例和领域规则。
Input Port外部触发业务的入口。
Output Port业务需要调用的外部能力抽象。
Adapter把 HTTP、DB、SDK 转成端口调用。

6. 示例代码

TypeScript 示例

type Order = { id: string; amount: number };
interface OrderRepository { save(order: Order): Promise<void>; }

class CreateOrderUseCase {
  constructor(private orders: OrderRepository) {}
  async execute(amount: number) {
    const order = { id: crypto.randomUUID(), amount };
    await this.orders.save(order);
    return order;
  }
}

Python 示例

from typing import Protocol
from uuid import uuid4

class OrderRepository(Protocol):
    def save(self, order: dict) -> None: ...

class CreateOrderUseCase:
    def __init__(self, orders: OrderRepository):
        self.orders = orders
    def execute(self, amount: int) -> dict:
        order = {"id": str(uuid4()), "amount": amount}
        self.orders.save(order)
        return order

7. 优缺点

优点缺点
业务核心稳定，外部实现可替换。端口和适配器数量增加。
适合 AI 生成代码时约束依赖方向。简单项目可能显得绕。

8. 和 Clean Architecture 的区别

Clean Architecture六边形架构
强调同心圆和内外层依赖。强调端口与适配器连接外部世界。

9. 一个练习题

为短信发送设计一个输出端口和两个适配器：阿里云短信、腾讯云短信。

10. 参考答案或改进建议

定义 SmsSender.send(phone, content) 端口；业务只依赖端口；两个云厂商 SDK 分别放到 adapter 中实现端口。

今日总结

六边形架构的核心是隔离外部世界。端口属于业务需要，适配器属于技术实现。
