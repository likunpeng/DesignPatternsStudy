1. 架构学习解决什么问题

设计模式通常解决局部代码变化，架构解决的是更大范围的问题：模块怎么分、依赖怎么走、业务规则放哪里、外部系统怎么隔离、团队怎么并行开发、系统怎么测试和演进。

如果只会写功能，不会设计边界，项目变大后常见问题是：

  • Controller、Service、工具类越来越胖。
  • 业务逻辑散落在页面、接口、数据库脚本和消息消费里。
  • 换数据库、换支付 SDK、拆服务时牵一发而动全身。
  • 测试只能依赖真实外部环境，回归成本越来越高。

2. 一个生活类比

设计模式像装修技巧：怎么做隔断、怎么收纳、怎么走线。架构像楼房设计：承重墙在哪里、上下水怎么走、消防通道怎么留、以后扩建会不会影响主体结构。

装修技巧很重要，但如果楼房结构错了，后面每次改造都会非常痛苦。

4. 不建议一开始学什么

  • 不要一开始就钻微服务治理。没有模块边界能力，微服务只会把复杂度放大。
  • 不要先背 DDD 名词。先从订单、支付、库存这类业务规则里找变化点。
  • 不要只看架构图。架构图必须能对应到代码目录、依赖方向、测试方式和发布方式。
  • 不要把架构当模板。小项目用重架构会拖慢交付，大项目没边界会拖垮维护。

6. 示例代码

下面用一个“创建订单”的最小例子说明：架构不是把代码拆得越多越好，而是让业务规则不直接绑死数据库和通知渠道。

TypeScript 示例

type OrderInput = { userId: string; amount: number };
type Order = { id: string; userId: string; amount: number };

interface OrderRepository {
  save(order: Order): void;
}

interface Notifier {
  send(userId: string, message: string): void;
}

class CreateOrderUseCase {
  constructor(
    private orders: OrderRepository,
    private notifier: Notifier,
  ) {}

  execute(input: OrderInput): Order {
    if (input.amount <= 0) {
      throw new Error("订单金额必须大于 0");
    }

    const order = { id: crypto.randomUUID(), ...input };
    this.orders.save(order);
    this.notifier.send(input.userId, "订单创建成功");
    return order;
  }
}

CreateOrderUseCase 是用例层，负责业务流程；OrderRepository 和 Notifier 是端口，真正的数据库和短信服务可以在外层实现。

Python 示例

from dataclasses import dataclass
from typing import Protocol
from uuid import uuid4


@dataclass
class Order:
    id: str
    user_id: str
    amount: int


class OrderRepository(Protocol):
    def save(self, order: Order) -> None:
        ...


class Notifier(Protocol):
    def send(self, user_id: str, message: str) -> None:
        ...


class CreateOrderUseCase:
    def __init__(self, orders: OrderRepository, notifier: Notifier):
        self.orders = orders
        self.notifier = notifier

    def execute(self, user_id: str, amount: int) -> Order:
        if amount <= 0:
            raise ValueError("订单金额必须大于 0")

        order = Order(str(uuid4()), user_id, amount)
        self.orders.save(order)
        self.notifier.send(user_id, "订单创建成功")
        return order

Python 可以用 Protocol 表达端口，也可以在更小的项目里直接传函数。重点不是形式，而是业务规则不要直接写死外部实现。

7. 优缺点

收益代价
业务规则更集中，变化点更容易定位。初期需要多写接口、目录和测试替身。
外部依赖可替换，测试可以脱离真实数据库和 SDK。简单 CRUD 项目可能显得绕。
模块边界清楚，团队协作更稳定。边界划错后也会增加沟通成本。

8. 和设计模式学习的区别

维度设计模式架构学习
范围类、函数、对象协作模块、子系统、团队和部署单元
关注点局部变化点整体边界和演进方向
产出代码结构架构图、目录结构、依赖规则、测试策略
失败表现某段代码难改整个系统难测、难拆、难协作

9. 一个练习题

选择一个你熟悉的小业务，例如“用户下单”“登录注册”“文章发布”。请画出它的四层结构：

  • 入口层：HTTP 接口、页面事件或命令行入口。
  • 用例层：这个业务动作的主流程。
  • 领域层：核心规则和核心对象。
  • 基础设施层：数据库、缓存、第三方 SDK、消息队列。

10. 参考答案或改进建议

以下单为例，一个合理的第一版结构可以是：

order/
  api/                 # Controller 或路由入口
  application/         # CreateOrderUseCase
  domain/              # Order、OrderItem、PriceRule
  infrastructure/      # MysqlOrderRepository、SmsNotifier

判断是否合理，看三个问题：业务规则是否集中在用例和领域层；外部依赖是否被基础设施层隔离；测试用例是否可以不启动真实数据库也能验证核心流程。

今日总结

架构学习的主线是“边界”。AI 协作会放大边界清晰度的价值：边界越清楚，AI 越容易生成正确代码；边界越混乱，AI 越容易扩散复杂度。下一步建议学习“AI 时代架构方法论”，先掌握如何给 AI 设定架构约束，再进入分层架构、Clean Architecture 和 DDD。
