Clean Architecture

Clean Architecture 的重点是保护业务规则，让框架、数据库、UI 和第三方服务都成为可替换的外层细节。

架构学习Clean ArchitectureTypeScript + Python内外层依赖

今日目标

  • 理解依赖只能从外层指向内层。
  • 区分 Entity、Use Case、Interface Adapter、Framework。
  • 知道 Clean Architecture 什么时候值得用。

1. 这个架构解决什么问题

很多项目把业务规则写进 Controller、ORM Model 或页面组件里，导致换框架、换数据库、做测试时非常痛苦。Clean Architecture 把核心规则放在内层，让外层围绕它适配。

2. 一个生活类比

公司核心制度不应该写在某个办公软件里。换办公软件时，制度不应该跟着重写。制度是内层，工具是外层。

3. 适用场景

  • 业务规则复杂，生命周期长。
  • 外部技术可能变化。
  • 需要高质量自动化测试。
  • 多个入口复用同一业务流程。

4. 不适用场景

  • 纯展示型页面或简单后台。
  • 业务规则很薄，主要是表单 CRUD。
  • 团队无法接受额外层次和抽象。

5. 核心结构

层职责
Entities企业级业务规则和核心对象。
Use Cases应用级流程编排。
Interface AdaptersController、Presenter、Repository 实现。
Frameworks数据库、Web 框架、UI、外部服务。

6. 示例代码

TypeScript 示例

class Account {
  constructor(public id: string, public balance: number) {}
  withdraw(amount: number) {
    if (amount <= 0 || amount > this.balance) throw new Error("余额不足");
    this.balance -= amount;
  }
}

interface AccountRepo { get(id: string): Promise<Account>; save(account: Account): Promise<void>; }

class WithdrawUseCase {
  constructor(private accounts: AccountRepo) {}
  async execute(accountId: string, amount: number) {
    const account = await this.accounts.get(accountId);
    account.withdraw(amount);
    await this.accounts.save(account);
  }
}

Python 示例

from typing import Protocol

class Account:
    def __init__(self, id: str, balance: int):
        self.id = id
        self.balance = balance
    def withdraw(self, amount: int) -> None:
        if amount <= 0 or amount > self.balance:
            raise ValueError("余额不足")
        self.balance -= amount

class AccountRepo(Protocol):
    def get(self, id: str) -> Account: ...
    def save(self, account: Account) -> None: ...

7. 优缺点

优点缺点
核心规则独立、可测试、可复用。结构更重，初期开发慢。
技术替换成本低。边界设计需要经验。

8. 和分层架构的区别

维度分层架构Clean Architecture
重点按职责分层内层业务规则不依赖外层细节
依赖常见为上层调用下层源码依赖必须向内

9. 一个练习题

设计“取消订单”用例，说明订单状态规则、库存释放、退款调用分别属于哪一层。

10. 参考答案或改进建议

订单能否取消属于 Entity/Domain；取消流程属于 Use Case；库存和退款是端口；具体库存系统和支付 SDK 是外层适配器。

今日总结

Clean Architecture 的核心是依赖方向。业务规则越重要，越应该放在内层；框架和数据库越具体，越应该放在外层。
