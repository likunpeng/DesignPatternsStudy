BFF 架构

BFF 是 Backend for Frontend，为不同前端提供贴合页面和端能力的后端接口，避免前端直接拼多个后端服务。

架构学习Backend for FrontendTypeScript + Python前后端协作

今日目标

  • 理解 BFF 解决多端数据适配问题。
  • 知道 BFF 不应该承载核心业务规则。
  • 能设计一个页面聚合接口。

1. 这个架构解决什么问题

Web、App、小程序需要的数据形状不同。如果前端直接调多个后端服务，会出现重复拼装、权限分散、接口不稳定。BFF 在后端聚合并适配前端需要的数据。

2. 一个生活类比

同一份食材，堂食、外卖、儿童餐的打包方式不同。BFF 就像面向不同就餐场景的打包台。

3. 适用场景

  • 多端应用数据需求不同。
  • 页面需要聚合多个后端服务。
  • 前端不适合直接暴露复杂服务接口。

4. 不适用场景

  • 只有一个简单前端。
  • 后端接口已经刚好满足页面需求。
  • 把核心业务规则都塞进 BFF。

5. 核心结构

部分职责
Frontend展示和交互。
BFF聚合、裁剪、适配数据。
Domain Service核心业务能力。
Auth/Policy统一鉴权和接口保护。

6. 示例代码

TypeScript 示例

async function getOrderPage(orderId: string) {
  const order = await fetch(`/orders/${orderId}`).then((r) => r.json());
  const coupons = await fetch(`/coupons?user=${order.userId}`).then((r) => r.json());
  return { title: `订单 ${order.id}`, order, coupons };
}

Python 示例

def get_order_page(order: dict, coupons: list[dict]) -> dict:
    return {
        "title": f"订单 {order['id']}",
        "order": order,
        "coupons": coupons,
    }

7. 优缺点

优点缺点
前端更简单，接口更贴合页面。BFF 容易变成业务垃圾桶。
适合多端差异。需要治理重复聚合逻辑。

8. 和 API Gateway 的区别

BFFAPI Gateway
面向具体前端体验和数据形状。面向统一入口、路由、限流、鉴权。

9. 一个练习题

为“订单详情页”设计 Web BFF 和 App BFF 的差异。

10. 参考答案或改进建议

Web 可能需要更完整的操作区和表格字段；App 需要更轻量的数据和移动端按钮状态。核心订单规则仍在订单服务，不应放进 BFF。

今日总结

BFF 的价值是适配前端，而不是替代领域服务。它应聚合和裁剪数据，不应承载核心业务规则。
