依赖倒置与依赖注入

依赖倒置让高层业务不绑死底层实现；依赖注入则是把具体实现从外部传进来，让替换和测试变得可控。

架构学习Dependency Inversion & DITypeScript + Python端口与实现

今日目标

  • 区分依赖倒置原则和依赖注入手段。
  • 知道什么时候需要接口，什么时候直接传函数即可。
  • 能用端口隔离数据库、缓存、第三方 SDK。

1. 这个方法解决什么问题

业务代码直接 new 数据库客户端、支付 SDK 或邮件服务，会让测试和替换变困难。依赖倒置要求业务依赖抽象，具体实现依赖业务定义的端口。

2. 一个生活类比

插座定义了用电协议，电器不需要知道电来自火电、水电还是太阳能。电器依赖插座标准，不依赖发电厂实现。

3. 适用场景

  • 外部依赖可能替换，例如支付、短信、存储。
  • 核心业务需要单元测试。
  • 一个端口有多个实现，例如真实实现、内存实现、Mock 实现。

4. 不适用场景

  • 稳定且无替换价值的语言标准库。
  • 只有一个很简单的纯函数依赖。
  • 为所有类都建接口，导致阅读成本高于收益。

5. 核心结构

概念含义
高层模块业务用例、领域规则。
低层模块数据库、HTTP、消息、SDK。
端口高层定义的抽象能力。
注入启动或组装阶段传入具体实现。

6. 示例代码

TypeScript 示例

interface EmailSender { send(to: string, content: string): Promise<void>; }

class RegisterUserUseCase {
  constructor(private email: EmailSender) {}
  async execute(user: { email: string }) {
    // 保存用户省略
    await this.email.send(user.email, "欢迎注册");
  }
}

class SmtpEmailSender implements EmailSender {
  async send(to: string, content: string) { console.log("SMTP", to, content); }
}

const useCase = new RegisterUserUseCase(new SmtpEmailSender());

Python 示例

from typing import Protocol

class EmailSender(Protocol):
    def send(self, to: str, content: str) -> None: ...

class RegisterUserUseCase:
    def __init__(self, email: EmailSender):
        self.email = email
    def execute(self, email: str) -> None:
        self.email.send(email, "欢迎注册")

class ConsoleEmailSender:
    def send(self, to: str, content: str) -> None:
        print("send", to, content)

7. 优缺点

优点缺点
业务可测试，可替换实现。需要组装依赖。
外部系统变化被隔离。接口过多会显得啰嗦。

8. 和相似概念的区别

概念区别
依赖倒置设计原则，规定依赖方向。
依赖注入实现手段，把依赖从外部传入。
IoC 容器自动管理对象创建和注入，不是必须。

9. 一个练习题

把订单支付逻辑改成依赖 PaymentGateway 端口，而不是直接调用支付宝 SDK。

10. 参考答案或改进建议

在 application 层定义 PaymentGateway.pay(orderId, amount)，支付宝、微信支付各写一个 infrastructure 适配器。用例只依赖端口，测试时传入内存实现。

今日总结

依赖倒置的核心不是“到处写 interface”，而是业务规则不要被外部实现牵着走。依赖注入只是让这个原则真正落地。
