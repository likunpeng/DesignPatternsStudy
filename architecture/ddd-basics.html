DDD 入门

DDD 不是把所有类都改名成领域对象，而是用业务语言建立模型，把核心规则放到真正表达业务概念的代码里。

架构学习Domain-Driven Design BasicsTypeScript + Python领域建模

今日目标

  • 理解实体、值对象、聚合、领域服务的基本含义。
  • 知道 DDD 适合复杂业务，不适合所有 CRUD。
  • 能用订单场景做一个最小领域模型。

1. 这个方法解决什么问题

当业务规则散落在 Service、SQL、前端和定时任务里，系统就很难回答“订单到底什么时候能取消”。DDD 通过领域模型集中表达核心规则。

2. 一个生活类比

医院系统不能只说“表 A、表 B、状态字段”。医生关心的是患者、诊断、处方、复诊。领域语言让业务和代码说同一种话。

3. 适用场景

  • 业务规则复杂且经常变化。
  • 团队需要和业务专家长期沟通。
  • 系统有多个子域，例如订单、库存、支付、履约。

4. 不适用场景

  • 简单数据管理系统。
  • 没有复杂规则，主要是增删改查。
  • 团队把 DDD 当术语堆砌，不愿做业务分析。

5. 核心结构

概念说明
实体有唯一身份，生命周期变化，例如 Order。
值对象无身份，用值表达概念，例如 Money、Address。
聚合一致性边界，例如 Order 和 OrderItem。
领域服务不自然属于某个实体的领域规则。
仓储持久化聚合的抽象。

6. 示例代码

TypeScript 示例

class Money {
  constructor(public cents: number) { if (cents < 0) throw new Error("金额不能为负"); }
}

class Order {
  private status: "created" | "paid" | "cancelled" = "created";
  constructor(public id: string, private total: Money) {}
  cancel() {
    if (this.status === "paid") throw new Error("已支付订单不能直接取消");
    this.status = "cancelled";
  }
}

Python 示例

from dataclasses import dataclass

@dataclass(frozen=True)
class Money:
    cents: int
    def __post_init__(self):
        if self.cents < 0:
            raise ValueError("金额不能为负")

class Order:
    def __init__(self, id: str, total: Money):
        self.id = id
        self.total = total
        self.status = "created"
    def cancel(self) -> None:
        if self.status == "paid":
            raise ValueError("已支付订单不能直接取消")
        self.status = "cancelled"

7. 优缺点

优点缺点
业务规则集中，表达力强。建模需要业务理解。
代码语言贴近业务语言。简单系统会显得复杂。

9. 一个练习题

为“优惠券”建模：哪些是实体，哪些是值对象？“过期不能使用”规则应该放在哪里？

10. 参考答案或改进建议

Coupon 通常是实体；Money、DateRange 可以是值对象；是否可用应放在 Coupon 或领域服务中，而不是散落在 Controller。

今日总结

DDD 的起点不是目录，而是业务语言。只有当模型能表达业务规则时，DDD 才有价值。
