Flux/Redux 单向数据流

Flux/Redux 的核心是让状态变化按单一方向流动：View 发出 Action，状态按规则更新，再驱动 View 渲染。

架构学习Flux / ReduxTypeScript + Python前端状态架构

今日目标

  • 理解为什么复杂前端需要单向数据流。
  • 掌握 Action、Reducer、Store、View 的职责。
  • 知道什么时候不需要 Redux 这类方案。

1. 这个架构解决什么问题

复杂页面中，多个组件互相改状态会导致数据来源不清。单向数据流让状态变化可追踪、可回放、可测试。

2. 一个生活类比

报销流程不是每个人直接改账本，而是提交申请，财务按规则处理，账本更新后大家看到新状态。

3. 适用场景

  • 多个组件共享复杂状态。
  • 状态变化需要可追踪。
  • 需要撤销、回放、调试状态变化。

4. 不适用场景

  • 局部表单状态。
  • 简单页面只有少量组件。
  • 为了跟风引入全局状态。

5. 核心结构

角色职责
View展示状态，派发 Action。
Action描述发生了什么。
Reducer根据旧状态和 Action 生成新状态。
Store保存全局或模块状态。

6. 示例代码

TypeScript 示例

type State = { count: number };
type Action = { type: "increment" } | { type: "reset" };

function reducer(state: State, action: Action): State {
  if (action.type === "increment") return { count: state.count + 1 };
  if (action.type === "reset") return { count: 0 };
  return state;
}

Python 示例

def reducer(state: dict, action: dict) -> dict:
    if action["type"] == "increment":
        return {"count": state["count"] + 1}
    if action["type"] == "reset":
        return {"count": 0}
    return state

7. 优缺点

优点缺点
状态变化清晰、可测试。样板代码可能增加。
适合复杂共享状态。简单页面会过度设计。

8. 和 MVVM 的区别

MVVMFlux/Redux
ViewModel 管页面状态和命令。Action/Reducer/Store 管状态变化链路。
适合页面级状态。适合复杂共享状态。

9. 一个练习题

为购物车设计 Action 和 Reducer：添加商品、修改数量、清空购物车。

10. 参考答案或改进建议

Action 描述 addItem、changeQty、clearCart；Reducer 不直接请求接口，只根据 Action 更新状态。异步请求应放在副作用层或 use case。

今日总结

单向数据流的价值是让状态变化可追踪。AI 生成前端状态代码时，应该明确状态只能通过规定动作改变。
