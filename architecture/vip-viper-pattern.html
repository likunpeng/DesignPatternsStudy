VIP/VIPER 架构模式

VIP/VIPER 用更严格的单向职责拆分复杂页面，常见于 iOS、Android 和大型客户端模块。

架构学习VIP/VIPERTypeScript + Python复杂页面模块化

今日目标

  • 理解 VIP/VIPER 为什么比 MVC/MVVM 更重。
  • 知道 Interactor、Presenter、Router 的职责。
  • 判断什么时候值得使用。

1. 这个模式解决什么问题

复杂 App 页面可能同时有网络请求、状态转换、导航、弹窗、权限和业务流程。VIP/VIPER 把这些职责拆开，避免 ViewController 或 ViewModel 巨大化。

2. 一个生活类比

大型活动需要导演、主持、后勤、路线引导和现场执行分工。一个人全包会失控。

3. 适用场景

  • 复杂移动端页面或多步骤流程。
  • 页面需要强测试和多人协作。
  • 导航、展示、业务流程都比较重。

4. 不适用场景

  • 普通列表页或简单表单。
  • 团队无法维护大量样板代码。
  • Web 前端已有更合适的状态管理方案。

5. 核心结构

角色职责
View展示和收集输入。
Interactor业务用例和数据交互。
Presenter把结果转换成展示模型。
Router导航和模块组装。
Entity业务数据结构。

6. 示例代码

TypeScript 示例

type Order = { id: string; amount: number };
class OrderPresenter { present(order: Order) { return { title: `订单 ${order.id}`, amountText: `${order.amount} 元` }; } }
class OrderInteractor {
  constructor(private presenter: OrderPresenter) {}
  loadOrder() { return this.presenter.present({ id: "1", amount: 99 }); }
}

Python 示例

class OrderPresenter:
    def present(self, order: dict) -> dict:
        return {"title": f"订单 {order['id']}", "amountText": f"{order['amount']} 元"}

class OrderInteractor:
    def __init__(self, presenter: OrderPresenter):
        self.presenter = presenter
    def load_order(self) -> dict:
        return self.presenter.present({"id": "1", "amount": 99})

7. 优缺点

优点缺点
职责极清晰，复杂页面可测试。样板代码多，简单页面过度设计。
导航和业务流程分离。学习成本高。

8. 和 MVVM 的区别

MVVMVIP/VIPER
ViewModel 承载状态和交互。Interactor、Presenter、Router 进一步拆分。
更轻量。更严格，适合复杂模块。

9. 一个练习题

把“提交订单确认页”拆成 View、Interactor、Presenter、Router。

10. 参考答案或改进建议

View 收集确认操作；Interactor 调用提交订单用例；Presenter 生成成功/失败展示模型；Router 负责跳转支付页或结果页。

今日总结

VIP/VIPER 不是越复杂越高级，而是在页面足够复杂时，用严格分工换取可测试和可维护。
