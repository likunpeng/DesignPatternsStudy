AI 协作下的 UI 架构

AI 生成 UI 很快，但如果没有边界，它会把页面、状态、请求、权限和业务规则写成一团。UI 架构规则能约束生成空间。

架构学习AI-Assisted UI ArchitectureTypeScript + PythonAI 协作约束

今日目标

  • 学会给 AI 明确 UI 架构边界。
  • 知道生成页面时哪些逻辑不能放进 View。
  • 能写一份 UI 代码生成规则。

1. 这个方法解决什么问题

AI 常把请求、状态、权限和业务规则直接写进组件。短期能跑，长期难测、难复用、难重构。需要用 MVC/MVP/MVVM/VIP 思想给 AI 设边界。

2. 一个生活类比

让施工队装修前要先告诉它哪些是承重墙、哪些线不能动。AI 生成 UI 前也要告诉它哪些层不能混。

3. 适用场景

  • 让 AI 生成页面、组件、表单、移动端模块。
  • AI 重构胖组件。
  • 多人和 AI 混合维护前端项目。

4. 不适用场景

  • 一次性静态页面。
  • 简单视觉稿还原，不涉及业务交互。

5. 核心结构

规则说明
View只做展示和事件转发。
ViewModel/Presenter管理展示状态和交互命令。
UseCase承载业务流程。
Repository/API Client封装请求和数据转换。
Domain放核心业务规则。

6. 示例代码

TypeScript 示例

// 给 AI 的约束：组件不要直接 fetch，调用 ViewModel 命令。
class OrderPageViewModel {
  loading = false;
  constructor(private loadOrders: () => Promise<unknown[]>) {}
  async init() {
    this.loading = true;
    await this.loadOrders();
    this.loading = false;
  }
}

Python 示例

# BFF 或服务端渲染也要保持展示模型边界。

    return {

    }

7. 优缺点

优点缺点
AI 生成代码更稳定。需要维护项目级规则。
减少胖组件和隐形耦合。简单页面可能不需要完整分层。

8. 和普通 Prompt 的区别

普通 Prompt架构约束 Prompt
生成一个订单页按 View、ViewModel、API Client 分层生成订单页，组件不得直接请求接口

9. 一个练习题

写一段给 AI 的规则，让它生成“用户设置页”时不要把请求、状态和业务规则都写进组件。

10. 参考答案或改进建议

要求组件只负责展示；ViewModel 管理 loading、error、formState；API Client 封装请求；保存前校验放 UseCase 或独立规则函数；生成后必须补 ViewModel 测试。

今日总结

AI 协作下的 UI 架构重点，是把表现层边界写清楚。AI 可以快速生成代码，但人要负责定义哪些逻辑应该在哪里。
