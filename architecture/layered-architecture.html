分层架构

分层架构解决的是“代码职责混在一起”的问题。它用稳定的依赖方向，把入口、业务流程、核心规则和外部实现分开。

架构学习Layered ArchitectureTypeScript + Python职责分层

今日目标

  • 理解分层不是按文件夹好看地拆分，而是控制依赖方向。
  • 能判断 Controller、Application、Domain、Infrastructure 各放什么。
  • 知道小项目什么时候可以简化分层。

1. 这个架构解决什么问题

最常见的坏味道是接口层直接写业务规则、直接查数据库、直接调第三方 SDK。短期很快，长期会让测试、替换和复用都变难。

2. 一个生活类比

餐厅里顾客点餐、服务员传单、后厨做菜、采购进货是不同职责。让服务员顺手改菜单价格、进厨房做菜、联系供应商，流程会很快失控。

3. 适用场景

  • 业务流程不止简单 CRUD。
  • 同一业务需要被 HTTP、定时任务、消息消费复用。
  • 希望核心规则可测试，不依赖真实数据库。
  • 团队多人协作，需要统一放置规则。

4. 不适用场景

  • 一次性脚本或很薄的管理后台。
  • 只有数据录入和查询，业务规则极少。
  • 团队还无法维护抽象，强行四层会增加沟通成本。

5. 核心结构

层职责不该做的事
Interface/API参数解析、鉴权入口、响应转换写核心业务规则
Application编排用例流程、事务边界关心数据库细节
Domain实体、值对象、领域规则依赖框架和 SDK
Infrastructure数据库、缓存、消息、第三方适配决定业务规则

6. 示例代码

TypeScript 示例

type Order = { id: string; amount: number; status: "created" };
interface OrderRepository { save(order: Order): Promise<void>; }

class CreateOrderUseCase {
  constructor(private orders: OrderRepository) {}
  async execute(amount: number): Promise<Order> {
    if (amount <= 0) throw new Error("金额必须大于 0");
    const order = { id: crypto.randomUUID(), amount, status: "created" as const };
    await this.orders.save(order);
    return order;
  }
}

async function createOrderController(body: { amount: number }, useCase: CreateOrderUseCase) {
  const order = await useCase.execute(body.amount);
  return { code: 0, data: order };
}

Python 示例

from typing import Protocol
from uuid import uuid4

class OrderRepository(Protocol):
    def save(self, order: dict) -> None: ...

class CreateOrderUseCase:
    def __init__(self, orders: OrderRepository):
        self.orders = orders
    def execute(self, amount: int) -> dict:
        if amount <= 0:
            raise ValueError("金额必须大于 0")
        order = {"id": str(uuid4()), "amount": amount, "status": "created"}
        self.orders.save(order)
        return order

关键点：入口层只调用用例；用例依赖仓储接口；真实数据库实现放在基础设施层。

7. 优缺点

优点缺点
职责清晰，测试更容易。文件和接口数量增加。
外部实现可替换。简单 CRUD 可能过度设计。

8. 和相似架构的区别

架构重点
三层架构表现层、业务层、数据层，适合入门。
Clean Architecture更强调内层业务规则不依赖外层实现。
六边形架构更强调端口和适配器。

9. 一个练习题

把“用户注册”拆成 API、Application、Domain、Infrastructure 四层，说明密码规则、保存用户、发送欢迎邮件分别放在哪里。

10. 参考答案或改进建议

密码格式校验可以在用例入口做，密码强度和用户状态规则应靠近 Domain；保存用户是 Repository 端口；邮件发送是 Notifier 端口，由 Infrastructure 实现。

今日总结

分层架构的价值不是目录多，而是让变化有固定位置。入口变化不影响业务规则，数据库变化不影响用例，业务规则能独立测试。
