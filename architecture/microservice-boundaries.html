微服务边界

微服务不是把代码拆成很多仓库，而是围绕业务能力、数据归属和团队协作建立边界。边界不清，拆得越多越痛苦。

今日目标

  • 理解什么时候不该拆微服务。
  • 掌握按业务能力和数据所有权划分边界。
  • 知道服务间通信和一致性的基本取舍。

1. 这个方法解决什么问题

单体过大时，团队协作、发布和模块演进会互相阻塞。微服务通过服务边界降低协作耦合，但会引入网络、数据一致性和运维复杂度。

2. 一个生活类比

公司扩大后会分部门，但不是把每个人随机分开。财务、销售、仓储各有职责和数据。如果边界乱，跨部门沟通成本会爆炸。

3. 适用场景

  • 单体模块边界已经清晰。
  • 不同业务能力有独立发布需求。
  • 团队规模和组织边界需要拆分。
  • 某些模块有独立扩展性或稳定性要求。

4. 不适用场景

  • 业务早期快速试错。
  • 连单体内部边界都不清楚。
  • 团队没有自动化测试、监控和部署能力。

5. 核心结构

判断点问题
业务能力这个服务负责哪项完整业务能力？
数据归属谁拥有并修改这份数据？
通信方式同步调用还是异步事件？
一致性必须强一致还是最终一致？

6. 示例代码

TypeScript 示例

// 订单服务只发布事实，不直接修改库存库。

interface EventBus { publish(event: OrderPlaced): Promise<void>; }

class PlaceOrderUseCase {
  constructor(private bus: EventBus) {}
  async execute(input: OrderPlaced) {
    // 保存订单省略
    await this.bus.publish(input);
  }
}

Python 示例

from typing import Protocol

class EventBus(Protocol):
    def publish(self, event: dict) -> None: ...

class PlaceOrderUseCase:
    def __init__(self, bus: EventBus):
        self.bus = bus
    def execute(self, order_id: str, sku: str, qty: int) -> None:
        self.bus.publish({"type": "OrderPlaced", "orderId": order_id, "sku": sku, "qty": qty})

7. 优缺点

优点缺点
独立发布、独立扩展、团队自治。分布式事务、网络失败、观测成本增加。
业务边界更清楚。边界错了会形成分布式泥团。

8. 和模块化单体的区别

维度模块化单体微服务
部署一起部署独立部署
调用进程内调用网络调用或消息
适合阶段早中期更稳边界成熟后再拆

9. 一个练习题

电商系统里订单、库存、支付是否应该拆成服务？分别拥有哪份数据？

10. 参考答案或改进建议

订单拥有订单数据，库存拥有库存数据，支付拥有支付交易数据。订单不应直接改库存库，而是调用库存能力或发布事件。早期可以先做模块化单体，边界稳定后再拆服务。

今日总结

微服务的第一原则是边界，而不是技术栈。没有清晰业务能力和数据归属，就不要急着拆。
