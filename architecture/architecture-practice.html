架构综合实践

架构综合实践的目标，是把需求、边界、代码结构、测试和演进计划串起来，而不是只画一张漂亮架构图。

架构学习Architecture PracticeTypeScript + Python从需求到落地

今日目标

  • 掌握一份小型架构方案的基本结构。
  • 能从需求识别变化点和风险点。
  • 把架构设计落到目录、接口、测试和验收标准。

1. 这个实践解决什么问题

很多架构学习停留在概念。真正落地时，需要回答：为什么这样拆、代码放哪里、怎么测试、以后怎么扩展、AI 生成代码要遵守什么规则。

2. 一个生活类比

旅行计划不能只有目的地，还要有路线、预算、交通、备选方案和风险预案。架构方案也不能只有图。

3. 适用场景

  • 新模块启动前。
  • 旧模块重构前。
  • 引入外部系统或新业务流程前。
  • 需要让 AI 协助实现前。

4. 不适用场景

  • 非常小的修复或文案调整。
  • 需求还没形成基本共识。
  • 只需要技术 Spike，不需要长期方案。

5. 核心结构

部分要回答的问题
背景目标为什么做，成功标准是什么？
边界范围做什么，不做什么？
领域模型核心概念和规则是什么？
模块设计目录、依赖、接口怎么组织？
质量策略测试、日志、监控怎么覆盖？
演进计划哪些先做，哪些预留？

6. 示例代码

TypeScript 示例

// 方案落地时，先定义用例输入输出和端口。
interface TicketRepo { save(ticket: { id: string; title: string }): Promise<void>; }
interface Notifier { notify(userId: string, message: string): Promise<void>; }

class SubmitTicketUseCase {
  constructor(private tickets: TicketRepo, private notifier: Notifier) {}
  async execute(input: SubmitTicketInput) {
    if (!input.title.trim()) throw new Error("标题不能为空");
    await this.tickets.save({ id: crypto.randomUUID(), title: input.title });
    await this.notifier.notify(input.userId, "工单已提交");
  }
}

Python 示例

from typing import Protocol
from uuid import uuid4

class TicketRepo(Protocol):
    def save(self, ticket: dict) -> None: ...

class SubmitTicketUseCase:
    def __init__(self, tickets: TicketRepo):
        self.tickets = tickets
    def execute(self, title: str) -> None:
        if not title.strip():
            raise ValueError("标题不能为空")
        self.tickets.save({"id": str(uuid4()), "title": title})

7. 优缺点

优点缺点
设计能落到代码和测试。需要花时间做前置分析。
便于团队和 AI 按同一规则实现。方案需要随业务反馈调整。

8. 和只写技术方案的区别

普通技术方案架构实践方案
偏工具和实现步骤同时说明业务边界、依赖方向和演进成本
容易忽略测试和验收明确质量策略和完成标准

9. 一个练习题

为“工单系统”写一份小型架构方案：包括模块边界、核心实体、用例、外部依赖、测试策略。

10. 参考答案或改进建议

核心实体可以是 Ticket、Comment、Assignee；用例包括 SubmitTicket、AssignTicket、CloseTicket；外部依赖包括通知和附件存储；测试应覆盖状态流转、权限和通知失败。

今日总结

架构综合实践的关键是闭环：从需求到边界，从边界到代码，从代码到测试，从测试到演进。能落地的架构才有价值。
