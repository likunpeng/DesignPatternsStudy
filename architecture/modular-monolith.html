模块化单体

模块化单体是在一个部署单元内建立清晰业务模块边界。它通常比过早微服务更现实，也更适合多数中小团队。

架构学习Modular MonolithTypeScript + Python模块边界

今日目标

  • 理解模块化单体和普通单体的区别。
  • 掌握模块边界、公开接口、内部实现的划分。
  • 知道什么时候再考虑拆微服务。

1. 这个架构解决什么问题

普通单体容易变成所有模块互相调用、共享数据库表、边界不清。模块化单体保留单体部署简单性，同时用模块边界控制复杂度。

2. 一个生活类比

同一栋办公楼里可以有财务、人事、销售不同部门。它们共享大楼基础设施，但职责和资料边界清楚。

3. 适用场景

  • 业务还在演进，不适合过早拆服务。
  • 团队规模不大，但模块复杂度上升。
  • 希望未来可拆服务，但现在先稳定边界。

4. 不适用场景

  • 已有明确独立扩展和独立发布需求。
  • 团队能成熟运维多个服务且边界已稳定。

5. 核心结构

部分职责
Module API模块对外公开能力。
Application模块内部用例。
Domain模块核心规则。
Infrastructure模块内部持久化和适配器。

6. 示例代码

TypeScript 示例

// order/public-api.ts
export interface OrderModule {
  createOrder(input: { userId: string; amount: number }): Promise<string>;
}

// 其他模块只能依赖 public-api，不直接 import order/internal。

7. 优缺点

优点缺点
部署简单，边界比普通单体清楚。需要纪律维护模块依赖。
为未来拆服务打基础。无法解决所有独立扩展需求。

8. 和微服务的区别

模块化单体微服务
一个部署单元，进程内调用。多个部署单元，网络调用。
适合边界探索期。适合边界成熟后。

9. 一个练习题

把电商系统拆成订单、库存、支付三个模块，说明每个模块公开什么 API。

10. 参考答案或改进建议

订单公开 create/cancel/query；库存公开 reserve/release；支付公开 pay/refund/queryPayment。模块间不要直接访问彼此内部表和类。

今日总结

模块化单体是最常用、最务实的架构形态之一。先把模块边界做好，再考虑是否需要微服务。
