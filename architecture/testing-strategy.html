测试策略与架构质量

测试不是补作业，而是架构边界能否长期成立的保护网。没有测试，架构只能靠记忆和自觉维持。

今日目标

  • 理解不同测试层级保护不同架构边界。
  • 知道领域规则、用例流程、适配器、服务契约分别怎么测。
  • 学会给 AI 生成代码设置测试要求。

1. 这个策略解决什么问题

代码能跑不代表架构健康。业务规则可能散落，依赖方向可能倒置，外部适配可能没有保护。测试策略把这些风险变成可执行检查。

2. 一个生活类比

建筑不是盖完看起来漂亮就行，还要做承重、消防、电路和防水验收。测试就是软件架构的验收体系。

3. 适用场景

  • 业务规则有长期维护价值。
  • AI 或多人频繁改代码。
  • 系统接入外部服务、数据库、消息队列。
  • 重构前需要安全网。

4. 不适用场景

  • 一次性脚本。
  • 探索性原型。
  • 完全没有稳定规则的早期实验。

5. 核心结构

测试类型保护对象
单元测试领域规则、纯函数、值对象。
用例测试应用流程和端口协作。
集成测试数据库、缓存、第三方适配器。
契约测试服务间 API 或事件格式。
E2E 测试关键用户路径。

6. 示例代码

TypeScript 示例

class Order {
  constructor(public status: "created" | "paid" | "cancelled") {}
  cancel() {
    if (this.status === "paid") throw new Error("已支付不能取消");
    this.status = "cancelled";
  }
}

function testCancelCreatedOrder() {
  const order = new Order("created");
  order.cancel();
  console.assert(order.status === "cancelled");
}

Python 示例

class Order:
    def __init__(self, status: str):
        self.status = status
    def cancel(self) -> None:
        if self.status == "paid":
            raise ValueError("已支付不能取消")
        self.status = "cancelled"

def test_cancel_created_order():
    order = Order("created")
    order.cancel()
    assert order.status == "cancelled"

7. 优缺点

优点缺点
重构更安全，AI 改动更可控。需要持续维护测试数据和边界。
能暴露架构耦合问题。过多 E2E 会慢且脆弱。

8. 和只测功能的区别

只测功能架构测试策略
关注结果是否正确同时关注规则位置、依赖方向、外部适配
偏黑盒组合单元、用例、集成、契约测试

9. 一个练习题

为“订单取消”设计测试：领域规则测什么，用例测什么，集成测试测什么？

10. 参考答案或改进建议

领域测试验证不同状态能否取消；用例测试验证取消后保存订单并发布事件；集成测试验证仓储能正确持久化；如果有消息事件，应加契约测试保护事件格式。

今日总结

测试策略是架构的一部分。好的测试不是数量多，而是能保护核心规则、模块边界和关键集成点。
