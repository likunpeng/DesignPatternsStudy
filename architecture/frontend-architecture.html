前端架构

前端架构解决的是页面变复杂后，组件、状态、请求、权限和业务规则混在一起的问题。

架构学习Frontend ArchitectureTypeScript + Python页面与状态边界

今日目标

  • 理解前端不只是 UI，也有业务边界。
  • 知道页面、组件、状态、API Client、业务规则如何分工。
  • 避免把所有逻辑塞进组件。

1. 这个架构解决什么问题

复杂页面常见问题是组件又渲染 UI、又拼请求、又处理权限、又写业务判断。最后组件很难测试，也很难复用。

2. 一个生活类比

舞台表演中，演员负责表现，导演负责流程，后台负责道具。演员不应该同时管理票务、灯光和库存。

3. 适用场景

  • 中后台、交易、工作流、配置平台等复杂前端。
  • 多个页面复用同一业务规则。
  • 状态和接口交互较多。

4. 不适用场景

  • 静态展示页。
  • 一次性活动页。
  • 简单表单，抽象成本超过收益。

5. 核心结构

层职责
Page路由入口、页面布局、组合业务模块。
Feature业务区块，包含交互流程。
Component纯 UI 展示和基础交互。
Model/Store状态、派生数据、动作。
API Client请求封装和数据转换。

6. 示例代码

TypeScript 示例

type Order = { id: string; status: "created" | "paid" };

function canCancel(order: Order) {
  return order.status === "created";
}

async function fetchOrder(id: string): Promise<Order> {
  const res = await fetch(`/api/orders/${id}`);
  return res.json();
}

function OrderActions({ order }: { order: Order }) {
  return canCancel(order) ? "显示取消按钮" : "不可取消";
}

Python 示例

# 前端没有 Python UI，但同样可以在 BFF 或服务端渲染层保持边界。
def can_cancel(order: dict) -> bool:
    return order["status"] == "created"

def order_view_model(order: dict) -> dict:
    return {"id": order["id"], "canCancel": can_cancel(order)}

7. 优缺点

优点缺点
组件更薄，规则可测试。目录和约定需要维护。
API 变化影响可控。过度拆分会降低开发速度。

8. 和后端架构的区别

维度前端后端
核心复杂度交互状态、用户体验、缓存一致性事务、数据一致性、系统集成
边界重点页面和业务状态服务和数据边界

9. 一个练习题

把订单详情页拆成 Page、Feature、Component、API Client，并说明“订单能否取消”的判断放在哪里。

10. 参考答案或改进建议

如果规则只影响展示，可放在 feature model；如果与后端一致性相关，后端必须也校验。前端判断只能提升体验，不能作为唯一业务约束。

今日总结

前端架构的重点是让组件变薄，让业务规则和数据转换有固定位置。UI 可以变化，但业务边界不能散落。
