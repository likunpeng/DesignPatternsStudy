后端架构

后端架构关注业务流程、数据一致性、外部系统集成、事务边界和可观测性。它的目标是让系统在变化和故障下仍然可维护。

架构学习Backend ArchitectureTypeScript + Python事务与集成

今日目标

  • 理解 Controller、Use Case、Repository、Gateway 的职责。
  • 知道事务、缓存、消息、鉴权应该放在哪里。
  • 避免把后端写成巨型 Service。

1. 这个架构解决什么问题

后端复杂度主要来自状态变化和系统集成。订单创建可能涉及库存、支付、优惠、消息、风控。没有清晰边界时，一个 Service 会越来越大。

2. 一个生活类比

物流中心不是一个人完成所有事，而是收单、分拣、打包、运输、签收各有流程和记录。后端也需要明确流程和责任。

3. 适用场景

  • 业务流程有多个步骤。
  • 涉及事务、缓存、消息或外部系统。
  • 需要审计、监控、失败补偿。

4. 不适用场景

  • 非常简单的内部工具。
  • 无状态代理服务。
  • 过早引入消息和复杂补偿。

5. 核心结构

模块职责
Controller协议转换，不写业务。
Use Case流程编排和事务边界。
Domain核心规则。
Repository数据持久化抽象。
Gateway外部服务抽象。

6. 示例代码

TypeScript 示例

interface InventoryGateway { reserve(sku: string, qty: number): Promise<void>; }
interface OrderRepo { save(order: { sku: string; qty: number }): Promise<void>; }

class PlaceOrderUseCase {
  constructor(private inventory: InventoryGateway, private orders: OrderRepo) {}
  async execute(input: { sku: string; qty: number }) {
    if (input.qty <= 0) throw new Error("数量不合法");
    await this.inventory.reserve(input.sku, input.qty);
    await this.orders.save(input);
  }
}

Python 示例

from typing import Protocol

class InventoryGateway(Protocol):
    def reserve(self, sku: str, qty: int) -> None: ...

class OrderRepo(Protocol):
    def save(self, order: dict) -> None: ...

class PlaceOrderUseCase:
    def __init__(self, inventory: InventoryGateway, orders: OrderRepo):
        self.inventory = inventory
        self.orders = orders
    def execute(self, sku: str, qty: int) -> None:
        if qty <= 0:
            raise ValueError("数量不合法")
        self.inventory.reserve(sku, qty)
        self.orders.save({"sku": sku, "qty": qty})

7. 优缺点

优点缺点
流程清楚，故障点可定位。需要明确事务和补偿策略。
外部系统隔离。接口和适配器会增加数量。

8. 和前端架构的区别

后端更关注原因
数据一致性后端是状态最终落点。
事务边界失败后要知道哪些操作提交了。
可观测性线上问题需要日志、指标、链路追踪。

9. 一个练习题

设计“退款”后端流程，列出用例、仓储、支付网关、消息通知的职责。

10. 参考答案或改进建议

RefundUseCase 编排退款；PaymentGateway 调支付平台；OrderRepo 更新订单；消息通知通过端口发布。支付回调也应进入用例，而不是直接改数据库。

今日总结

后端架构的核心是流程和状态。把事务边界、外部依赖和领域规则分清楚，系统才经得起业务变化和线上故障。
