事件驱动架构

事件驱动架构用“已经发生的事实”连接模块或服务，让发送方不需要知道所有后续处理者。

架构学习Event-Driven ArchitectureTypeScript + Python异步解耦

今日目标

  • 理解事件和命令的区别。
  • 知道事件驱动适合异步解耦，不适合所有流程。
  • 了解 Outbox 作为可靠发布的常见做法。

1. 这个架构解决什么问题

订单创建后可能要扣库存、发短信、加积分、推送物流。如果订单模块直接调用所有模块，会越来越耦合。事件让订单模块只发布“订单已创建”。

2. 一个生活类比

公司发布公告后，各部门根据公告自行处理，不需要公告发布者逐个指挥每个部门。

3. 适用场景

  • 一个业务事实触发多个后续动作。
  • 后续动作可以异步完成。
  • 模块之间需要降低直接依赖。

4. 不适用场景

  • 必须立即强一致的流程。
  • 调用链很短且同步更清晰。
  • 团队缺少消息可靠性和监控能力。

5. 核心结构

概念说明
Event已经发生的事实。
Publisher发布事件。
Subscriber订阅并处理事件。
Event Bus传递事件。
Outbox把业务数据和待发送事件一起落库，提高可靠性。

6. 示例代码

TypeScript 示例

type OrderCreated = { type: "OrderCreated"; orderId: string };
interface EventBus { publish(event: OrderCreated): Promise<void>; }
class CreateOrderUseCase {
  constructor(private bus: EventBus) {}
  async execute() { await this.bus.publish({ type: "OrderCreated", orderId: "1" }); }
}

Python 示例

class EventBus:
    def publish(self, event: dict) -> None:
        print(event)

class CreateOrderUseCase:
    def __init__(self, bus: EventBus):
        self.bus = bus
    def execute(self) -> None:
        self.bus.publish({"type": "OrderCreated", "orderId": "1"})

7. 优缺点

优点缺点
模块解耦，扩展订阅者方便。流程变隐式，排查更难。
适合异步和削峰。需要处理重复消息、顺序、失败重试。

8. 和命令的区别

命令事件
要求某人做某事。声明某事已经发生。
通常有明确接收者。可以有多个订阅者。

9. 一个练习题

订单支付成功后，设计一个事件和三个订阅者。

10. 参考答案或改进建议

事件为 PaymentSucceeded；订阅者可以是订单状态更新、积分发放、通知发送。支付模块不应直接知道所有后续动作。

今日总结

事件驱动适合解耦异步后续动作，但它不是万能方案。可靠发布、幂等和监控是落地关键。
