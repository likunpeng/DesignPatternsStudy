MVC 架构模式

MVC 把应用拆成 Model、View、Controller，用来避免界面展示、用户输入和业务数据混在一起。

架构学习MVCTypeScript + Python表现层架构

今日目标

  • 理解 MVC 解决的职责混乱问题。
  • 知道 Controller 为什么容易变胖。
  • 能判断现代 Web/前端中的 MVC 变体。

1. 这个模式解决什么问题

当页面代码同时处理输入、渲染、业务规则和数据访问时，任何改动都会影响整块代码。MVC 通过分工降低耦合。

2. 一个生活类比

餐厅里菜单是 View，厨房和菜品规则是 Model，服务员接单并协调厨房是 Controller。服务员不应该亲自修改菜谱。

3. 适用场景

  • Web 应用、桌面应用、传统服务端渲染。
  • 页面交互不算特别复杂，但需要基本分层。
  • 希望把输入处理和数据模型分开。

4. 不适用场景

  • 复杂响应式状态页面，MVVM 或单向数据流更清晰。
  • Controller 已经变成巨型流程编排器。
  • 简单静态页面。

5. 核心结构

角色职责
Model数据和业务规则。
View展示数据，不决定业务规则。
Controller接收输入，调用 Model，选择 View。

6. 示例代码

TypeScript 示例

class UserModel { constructor(public name: string) {} }
function userView(user: UserModel) { return `<h1>${user.name}</h1>`; }
function userController(id: string) {
  const user = new UserModel(`用户 ${id}`);
  return userView(user);
}

Python 示例

class UserModel:
    def __init__(self, name: str):
        self.name = name

def user_view(user: UserModel) -> str:
    return f"<h1>{user.name}</h1>"

def user_controller(id: str) -> str:
    user = UserModel(f"用户 {id}")
    return user_view(user)

7. 优缺点

优点缺点
结构直观，入门成本低。Controller 容易变胖。
适合传统 Web 分层。复杂 UI 状态表达不如 MVVM 清晰。

8. 和相似模式的区别

模式区别
MVPView 更被动，Presenter 更容易测试。
MVVMViewModel 暴露可绑定状态。

9. 一个练习题

找一个登录页面，指出其中哪些代码属于 View、Controller、Model。

10. 参考答案或改进建议

输入框和按钮是 View；提交事件处理是 Controller；用户、登录规则和认证结果是 Model 或业务服务。不要把密码规则直接散落在模板里。

今日总结

MVC 是表现层分工的起点。AI 协作时要明确：View 不写业务规则，Controller 不直接变成所有逻辑的垃圾桶。
