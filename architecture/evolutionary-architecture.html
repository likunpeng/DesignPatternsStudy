演进式架构

演进式架构强调先满足当前真实需求，再通过测试、边界和重构逐步演进，避免一开始设计过重，也避免长期不治理。

架构学习Evolutionary ArchitectureTypeScript + Python渐进演进

今日目标

  • 理解架构不是一次性设计完成。
  • 知道何时保持简单，何时重构升级。
  • 掌握从单体到模块化单体再到微服务的常见路径。
  • 学会用 AI 辅助安全演进，而不是盲目重写。

1. 这个方法解决什么问题

过早设计会浪费成本，完全不设计会形成泥团。演进式架构在二者之间建立机制：先简单实现，但保留测试、边界和决策记录，让系统能安全改变。

2. 一个生活类比

城市不是一天建成的。先有道路和基础设施，再根据人口增长修地铁、分区和立交桥。软件架构也应随真实压力演进。

3. 适用场景

  • 业务还在变化，但有长期发展可能。
  • 旧系统需要逐步治理，不能推倒重写。
  • 团队希望避免过早微服务化。
  • AI 协助重构，需要明确安全步骤。

4. 不适用场景

  • 有强监管或强安全要求，必须前置完整设计。
  • 一次性项目。
  • 没有测试和监控，演进风险不可控。

5. 核心结构

阶段重点
简单单体快速验证业务，不提前拆服务。
分层单体明确入口、用例、领域、基础设施。
模块化单体按业务能力拆模块，控制模块依赖。
服务化边界稳定后按数据归属独立部署。

6. 示例代码

TypeScript 示例

// 演进前：一个函数里混合规则和实现。
async function createOrder(amount: number) {
  if (amount <= 0) throw new Error("金额不合法");
  console.log("保存订单");
}

// 演进后：先抽出用例和端口，不急着拆微服务。
interface OrderRepo { save(amount: number): Promise<void>; }
class CreateOrderUseCase {
  constructor(private repo: OrderRepo) {}
  async execute(amount: number) {
    if (amount <= 0) throw new Error("金额不合法");
    await this.repo.save(amount);
  }
}

Python 示例

from typing import Protocol

class OrderRepo(Protocol):
    def save(self, amount: int) -> None: ...

class CreateOrderUseCase:
    def __init__(self, repo: OrderRepo):
        self.repo = repo
    def execute(self, amount: int) -> None:
        if amount <= 0:
            raise ValueError("金额不合法")
        self.repo.save(amount)

7. 优缺点

优点缺点
避免过早复杂化。需要持续治理，不能放任不管。
适合真实业务不确定性。如果没有测试，演进会变成冒险。

8. 和一次性大设计的区别

一次性大设计演进式架构
试图提前预测所有变化识别关键变化点，逐步调整
前期成本高前期轻量，但要求持续反馈

9. 一个练习题

一个订单模块现在所有逻辑都在一个 Service 里。请设计三步演进路线，不允许直接拆微服务。

10. 参考答案或改进建议

第一步补核心规则测试；第二步抽出用例和仓储端口；第三步按订单、支付、库存建立模块边界。只有当边界稳定、数据归属清楚、团队具备运维能力后，再考虑拆服务。

今日总结

演进式架构不是“不设计”，而是用测试、边界和决策记录支撑持续改变。AI 可以加速重构，但演进路线必须由人把关。
