设计原则教程

设计原则不是背口号，而是帮助你在代码变复杂之前看见风险。七大原则最常见的价值，不是写出更花哨的结构，而是让需求变化时少改旧代码、少引入连锁错误。

前置基础Design PrinciplesTypeScript + Python七大原则

今日目标

  • 理解七大设计原则分别在解决什么问题。
  • 通过坏代码和改进代码，理解原则不是空话。
  • 知道每条原则在真实业务里最常见的落地点。
  • 避免把设计原则机械套用成模板。

1. 单一职责原则

核心思想：一个类或模块只负责一件事，变化原因尽量只有一个。

生活类比：餐厅里让一个人同时做饭、收银、洗碗，任何一个环节出问题都会拖垮全部流程。

坏代码

class OrderService {
  create(order: { email: string; amount: number }): void {
    if (order.amount <= 0) {
      throw new Error("金额不合法");
    }

    console.log("保存订单到数据库");
    console.log(`发送邮件给 ${order.email}`);
  }
}

问题在于：校验规则变化、存储方式变化、通知方式变化，都会迫使你修改同一个类。

为什么这种类会越改越重

class OrderService {
  create(order: { email: string; amount: number; source: string }): void {
    if (order.amount <= 0) {
      throw new Error("金额不合法");
    }

    if (order.source === "app") {
      console.log("记录 App 渠道统计");
    }

    console.log("保存订单到数据库");
    console.log("写审计日志");
    console.log(`发送邮件给 ${order.email}`);
    console.log("推送站内消息");
  }
}

一开始只是“顺手多写一行”，后面统计、日志、通知、风控都会继续往里堆。最后你会得到一个谁都不敢动的大类，因为任何修改都可能误伤别的职责。

TypeScript 示例

type OrderInput = { email: string; amount: number };

class OrderValidator {
  validate(input: OrderInput): void {
    if (input.amount <= 0) {
      throw new Error("金额不合法");
    }
  }
}

class OrderRepository {
  save(input: OrderInput): void {
    console.log("保存订单到数据库");
  }
}

class OrderNotifier {
  send(email: string): void {
    console.log(`发送邮件给 ${email}`);
  }
}

class OrderService {
  constructor(
    private validator: OrderValidator,
    private repository: OrderRepository,
    private notifier: OrderNotifier,
  ) {}

  create(input: OrderInput): void {
    this.validator.validate(input);
    this.repository.save(input);
    this.notifier.send(input.email);
  }
}

现在每个协作者都有更明确的变化原因：校验规则改了，去改 OrderValidator；通知方式改了，去改 OrderNotifier；主流程只负责协调，不负责把所有细节都吞进去。

Python 示例

class OrderValidator:
    def validate(self, amount: int) -> None:
        if amount <= 0:
            raise ValueError("金额不合法")


class OrderRepository:
    def save(self, order_data: dict) -> None:
        print(f"保存订单到数据库: {order_data}")


class OrderNotifier:
    def send(self, email: str) -> None:
        print(f"发送邮件给 {email}")


class OrderService:
    def __init__(self, validator: OrderValidator, repository: OrderRepository, notifier: OrderNotifier):
        self.validator = validator
        self.repository = repository
        self.notifier = notifier

    def create(self, email: str, amount: int) -> None:
        self.validator.validate(amount)
        self.repository.save({"email": email, "amount": amount})
        self.notifier.send(email)

Python 里也一样，不是为了“面向对象而拆类”，而是为了让不同变化各回各家。一个类如果总是因为不同理由被改，通常就是职责已经混在一起了。

判断标准不是“类越小越好”，而是“这个类是不是因为不同原因被反复修改”。

2. 开闭原则

核心思想：对扩展开放，对修改关闭。新增能力时，优先增加新代码，而不是反复改稳定代码。

生活类比：插排多一个插孔，不需要拆墙重拉电线；只要能扩展接口，就不用破坏原结构。

坏代码

function pay(type: string, amount: number): void {
  if (type === "alipay") {
    console.log(`支付宝支付 ${amount} 元`);
  } else if (type === "wechat") {
    console.log(`微信支付 ${amount} 元`);
  } else if (type === "bank") {
    console.log(`银行卡支付 ${amount} 元`);
  }
}

每新增一种支付方式，都要改老函数。改得越频繁，越容易把旧逻辑改坏。

为什么这会越来越难维护

function pay(type: string, amount: number): void {
  if (type === "alipay") {
    console.log(`支付宝支付 ${amount} 元`);
  } else if (type === "wechat") {
    console.log(`微信支付 ${amount} 元`);
  } else if (type === "bank") {
    console.log(`银行卡支付 ${amount} 元`);
  } else if (type === "coupon") {
    console.log(`优惠券抵扣 ${amount} 元`);
  } else if (type === "points") {
    console.log(`积分支付 ${amount} 元`);
  }
}

问题不只是 if-else 长，而是每次加一种支付方式，你都要回头改这个核心入口。它会逐渐变成最脆弱、最容易被误改的函数。

TypeScript 示例

interface PaymentMethod {
  pay(amount: number): void;
}

class Alipay implements PaymentMethod {
  pay(amount: number): void {
    console.log(`支付宝支付 ${amount} 元`);
  }
}

class WechatPay implements PaymentMethod {
  pay(amount: number): void {
    console.log(`微信支付 ${amount} 元`);
  }
}

class CheckoutService {
  checkout(method: PaymentMethod, amount: number): void {
    method.pay(amount);
  }
}

class BankCardPay implements PaymentMethod {
  pay(amount: number): void {
    console.log(`银行卡支付 ${amount} 元`);
  }
}

这里的关键不是“永远不改旧代码”，而是新增支付方式时，你主要是在增加一个新实现，比如 BankCardPay，而不是继续往老函数里塞分支。

Python 示例

from typing import Protocol


class PaymentMethod(Protocol):
    def pay(self, amount: int) -> None:
        ...


class Alipay:
    def pay(self, amount: int) -> None:
        print(f"支付宝支付 {amount} 元")


class WechatPay:
    def pay(self, amount: int) -> None:
        print(f"微信支付 {amount} 元")


class CheckoutService:
    def checkout(self, method: PaymentMethod, amount: int) -> None:
        method.pay(amount)


class BankCardPay:
    def pay(self, amount: int) -> None:
        print(f"银行卡支付 {amount} 元")

Python 里这个原则通常落在“函数注册表”“策略对象”“插件实现”这些地方。本质一样：新需求优先加新实现，而不是不停回头改那个总入口。

开闭原则不等于永远不改旧代码，而是别把变化集中压在一个核心函数里。

3. 里氏替换原则

核心思想：子类应该能够替换父类使用，而不破坏原有程序的正确性。

生活类比：你租的是“带空调的酒店房间”，结果入住后前台告诉你“这个房型虽然也叫房间，但不能开空调”。名称没变，承诺却变了，这就是替换失败。

坏代码

abstract class PaymentMethod {
  abstract pay(amount: number): void;
  abstract refund(amount: number): void;
}

class AlipayPayment extends PaymentMethod {
  pay(amount: number): void {
    console.log(`支付宝支付 ${amount} 元`);
  }

  refund(amount: number): void {
    console.log(`支付宝退款 ${amount} 元`);
  }
}

class GiftCardPayment extends PaymentMethod {
  pay(amount: number): void {
    console.log(`礼品卡支付 ${amount} 元`);
  }

  refund(amount: number): void {
    throw new Error("礼品卡不支持原路退款");
  }
}

function cancelOrder(payment: PaymentMethod, amount: number): void {
  payment.refund(amount);
}

调用方看到的是 PaymentMethod，自然会以为“任何支付方式都能退款”。但换成 GiftCardPayment 后直接抛异常，说明子类没有遵守父类原本承诺的行为。

为什么这里是替换失败

const payment: PaymentMethod = new GiftCardPayment();
cancelOrder(payment, 100); // 调用方以为一定能退款，结果运行时报错

里氏替换看的是调用方视角：如果一段代码在传入父类型时成立，换成某个子类型后却突然不成立，那就是替换失败。

TypeScript 示例

interface PaymentMethod {
  pay(amount: number): void;
}

interface RefundablePayment {
  refund(amount: number): void;
}

class AlipayPayment implements PaymentMethod, RefundablePayment {
  pay(amount: number): void {
    console.log(`支付宝支付 ${amount} 元`);
  }

  refund(amount: number): void {
    console.log(`支付宝退款 ${amount} 元`);
  }
}

class GiftCardPayment implements PaymentMethod {
  pay(amount: number): void {
    console.log(`礼品卡支付 ${amount} 元`);
  }
}

function checkout(payment: PaymentMethod, amount: number): void {
  payment.pay(amount);
}

function cancelOrder(payment: RefundablePayment, amount: number): void {
  payment.refund(amount);
}

改完以后，契约被拆清楚了：所有支付方式都能 pay，但不是所有支付方式都能 refund。调用方依赖哪个能力，就接收哪个抽象。

Python 示例

from typing import Protocol


class PaymentMethod(Protocol):
    def pay(self, amount: int) -> None:
        ...


class RefundablePayment(Protocol):
    def refund(self, amount: int) -> None:
        ...


class AlipayPayment:
    def pay(self, amount: int) -> None:
        print(f"支付宝支付 {amount} 元")

    def refund(self, amount: int) -> None:
        print(f"支付宝退款 {amount} 元")


class GiftCardPayment:
    def pay(self, amount: int) -> None:
        print(f"礼品卡支付 {amount} 元")


def checkout(payment: PaymentMethod, amount: int) -> None:
    payment.pay(amount)


def cancel_order(payment: RefundablePayment, amount: int) -> None:
    payment.refund(amount)

Python 里这条原则通常体现为：别让一个对象“看起来像能用”，结果传进去才发现少关键能力。用更准确的协议拆开能力边界，比勉强复用一个大父类型更稳。

里氏替换最常见的坑，不是“不能继承”，而是“子类表面上继承了父类，实际上却偷偷缩小了能力、改变了规则，导致调用方对父类的预期失效”。

4. 接口隔离原则

核心思想：不要逼使用者依赖它不需要的方法。宁可多个小接口，也不要一个胖接口。

生活类比：程序员的工牌只需要开电脑房和办公区门，不需要顺带拥有仓库、厨房、保洁间的权限按钮。

坏代码

interface Worker {
  code(): void;
  deliver(): void;
  clean(): void;
}

class Programmer implements Worker {
  code(): void { console.log("写代码"); }
  deliver(): void { throw new Error("不会送餐"); }
  clean(): void { throw new Error("不会保洁"); }
}

一旦实现类里开始出现“这个方法我其实不会”“这里只能抛异常”，就说明接口已经把不相关的职责硬塞在一起了。

为什么胖接口会制造假实现

interface ReportExporter {
  exportPDF(): void;
  exportExcel(): void;
  sendEmail(): void;
  uploadToS3(): void;
}

class LocalExcelExporter implements ReportExporter {
  exportPDF(): void { throw new Error("不支持"); }
  exportExcel(): void { console.log("导出 Excel"); }
  sendEmail(): void { throw new Error("不支持"); }
  uploadToS3(): void { throw new Error("不支持"); }
}

这类代码最糟的地方不是丑，而是调用方会被误导：接口看起来什么都能做，实际上很多实现只支持其中一小部分能力。

TypeScript 示例

interface Coder {
  code(): void;
}

interface Deliverer {
  deliver(): void;
}

interface Cleaner {
  clean(): void;
}

class Programmer implements Coder {
  code(): void {
    console.log("写代码");
  }
}

class DeliveryPerson implements Deliverer {
  deliver(): void {
    console.log("送外卖");
  }
}

这样拆开后，每个角色只实现自己真正需要的能力。调用方要找会写代码的人，就依赖 Coder；要找配送员，就依赖 Deliverer。

Python 示例

from typing import Protocol


class Coder(Protocol):
    def code(self) -> None:
        ...


class Deliverer(Protocol):
    def deliver(self) -> None:
        ...


class Cleaner(Protocol):
    def clean(self) -> None:
        ...


class Programmer:
    def code(self) -> None:
        print("写代码")


# 程序员不需要实现 deliver() 和 clean()
# 如果需要，就单独定义对应角色的人
class DeliveryPerson:
    def deliver(self) -> None:
        print("送外卖")

Python 里胖接口也很常见，尤其是在工具类、SDK 封装和导出服务里。看到一个类实现很多“其实用不上”的方法，通常就该警觉接口是否已经设计过头了。

接口隔离的目标不是拆得越细越好，而是避免“实现一个接口，却被迫写一堆无意义的方法”。

5. 依赖倒置原则

核心思想：高层模块不应该直接依赖低层实现，双方都应该依赖抽象。

生活类比：遥控器不该焊死在某个电视品牌主板上，它只应该依赖一套统一的按键协议。

坏代码

class MySQLDatabase {
  insert(data: string): void {
    console.log(`MySQL 保存 ${data}`);
  }
}

class UserService {
  private database = new MySQLDatabase();

  save(name: string): void {
    this.database.insert(name);
  }
}

如果以后换 PostgreSQL、MongoDB，就只能修改 UserService。

真正的问题不只是“换数据库麻烦”

class MySQLDatabase {
  insert(data: string): void {
    console.log(`MySQL 保存 ${data}`);
  }
}

class UserService {
  private database = new MySQLDatabase();

  save(name: string): void {
    this.database.insert(name);
  }
}

这段代码还有两个隐藏问题：一是测试时很难替换成假对象；二是高层业务被迫知道底层实现细节。业务本来只关心“保存用户”，不该关心底层到底是 MySQL 还是别的存储。

TypeScript 示例

interface Database {
  insert(data: string): void;
}

class MySQLDatabase implements Database {
  insert(data: string): void {
    console.log(`MySQL 保存 ${data}`);
  }
}

class UserService {
  constructor(private database: Database) {}

  save(name: string): void {
    this.database.insert(name);
  }
}

class MemoryDatabase implements Database {
  insert(data: string): void {
    console.log(`内存保存 ${data}`);
  }
}

现在 UserService 只依赖 Database 这个能力。换数据库、写测试替身、做本地内存实现，都不需要改业务类本身。

Python 示例

from typing import Protocol


class Database(Protocol):
    def insert(self, data: str) -> None:
        ...


class MySQLDatabase:
    def insert(self, data: str) -> None:
        print(f"MySQL 保存 {data}")


class UserService:
    def __init__(self, database: Database):
        self.database = database

    def save(self, name: str) -> None:
        self.database.insert(name)


class MemoryDatabase:
    def insert(self, data: str) -> None:
        print(f"内存保存 {data}")

Python 里依赖倒置经常表现为：构造函数传入协作者，而不是在类内部自己 new 一个。这样既方便替换实现，也方便注入测试替身。

依赖倒置的重点不是“把一切都抽成接口”，而是高层业务别直接绑死在某个底层实现上。

6. 迪米特法则

核心思想：一个对象应当尽量少了解其他对象的内部细节，只和“直接朋友”通信。

生活类比：你在餐厅结账时应该找服务员，而不是自己跑去后厨翻账本、找收银系统、再去打印小票。

坏代码

const temp = computer.getCpu().getCore(0).getTemperature();

这行代码知道了太多内部结构：电脑里有 CPU，CPU 里有核心，核心里有温度。只要中间某层结构改动，调用方就一起崩。

为什么“链式拿数据”会变脆弱

const temp = computer.getCpu().getCore(0).getTemperature();

// 假设以后 CPU 不再暴露 core，而是直接暴露 sensor
// 那所有这样写的调用方都得跟着改

迪米特法则不是反对点语法，而是反对调用方穿透多层对象去摸内部结构。你知道的层级越多，对方一改结构，你就越容易受伤。

TypeScript 示例

class Core {
  getTemperature(): number {
    return 68;
  }
}

class Cpu {
  private core = new Core();

  getAverageTemperature(): number {
    return this.core.getTemperature();
  }
}

class Computer {
  private cpu = new Cpu();

  getCpuTemperature(): number {
    return this.cpu.getAverageTemperature();
  }
}

const computer = new Computer();
const temp = computer.getCpuTemperature();

现在调用方只知道“向电脑要 CPU 温度”，不知道里面到底是一个核心、多个核心，还是别的传感器结构。内部怎么变，封装在 Computer 和 Cpu 里面。

Python 示例

class Core:
    def get_temperature(self) -> int:
        return 68


class Cpu:
    def __init__(self):
        self._core = Core()

    def get_average_temperature(self) -> int:
        return self._core.get_temperature()


class Computer:
    def __init__(self):
        self._cpu = Cpu()

    def get_cpu_temperature(self) -> int:
        return self._cpu.get_average_temperature()


computer = Computer()
temp = computer.get_cpu_temperature()

Python 里这一条也很重要，尤其是在字典、对象、ORM 模型层层嵌套的时候。不要让上层业务一路穿透到底层细节，尽量由中间对象提供更直接的意图方法。

迪米特法则的本质，是把内部层级藏起来，让调用方只拿到它真正需要的结果。

和依赖倒置原则的区别

原则主要关注点典型问题
依赖倒置原则依赖方向高层业务直接绑死在 MySQL、Redis、短信 SDK 这类具体实现上。
迪米特法则暴露细节的层级调用方一路穿透 a.b.c.d 去摸对象内部结构。

它们都在降低耦合，但不是同一件事。依赖倒置更像是在问“我依赖的是不是具体实现”，迪米特法则更像是在问“我是不是知道了太多内部结构”。一个管依赖谁，一个管知道多少。

7. 组合优于继承

核心思想：优先通过组合对象来获得能力，而不是通过继承把行为绑死在层级结构上。

生活类比：给狗装飞行能力，最合理的方式不是让“狗继承鸟”，而是给它加一个“飞行组件”。

坏代码

class Animal {
  eat(): void {
    console.log("吃东西");
  }
}

class Bird extends Animal {
  fly(): void {
    console.log("飞行");
  }
}

class FlyingDog extends Bird {}

FlyingDog 继承了飞，但也被迫继承了“鸟”的语义。类层次会越来越怪。

为什么继承在这里会越走越歪

class Bird extends Animal {
  fly(): void {
    console.log("飞行");
  }
}

class FlyingDog extends Bird {}
class FlyingFish extends Bird {}
class FlyingCat extends Bird {}

一开始你只是想复用 fly()，结果为了这个能力，不得不把狗、鱼、猫都塞进“鸟”的层级里。说明这个复用方向从语义上就不对。

TypeScript 示例

interface Flyable {
  fly(): void;
}

class Wings implements Flyable {
  fly(): void {
    console.log("飞行");
  }
}

class Dog {
  constructor(private flyable?: Flyable) {}

  run(): void {
    console.log("奔跑");
  }

  tryFly(): void {
    this.flyable?.fly();
  }
}

const normalDog = new Dog();
normalDog.run();

const superDog = new Dog(new Wings());
superDog.tryFly();

现在“会不会飞”变成一个可拼装能力，而不是血统问题。狗还是狗，只是有的狗额外装了飞行能力。

Python 示例

from typing import Protocol


class Flyable(Protocol):
    def fly(self) -> None:
        ...


class Wings:
    def fly(self) -> None:
        print("飞行")


class Dog:
    def __init__(self, flyable: Flyable | None = None):
        self.flyable = flyable

    def run(self) -> None:
        print("奔跑")

    def try_fly(self) -> None:
        if self.flyable:
            self.flyable.fly()


normal_dog = Dog()
normal_dog.run()

super_dog = Dog(Wings())
super_dog.try_fly()

这也是为什么很多模式更偏爱组合：装饰器、代理、策略，本质上都是把能力拆出来再组合，而不是不断加深继承树。

继承适合稳定的“是一种”关系；组合更适合变化频繁的能力拼装。

8. 七大原则之间的关系

原则重点最常见落点
单一职责别把太多变化原因塞进一个类服务类、控制器、工具类拆分
开闭原则新增能力尽量加新代码策略、工厂、插件化结构
里氏替换子类替换父类不能破坏约定继承层次设计
接口隔离不要胖接口角色接口、SDK 接口、能力接口
依赖倒置高层依赖抽象，不绑死实现仓储层、网关层、外部依赖封装
迪米特法则少暴露内部细节链式调用过深、领域对象穿透
组合优于继承能力拼装比层级扩张更灵活桥接、装饰器、代理、策略

9. 练习题

下面这段代码同时违反了哪些设计原则？应该怎么改？

class ReportService {
  export(type: string, user: string): void {
    if (type === "pdf") {
      console.log("导出 PDF");
    } else if (type === "excel") {
      console.log("导出 Excel");
    }

    console.log("写数据库日志");
    console.log(`发送通知给 ${user}`);
  }
}
  • 先判断它违反了哪些原则。
  • 再思考：导出、日志、通知是否应该放在一个类里？
  • 如果后续新增 CSV、图片导出，哪种改法更稳定？

10. 参考答案或改进建议

这段代码至少违反了：

  • 单一职责原则：导出、记录日志、发送通知耦合在一起。
  • 开闭原则：每加一种导出类型都要修改 if-else。
  • 依赖倒置原则：如果以后把日志或通知接到外部系统，高层导出逻辑会直接受影响。

一种更合理的改法是：把导出做成策略接口，把日志和通知拆成独立协作者，由主流程协调调用。

今日总结

设计原则的价值，不在于你能背出多少条定义，而在于看到坏代码时能立刻意识到：这里为什么难扩展、为什么容易出 Bug、为什么一改就牵一发而动全身。后面的设计模式，本质上就是这些原则在不同场景下的落地方式。
